# ADR-0005: Мультитенантность через row-level isolation и идентификацию тенанта по URL/токену

- Status: Accepted
- Date: 2026-06-06
- Change: sre-oncall-platform (commit c7941dd)
- Affected: все сервисы, frontend (роутинг `/[tenant]/...`)

## Context

Тенант = команда внутри одной организации: один деплой, одна БД, доверенные пользователи. Каждая команда имеет собственные расписания, политики эскалации, инциденты и конфигурацию уведомлений; пользователь может состоять в нескольких командах. Ingestion должен определять тенанта по вебхуку без аутентифицированного пользователя.

## Options considered

- **Row-level isolation (`tenant_id NOT NULL` в каждой доменной таблице)** — простые миграции, достаточная изоляция для внутреннего инструмента.
- **Schema-per-tenant** — более строгая изоляция, но усложняет миграции и избыточна для внутреннего инструмента с доверенными пользователями. Отклонена.

## Decision

Изоляция реализована на уровне строк: каждая доменная таблица содержит `tenant_id NOT NULL`, все запросы фильтруются по нему. Справочник тенантов, токенов и конфигурации уведомлений живёт в схеме `scheduling`. Идентификация тенанта:

- **Ingestion-вебхуки**: заголовок `X-Webhook-Token` → lookup в Redis (`HGET oncall:tokens:{SHA256(token)}` → `tenant_id`). Индекс поддерживает scheduling (`HSET` при создании токена, `HDEL` при отзыве); ingestion не читает схему `scheduling` и не вызывает scheduling по HTTP.
- **Все остальные API**: tenant slug в URL (`/api/incidents/v1/{tenant}/...`); фронтенд зеркалирует его как верхний сегмент маршрута.

В БД хранится только SHA-256 хэш токена; plaintext показывается администратору ровно один раз при создании.

## Consequences

- Каждая новая доменная таблица обязана нести `tenant_id`, каждый новый запрос — фильтр по нему; пропуск фильтра — кросс-тенантная утечка.
- Redis-индекс токенов — межсервисный контракт scheduling → ingestion; его формат (`oncall:tokens:{hash}`) нельзя менять односторонне.
- Восстановление утерянного токена невозможно — только перевыпуск.
- Удаление тенанта каскадно только в данных scheduling; инциденты, эскалации и журнал уведомлений остаются (документированное ограничение, зафиксировано в sync-specs-with-implementation).
