# ADR-0009: Межсервисная аутентификация через общий `X-Admin-Key` и условное маскирование секретов

- Status: Accepted
- Date: 2026-06-10
- Change: fix-p0-delivery-and-filters (commit 007c021)
- Affected: pkg/auth, services/scheduling, services/escalation, services/notification, deploy/k8s

## Context

Все маршруты scheduling закрыты JWT-middleware, но межсервисные клиенты escalation и notification не отправляли ни Bearer-токен, ни ключ — в k8s с включённым JWKS эскалация получала 401 и публиковала события без дежурного. Дополнительно: `GET /tenants/{slug}/notification-config` маскирует `mattermost_webhook_url` для защиты UI, а notification использует тот же эндпоинт и получал маскированное значение.

## Options considered

- **Существующий `X-Admin-Key` как сервисный ключ** — механизм уже реализован в `pkg/auth.Middleware`, не требует изменений Keycloak; минимальный диф для P0.
- **Keycloak client-credentials** — правильнее (ротация, аудит), но требует настройки realm, клиентов и токен-кэша. Отклонено для P0, зафиксировано как follow-up.
- **Отдельный internal-эндпоинт без маскирования** — два маршрута с разными политиками легко рассинхронизировать; «internal»-путь всё равно требует той же проверки ключа. Отклонено.
- **Внутренняя сеть/порт без auth** — усложняет деплой и ослабляет защиту в общем неймспейсе. Отклонено.

## Decision

Schedclient'ы escalation и notification передают заголовок `X-Admin-Key` в каждом запросе к scheduling; ключ выдаётся через общий Kubernetes Secret. `pkg/auth` помечает в контексте способ аутентификации (`service` при ключе, `user` при JWT); `mattermost_webhook_url` маскируется только для `user`-запросов — один эндпоинт, один контракт, маскирование по умолчанию (полный URL только при явном `service`).

## Consequences

- Сервисный ключ обходит tenant-проверки и даёт cross-tenant доступ — принято осознанно (сервисы и так работают со всеми тенантами) и зафиксировано в спеке как свойство сервисного режима.
- Утечка общего статического ключа открывает все scheduling-эндпоинты: ключ живёт только в Secret и не логируется; переход на client-credentials — открытый follow-up.
- Маскирование — свойство представления, не хранения; альтернативных маршрутов, раскрывающих немаскированный URL пользовательским JWT, существовать не должно (закрепляется изменением remove-unmasked-notification-config-routes).
- Новые межсервисные HTTP-вызовы должны использовать тот же механизм, а не ходить без аутентификации.
