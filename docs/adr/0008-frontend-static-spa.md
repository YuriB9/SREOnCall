# ADR-0008: Фронтенд — статический SPA без собственного бэкенда

- Status: Accepted
- Date: 2026-06-06
- Change: sre-oncall-frontend (commit c7941dd)
- Affected: frontend/

## Context

Бэкенд предоставляет полный REST API за Ingress; пути привязаны к тенанту, аутентификация — Bearer JWT от Keycloak ([ADR-0006](0006-keycloak-oidc-groups-auth.md)). Приложение — внутренний инструмент, SEO не требуется. Обновлений в реальном времени (WebSocket/SSE) бэкенд v1 не предоставляет.

## Options considered

- **Vite + React SPA, статический бандл** — раздаётся любым существующим Ingress или nginx-сайдкаром; отдельный Deployment и Node-сервер не нужны.
- **Next.js (App Router) + BFF** — ценность BFF/middleware низкая (нечего скрывать и агрегировать в v1), а операционная стоимость Node-сервера существенна. Отклонён.
- **NextAuth.js для OIDC** — требует бэкенд; отклонён вместе с BFF.

## Decision

Фронтенд — SPA на Vite + React + TypeScript, раздаваемое как статический бандл, без SSR и BFF-прокси. Keycloak Authorization Code Flow + PKCE целиком на клиенте через `oidc-client-ts` (токен в `sessionStorage`, silent renew). Все вызовы API — через TanStack Query с поллингом (инциденты — каждые 12 сек) вместо WebSocket/SSE. Роутинг зеркалирует параметр `{tenant}` бэкенда как верхний сегмент URL; ролевые гарды на клиенте — UX-удобство, реальная граница безопасности на сервере.

## Consequences

- Деплой фронтенда — публикация статики; новых эндпоинтов бэкенда для UI не требуется.
- Новый инцидент появляется в UI с задержкой до интервала поллинга; переход на SSE/WebSocket — отдельное изменение бэкенда и фронтенда.
- Конфигурация Keycloak обязана разрешать PKCE flow с origin фронтенда и отдавать claim `groups` в обоих токенах ([ADR-0006](0006-keycloak-oidc-groups-auth.md)).
- Любая логика, требующая скрытия секретов или агрегации ответов сервисов, не может жить во фронтенде — потребует пересмотра этого решения (BFF).
