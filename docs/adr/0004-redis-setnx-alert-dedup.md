# ADR-0004: Дедупликация алертов через Redis SETNX по fingerprint

- Status: Accepted
- Date: 2026-06-06
- Change: sre-oncall-platform (commit c7941dd); контракт отказа уточнён в sync-specs-with-implementation (commit 007c021, 2026-06-10)
- Affected: services/ingestion, pkg/redis

## Context

Ingestion принимает вебхуки от Alertmanager и Grafana и должен гасить повторные доставки одного алерта в условиях волн алертов, когда задержка записи критична.

## Options considered

- **Redis SETNX с TTL** — атомарен, работает за субмиллисекунды; подходит для огневого характера обработки вебхуков.
- **Уникальный индекс в БД** — работает, но добавляет задержку при записи и нагрузку на базу во время волн алертов. Отклонён.

## Decision

Входящие алерты получают fingerprint — SHA-256 отсортированных пар ключ-значение лейблов + поле source. Redis SETNX с настраиваемым TTL (по умолчанию 5 минут) защищает от повторной публикации одного алерта в рамках окна дедупликации. Resolved-алерты минуют дедупликацию, и ключ удаляется (чтобы инцидент мог закрыться).

## Consequences

- Алгоритм fingerprint задокументирован, чтобы интеграции могли предсказывать поведение; его изменение меняет границы дедупликации и требует осознанного решения.
- Публикация в RabbitMQ синхронна относительно вебхука: при сбое публикации ingestion возвращает HTTP 503 и откатывает dedup-ключ, чтобы повторная доставка вебхука не была потеряна (зафиксировано в sync-specs-with-implementation).
- Окно дедупликации эфемерно: потеря Redis означает лишь возможный дубликат события, не потерю алерта.
