# ADR-0015: Жизненный цикл и устойчивость консьюмеров RabbitMQ

- Status: Accepted
- Date: 2026-06-13
- Change: consumer-resilience
- Affected: pkg/amqp, services/incident, services/escalation, services/notification

## Context

Транспорт шины централизован в `pkg/amqp` ([ADR-0002](0002-rabbitmq-event-bus.md)), а
payload-контракты — в `pkg/events` ([ADR-0014](0014-pkg-events-bus-contracts.md)). Но
**жизненный цикл потребления** оставался размазан по сервисам и был ненадёжен (аудит
конкурентности, `docs/audit/02-concurrency.md`, и надёжности, `04-error-handling.md`):

- **C1 (critical):** консьюмеры не переподключаются. Цикл `Channel → Qos → Consume →
  for/select` открывает поток доставок один раз; при разрыве соединения с брокером поток
  закрывается, `Run` возвращает ошибку, горутина в `main` логирует её и завершается навсегда.
  Сервис остаётся «healthy» (пробы статические), но конвейер встаёт до ручного рестарта пода.
- **C2/C3 (major):** нет graceful-drain — `main` не ждёт фоновые горутины, `amqpConn` не
  закрывается, in-flight сообщение дообрабатывается с уже отменённым `ctx` → частичные записи
  и дубли.
- **E2 (major):** нет `recover` — паника на одном «отравленном» сообщении убивает процесс,
  после рестарта сообщение переотдаётся (requeue) и роняет сервис снова (crash-loop).
- **C4 (major):** `Connection.Channel()` держит мьютекс через `dialWithRetry` со `time.Sleep`
  до ~31 c — паблишеры сериализуются за залоченным мьютексом при сбое брокера.
- **C5/C6 (minor):** «глухой» `time.Sleep` в бэкоффе игнорирует отмену; `Publish` передаёт
  `context.Background()` в `PublishWithContext`.
- **C8/F7 (minor):** `Qos(10)` бесполезен при последовательной обработке; цикл консьюмера
  скопирован в трёх сервисах, notification дважды разбирает конверт.

Это значимое архитектурное решение по транспорту (RabbitMQ), поэтому фиксируется ADR.

## Options considered

- **Единый каркас `pkg/amqp.Consume` + исправление `Connection`/`Publisher` + `errgroup` в
  `main`.** Централизует supervisor-реконнект, recover, drain-контекст и worker-pool в одном
  месте; сервис передаёт только функцию обработки. Принято.
- **Supervisor-петля и recover отдельно в каждом `main`.** Отклонено: тиражирует C1/E2-фиксы
  по трём сервисам, не решает F7 (копипаста цикла), расходится при правках.
- **Внешний менеджер реконнекта (напр. библиотека).** Отклонено: текущий объём консьюмеров мал
  и однотипен, своя обёртка над `amqp091` достаточна и не добавляет зависимостей.

## Decision

### Каркас `pkg/amqp.Consume(ctx, conn, opts, handler)`

Единственный цикл потребления для всех сервисов. Свойства:

- **Supervisor-реконнект (C1):** `Consume` крутит внешний цикл, переоткрывая канал и поток
  доставок при закрытии `msgs` (блип брокера) с отменяемым экспоненциальным backoff (cap 30s);
  завершается только при отмене `ctx`.
- **`recover` на сообщение (E2):** паника в обработке одной доставки логируется со стеком и
  конвертируется в `Nack(false, false)` (drop, без requeue) — «отравленное» сообщение
  изолируется, процесс жив.
- **Drain-контекст (C3):** каждая доставка обрабатывается с
  `WithTimeout(WithoutCancel(ctx), HandlerTimeout)` — in-flight работа доводится до конца после
  сигнала остановки, а не обрывается на полуслове.
- **Bounded worker-pool (C8):** `errgroup.SetLimit(Concurrency)`, `prefetch = Concurrency`.
  По умолчанию `Concurrency = 1` — строго последовательная обработка, сохраняющая порядок
  событий (`incident.created` перед `incident.updated`, критичный для escalation); параллелизм —
  осознанный opt-in.
- **Единый разбор конверта (F7):** каркас сам разбирает `Envelope`; невалидный конверт → drop;
  handler получает готовый `Envelope` и декодирует payload из `env.Payload`. Двойной разбор
  устранён.
- **Политика подтверждений:** handler возвращает `nil` → `Ack`; ошибку → `Nack` с requeue;
  ошибку, обёрнутую `Drop()`, → `Nack` без requeue. Сохраняет различие политик по сервисам
  (incident/escalation requeue'ят сбои обработки; notification drop'ает) без зашивания в каркас.
  Полноценная стратегия DLQ — вне области (отдельная находка надёжности шины).

### `Connection`/`Publisher`

- `Channel(ctx)` не держит основной мьютекс во время дозвона/сна; реконнект сериализуется
  отдельным `reconnectMu` с double-check, `time.Sleep` → отменяемое ожидание (C4, C5).
- `Publish` пробрасывает переданный `ctx` в `PublishWithContext` (C6).

### Разводка в `main`

Фоновые горутины (консьюмер; у escalation — ещё монитор) собираются в `errgroup`, который
дожидается их **после** `srv.Shutdown`, затем `amqpConn.Close()` (C2). `monitor.step` получает
`recover` на единицу работы (E2 для монитора).

Wire-формат `Envelope`/payload **не меняется** — решение касается жизненного цикла обработки,
не контракта; сообщения в очередях и от старых версий читаются как прежде (не BREAKING).

## Consequences

- После блипа брокера консьюмер восстанавливается сам — устранена «тихая смерть» конвейера
  (главный операционный дефект).
- На SIGTERM in-flight сообщение дообрабатывается и подтверждается; «отравленное» сообщение
  изолируется (drop), а не валит сервис.
- Новый каркас — единственная точка эволюции жизненного цикла консьюмера; будущие сервисы
  с консьюмерами используют `pkg/amqp.Consume`, не копируют цикл.
- `Channel(ctx)`/`Publish` со сменой сигнатуры — внутренний контракт `pkg`; смена не выходит
  за монорепо.
- Открывает: CH10 (`/readyz` с сигналом живости консьюмера — каркас даёт точку для метрики
  состояния), CH14 (переиспользуемый канал паблишера на переработанном `pkg/amqp`).
- Дефолтная последовательная обработка (`Concurrency=1`) сохраняет существующие гарантии
  порядка; рост пропускной способности через параллелизм требует осознанного включения с
  учётом порядка событий.
