# ADR-0013: Защита от SSRF в тенант-задаваемых вебхуках

- Status: Accepted
- Date: 2026-06-13
- Change: guard-webhook-ssrf
- Affected: pkg/ssrf, services/scheduling, services/notification

## Context

Аудит безопасности (`docs/audit/05-security.md`, находка **S2**, major/Medium) выявил SSRF: URL входящего
вебхука Mattermost полностью подконтролен тенант-админу и нигде не ограничивается по адресу назначения.

- На записи (`PUT /api/schedules/v1/tenants/{slug}/notification-config`) сохранялась любая непустая строка.
- На отправке `webhookURLUsable` проверял только синтаксис, после чего notification-под делал `POST` на
  этот URL.

Аутентифицированный тенант-админ мог указать `http://169.254.169.254/...` (cloud-metadata),
`http://localhost:.../`, или адрес внутреннего k8s-сервиса и заставить notification-под обращаться во
внутреннюю сеть от своего имени (blind SSRF: разведка сети, доступ к metadata-эндпоинтам).

## Options considered

- **Блок приватных диапазонов + dial-time guard (принято).** Разрешён любой публичный `https`-хост;
  блокируются loopback/private/link-local/unspecified/multicast/ULA на записи (быстрый 422) и на дозвоне
  (defense in depth). Закрывает вектор SSRF без эксплуатационной нагрузки на тенантов; работает с
  self-hosted Mattermost на произвольных публичных доменах.
- **Allowlist доменов Mattermost.** Строже, но требует ведения списка доменов и ломает self-hosted
  Mattermost на произвольных доменах — легитимный сценарий продукта. Отклонено.
- **Только валидация на записи (без dial-guard).** Не закрывает уже сохранённые небезопасные URL,
  DNS-rebinding и редиректы на приватные адреса. Отклонено.
- **Только dial-guard (без 422 на записи).** Пользователь не получает понятной ошибки и узнаёт о проблеме
  лишь по молчаливому `failed` в журнале доставки. Отклонено.
- **Готовый сторонний SSRF-пакет.** Добавляет зависимость ради небольшого объёма логики; диапазоны и
  dial-guard реализуются стандартной библиотекой. Отклонено на данном этапе.

## Decision

- **Новый общий пакет `pkg/ssrf`** (нужен и scheduling, и notification):
  - `ValidateURL(ctx, raw)` — требует схему `https`, резолвит хост и блокирует все небезопасные диапазоны
    (sentinel-ошибки `ErrNotHTTPS`, `ErrBlockedAddress`, `ErrInvalidURL`).
  - `GuardedDialContext(base)` — обёртка над `net.Dialer`: резолвит хост один раз, отклоняет на любом
    заблокированном IP и дозванивается только на проверенные адреса (соединение не может попасть на
    другой IP, чем проверенный).
  - Предикат диапазонов опирается на стандартную библиотеку: `IsPrivate` (RFC 1918 + `fc00::/7`),
    `IsLinkLocalUnicast` (`169.254.0.0/16` incl. metadata, `fe80::/10`), `IsLoopback`, `IsUnspecified`,
    multicast.
- **scheduling** валидирует непустой `mattermost_webhook_url` на записи → небезопасный URL отклоняется
  с HTTP **422**, сохранённое значение не меняется.
- **notification** применяет `GuardedDialContext` к HTTP-клиенту Mattermost-диспетчера (defense in depth).

## Consequences

- **Уже сохранённые небезопасные URL не мигрируются**: на отправке они начнут блокироваться guarded-dialer
  (доставка `failed` + лог). Тенант-админ, увидев отказ, задаёт корректный публичный https-URL.
- **Валидация на записи требует резолва DNS**: при недоступности DNS или для хоста, резолвящегося только
  во внутренней сети, PUT вернёт 422. Это осознанный компромисс ради SSRF-защиты.
- **`http`-вебхуки больше не принимаются** — только `https`. Для Mattermost это штатное требование.
- Новых env и изменений событий/схемы БД нет; успешный путь для публичных https-URL не меняется.
- Барьер `pkg/ssrf` переиспользуем для других исходящих тенант-задаваемых URL, если такие появятся.
