# ADR-0016: Оптимистичная конкуренция и транзакции на переходах состояний

- Status: Accepted
- Date: 2026-06-14
- Change: db-atomicity-and-state-transitions
- Affected: services/incident, services/escalation

## Context

Аудит работы с БД (`docs/audit/03-database.md`) показал, что доступ к Postgres (pgx)
гигиеничен, но **переходы состояний не защищены от конкуренции и не атомарны**:

- **D1 (major):** монитор эскалаций читает просроченные состояния через
  `FOR UPDATE SKIP LOCKED`, но **вне транзакции** (`pool.Query`) — блокировка снимается сразу
  по завершении SELECT. `UpdateEscalationState` пишет безусловно `WHERE id=$`. Две реплики
  escalation (или перекрытие тиков) прочитают одно просроченное состояние и обе выполнят
  advance → двойная эскалация и двойная публикация `escalation.triggered`.
- **D3 (major):** `PatchStatus` валидирует переход против прочитанного статуса и пишет
  безусловно `WHERE id=$ AND tenant_id=$`. Параллельный PATCH (UI `acknowledge` + конвейерный
  авто-резолв) проходит валидацию против одного `open` и перетирает результат — last-writer-wins,
  стейт-машина обойдена.
- **D2 (major):** составные записи (создание инцидента, смена статуса, advance эскалации) идут
  отдельными автокоммитами — ошибка/отмена между шагами оставляет частичное состояние.

Это значимое архитектурное решение по хранилищу (pgx/Postgres): как проект обеспечивает
корректность переходов состояний при конкуренции — поэтому фиксируется ADR.

## Options considered

- **Guarded-CAS (optimistic concurrency) + транзакционный `withTx` на составные записи.**
  UPDATE дополняется условием на ожидаемое предыдущее состояние
  (`AND current_tier=$exp AND status=$exp` для эскалации; `AND status=$exp` для инцидента) и
  проверкой `RowsAffected()`. `0` строк = «состояние уже изменил другой обработчик» →
  `errs.ErrConflict`. Составные записи (переход + история; создание инцидента + лейблы +
  история + алерт) оборачиваются в одну транзакцию (`Begin`/`Rollback`/`Commit`).
- **Транзакция вокруг `SELECT … FOR UPDATE SKIP LOCKED` на весь батч монитора.** Корректно
  (блокировка живёт весь read-modify-write), но держит транзакцию и row-локи на всё время
  обработки батча, включая синхронные вызовы scheduling → плохо масштабируется, удлиняет
  блокировки и связывает соединение пула.
- **Версионный столбец `version int` + `WHERE version=$`.** Более общий optimistic-lock, но
  требует миграции схемы во всех таблицах состояний и ручного инкремента версии.
- **Пессимистичная блокировка приложения (advisory lock / внешний лок).** Усложняет
  деплой/масштабирование, единая точка отказа.

## Decision

Принят **guarded-CAS + `withTx`** (первый вариант). Конкретно:

- В каждый store добавлен приватный `withTx(ctx, fn func(pgx.Tx) error) error` и узкий
  интерфейс `dbConn` (`Exec`/`Query`/`QueryRow`), которому удовлетворяют и `*pgxpool.Pool`, и
  `pgx.Tx`. Это позволяет переиспользовать один и тот же SQL и из пула (одиночные операции), и
  из транзакции, без дублирования запросов.
- **escalation:** `AdvanceEscalationState(ctx, st, expectedTier, expectedStatus, hist)` —
  guarded-UPDATE + (опционально) `AppendHistory` в одной транзакции; `RowsAffected()==0` →
  `ErrConflict`. `AdvanceOrExhaust` захватывает строку этим CAS **до** публикации события; при
  конфликте тихо пропускает (debug-лог), не публикуя дубликат. `FOR UPDATE SKIP LOCKED` оставлен
  как дешёвая разгрузка реплик, но корректность теперь даёт CAS, а не он.
- **incident:** `TransitionStatus(ctx, …, status, expectedStatus, …, hist)` — guarded-UPDATE
  статуса + `AppendHistory` в одной транзакции; конфликт → `ErrConflict` → HTTP 409.
  `CreateIncidentTx(ctx, inc, labels, hist, ia)` — создание инцидента, лейблов, истории и
  первого алерта в одной транзакции.
- Публикация событий в RabbitMQ выполняется вызывающим кодом **после** успешного commit
  (best-effort, с warn-логом) — внешний I/O не входит в транзакцию.
- Конфликт переиспользует общий сентинел `pkg/errs.ErrConflict` (алиасы `store.ErrConflict` в
  обоих сервисах) — единое значение для `errors.Is` и маппинга на 409
  ([ADR через pkg/errs из CH06](../audit/00-roadmap.md)).

Паттерн **«guarded-CAS по естественному дискриминатору состояния + транзакция на составную
запись»** становится нормой для всех будущих переходов состояний в проекте (incident,
escalation и далее).

## Consequences

- Просроченное состояние эскалации продвигается ровно один раз даже при нескольких
  репликах/перекрытии тиков; дубль `escalation.triggered`/`exhausted` исключён.
- Конкурирующая смена статуса инцидента отклоняется HTTP 409 вместо тихого обхода
  стейт-машины. При единственном писателе поведение не меняется.
- Составные записи атомарны: requeue-переобработка не видит полу-созданный инцидент;
  смена статуса без записи истории невозможна.
- **Схема БД не меняется, миграций нет.** Дискриминаторы перехода берутся из существующих
  колонок (`current_tier`+`status`, `status`). Откат — простой revert кода.
- **Wire-формат событий не меняется** — не BREAKING для очередей и консьюмеров старых версий.
- Ограничение объёма: CAS применён к advance-эскалации и смене статуса инцидента (D1/D3).
  `Stop`/`MaybeResolve`/`PutLabels`/`AddComment` получили только фикс логирования истории (E4),
  расширение CAS на них — отдельная будущая работа (не создаёт двойной эскалации: CAS монитора
  отбросит устаревший advance, если `Stop` опередил).
- Пагинация инцидентов переведена на keyset `(created_at, id)` с непрозрачным курсором (D5) —
  устранены пропуски/дубли на равных таймстампах; формат курсора непрозрачен для клиента
  (не BREAKING для API-схемы).
