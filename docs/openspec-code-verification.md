# Сверка спецификаций OpenSpec с фактическим кодом

> Дата: 2026-06-10. Продолжение [openspec-architecture-review.md](openspec-architecture-review.md):
> каждая находка ревью проверена против кода в `services/` и `frontend/src/`.
> Сверка статическая (чтение кода, без запуска системы).

## 1. Главный вывод

Код фронтенда почти везде следует **бэкенд**-контрактам, а не UI-спекам: разработчики
уже разрешили большинство противоречий раздела 3 ревью в пользу бэкенда (местами это
зафиксировано прямо в комментариях кода). Поэтому **основная работа — обновить шесть
UI-спек под реальность**, а не чинить код.

Однако сверка нашла и **четыре подтверждённых бага в самом коде**, два из которых
полностью ломают доставку уведомлений в развёрнутой конфигурации — а это ядро ценности
on-call-платформы.

## 2. Критичные баги кода (найдены при сверке)

### 2.1 Межсервисные вызовы к scheduling не проходят авторизацию

Все маршруты scheduling, включая `GET .../schedules/{id}/oncall` и
`GET /tenants/{slug}/notification-config`, обёрнуты в JWT-middleware
(`scheduling/cmd/server/main.go:100-147`; `pkg/auth/auth.go:62-87` возвращает 401 без
Bearer-токена; единственный обход — заголовок `X-Admin-Key`).

При этом ни один сервисный клиент не отправляет ни Authorization, ни `X-Admin-Key`:

- `escalation/internal/schedclient/client.go:31-43` — голый GET за текущим дежурным;
- `notification/internal/schedclient/client.go:31-53` — голый GET за конфигом тенанта.

В k8s-манифестах `KEYCLOAK_JWKS_URL` задан **всем** сервисам
(`deploy/k8s/scheduling/configmap.yaml:8` и др.), то есть в боевом деплое аутентификация
у scheduling включена. Последствия цепочкой:

1. escalation получает 401 при резолве дежурного → `triggerTier` логирует warn и публикует
   `escalation.triggered` с **пустым** `oncall_user_id` (`escalator.go:191-201`);
2. notification видит пустой `oncall_user_id` → «skipping user notifications»
   (`notifier.go:85-88`).

**Итог: ни email, ни Mattermost дежурному не уходят вообще.** Работает только
exhausted-оповещение в канал — и то см. 2.2. (В docker-compose JWKS у Go-сервисов не
задан, поэтому локально всё работает — баг проявляется только в k8s.)

### 2.2 Notification получает маскированный Mattermost webhook URL

`GET /tenants/{slug}/notification-config` намеренно маскирует `mattermost_webhook_url`
до `scheme://host` (`scheduling/internal/handler/handler.go:666-673`) — это защита для UI.
Но notification-сервис ходит **на этот же эндпоинт** (`notification/internal/schedclient/client.go:32`)
и кэширует ответ (`tenantcache/cache.go`). Даже если решить проблему 2.1, notification
будет слать Mattermost-сообщения на обрезанный URL `https://host` без пути хука —
все отправки упадут.

Немаскированный вариант существует — legacy `GET /api/schedules/v1/{tenant}/notification-config`
(`handler.go:437-448`, без маскирования) — но клиент его не использует. Нужен либо
внутренний (без маскирования и с сервисной авторизацией) эндпоинт, либо переезд конфига
в БД notification.

### 2.3 Мульти-выбор статусов в фильтре инцидентов возвращает пусто

UI позволяет выбрать несколько статусов и шлёт их одной строкой:
`status: statusFilter.join(',')` (`IncidentListPage.tsx:70`) → `?status=open,acknowledged`.
Бэкенд кладёт значение в равенство: `i.status = $1`
(`incident/internal/store/store.go:74-76`). Строка `"open,acknowledged"` не совпадает ни
с одним статусом → **выбор двух чипов фильтра даёт пустую таблицу инцидентов**.

### 2.4 Сохранение настроек уведомлений затирает webhook URL

Фронтенд грамотно не предзаполняет поле URL маскированным значением
(комментарий в `TenantSettingsPage.tsx:302-304`), но `handleSubmit` всегда шлёт
`mattermost_webhook_url: webhookUrl` (`TenantSettingsPage.tsx:327`) — если администратор
поменял только канал и оставил поле URL пустым, PUT перезапишет сохранённый URL пустой
строкой (бэкенд делает upsert всего объекта, `handler.go:676-688`). Нужно либо опускать
пустое поле на клиенте, либо игнорировать пустой/отсутствующий URL на сервере.

## 3. Прочие расхождения «спека ↔ код», не замеченные ревью

| Что | Спека | Код |
|---|---|---|
| Приём Zabbix | обязательный эндпоинт (`alert-ingestion/spec.md:57`) | **отсутствует полностью** — только alertmanager и grafana (`ingestion/cmd/server/main.go:100-101`) |
| Пути ingestion | `/api/ingestion/v1/{source}` | `/api/ingest/v1/webhook/{source}` |
| Код ошибки публикации | HTTP 500 | HTTP 503 (`alertmanager.go:84`) |
| Таймаут уровня эскалации | `timeout_minutes` | `timeout_seconds` — во всех слоях: API (`escalation/handler.go:78`), домен, фронт (`types.ts:101`) |
| Модель политики | `schedule_id` у политики + правило «tier 1 использует основное расписание» | у политики нет `schedule_id`; каждый tier несёт свой `notify_schedule_id` (`handler.go:74-80`) |
| Поле порядка tier | `order` | `tier_number` |
| Редактирование политики | `PUT /policies/{id}` (UI-спека) | PUT нет; фронт реализует replace: POST новой → перенос default → DELETE старой (`api/escalations.ts:66-86`) — атомарности нет, при сбое посередине остаются две политики |
| Grafana payload | Grafana Alerting (унифицированный) | legacy-формат `state: alerting/ok` (`grafana.go:15-26`) |
| Эндпоинты только в коде | — | `GET /default-policy`, `GET /incidents/{id}/history` (escalation), `GET /incidents/{id}`, `GET /incidents/{id}/alerts` (incident), `DELETE .../overrides/{id}` (scheduling) — в спеках не описаны |
| Мёртвый код фронта | — | `useOnCallNow` и `useScheduleWindow` (`api/schedules.ts`) не используются; второй бьёт в `/oncall?from&to`, который код бэкенда игнорирует (`handler.go:249-261`) — ловушка для будущего использования |

## 4. Вердикты по находкам раздела 3 ревью

| # | Находка ревью | Что в коде | Вердикт |
|---|---|---|---|
| 3.1 | Виджет: несуществующий `/{tenant}/oncall?at=now` | Виджеты per-schedule через `/schedules/{id}/oncall` (`SchedulesPage.tsx:296-307`); уровень эскалации в виджете не показывается | **Чинить UI-спеку** под код |
| 3.2 | Gantt бьёт в `/oncall?from&to` | Страница использует правильный `/shifts` (`SchedulesPage.tsx:311-322`); неправильный эндпоинт остался лишь в мёртвом хуке | **Чинить UI-спеку**; удалить мёртвый хук |
| 3.3 | Профиль: не тот сервис, нет тенанта | Код ходит в notification (`api/profile.ts`, комментарий прямо фиксирует выбор) и даёт селектор тенанта при нескольких командах (`ProfilePage.tsx:202-231`) | **Чинить UI-спеку** под код |
| 3.4 | Default-политика: PATCH `is_default` | Код: `PUT /default-policy` с `policy_id` (`api/escalations.ts:103-110`) — как в бэкенд-спеке | **Чинить UI-спеку** |
| 3.5 | Пути/методы настроек тенанта | Код: `/tenants/{slug}/...`, PUT, enum source валидируется (`handler.go:596-600`) — как в бэкенд-спеке | **Чинить UI-спеку** |
| 3.6 | Маскированный URL затирается | Маскирование есть; от masked-roundtrip фронт защищён, но пустое поле затирает URL (баг 2.4), а notification получает masked URL (баг 2.2) | **Чинить код** (2.2, 2.4) и дополнить спеки |
| 3.7 | Фильтр source отсутствует в API | Подтверждено: ни бэкенд (`store.go:70-90`), ни UI его не реализуют (UI фильтрует только статус и severity); фильтра `label` из бэкенд-спеки тоже нет | **Чинить обе спеки** (либо реализовать) |
| 3.8 | 200-и-в-очередь vs 500 | Код синхронный: publish с 3 ретраями (`pkg/amqp/amqp.go:84-92`), при сбое — 503 и откат dedup-ключа (`handler.go:77-82`) | **Чинить спеку**: модель синхронная, код 503 |
| 3.9 | Lookup токена: БД vs Redis | Код: Redis `HGET oncall:tokens:{hash}` (`ingestion/cmd/server/main.go:142`); scheduling ведёт индекс (`handler.go:617-621`) | **Чинить спеку alert-ingestion** (убрать «таблицу») |
| 3.10 | Tier-1 уведомление не гарантировано | Код: `AssignPolicy` триггерит tier 1 немедленно (`escalator.go:79-83`) — как ожидает notification-спека | **Чинить спеку escalation-policies** (добавить требование) |

## 5. Вердикты по пробелам раздела 4 ревью

| # | Пробел | Как решено в коде |
|---|---|---|
| 4.1 | Severity инцидента | Берётся от **первого** алерта и больше не пересчитывается (`consumer.go:132-137`); алерты, догруппированные позже, на severity не влияют; их лейблы тоже **не мёржатся** в инцидент (MergeLabels вызывается только при создании). Зафиксировать в спеке или реализовать max-севериті |
| 4.2 | Payload эскалации ≠ требования email | Подтверждено: событие несёт только id/tier/oncall (`escalator.go:213-220`); email содержит ID, тенант, tier, время — **без severity, статуса и ссылки**, а «Title» подменён строкой «Escalation tier N» (`notifier.go:149-154`, `email.go:29-34`). Требование спеки не выполнено |
| 4.3 | Эскалация vs resolved/re-open | Наполовину закрыт: Stop срабатывает и на `resolved` (`consumer.go:79-86`) — лучше спеки. Но переоткрытие (`resolved → open`, statemachine это разрешает) эскалацию **не перезапускает** — событие `incident.updated/open` игнорируется |
| 4.4 | Осиротевший resolved-алерт | Тихо игнорируется: `ResolveAlert` вернул пусто → `return nil` (`consumer.go:189-196`). Допустимо, но в спеке не зафиксировано |
| 4.5 | События эскалации в истории инцидента | Не реализовано: история эскалации живёт отдельно (`GET /api/escalations/v1/.../history` — эндпоинт вне спеки), вкладка «История» в UI читает только incident history. UI-спека обещает то, чего нет |
| 4.6 | Каскад удаления тенанта | Подтверждено: `DeleteTenant` удаляет одну строку в БД scheduling (`store.go:256-265`); инциденты, эскалации и журналы уведомлений в других БД осиротевают |
| 4.7 | Право создания тенанта | `POST /tenants` доступен **любому аутентифицированному** пользователю — `RequireTenantMember/Admin` на него не навешан, комментарий «admin key required in practice» (`main.go:107`) ничем не подкреплён в коде |
| 4.8 | Пустой `config.yaml` | Без изменений, актуально |

Где спека и код совпали: rate-limit (5 уведомлений / 600 с, `config.go:62-63`),
дедупликация (SETNX + TTL + откат при сбое публикации, `dedup.go`), ретраи SMTP/Mattermost
(3 попытки с экспоненциальной задержкой), одноразовый показ токена, маскирование URL для
UI, изоляция тенантов через `RequireTenantMember`, гарды фронтенда, оптимистичные
обновления с откатом, горячие клавиши, мобильный список смен.

## 6. Статус изменения `harden-auth-shell`

Не реализовано ни в одной части — все четыре замечания воспроизводятся в коде:

- `parseGroups` — узкая трактовка: `/{tenant}` или `/{tenant}/admins`, подгруппа
  `/team/oncall` не даёт членства (`usePermissions.ts:10-21`), тогда как серверный
  `IsMember` принимает любой префикс `/{tenant}/` (`pkg/auth/auth.go:36-44`) —
  расхождение, ради которого изменение и создавалось, подтверждено;
- `CallbackPage` при ошибке делает `navigate('/')` — риск цикла редиректов
  (`CallbackPage.tsx:20`);
- `SessionBanner` запускает 30-секундный отсчёт на **любой** `silentRenewError`
  (`SessionBanner.tsx:39`);
- диагностики «непустой groups при пустых permissions» нет.

## 7. Приоритизированный план

**P0 — доставка уведомлений сломана в k8s-деплое:**

1. Сервисная авторизация для escalation→scheduling и notification→scheduling
   (передавать `X-Admin-Key`, либо выделить внутренние эндпоинты без JWT) — баг 2.1.
2. Немаскированный конфиг для notification (внутренний эндпоинт или перенос конфига
   в БД notification) — баг 2.2.
3. Фильтр по нескольким статусам: `status IN (...)` на бэкенде или по одному значению
   с клиента — баг 2.3.
4. Не затирать `mattermost_webhook_url` пустым значением при сохранении формы — баг 2.4.

**P1 — функциональные долги:**

5. Zabbix-ингест: реализовать или официально вычеркнуть из спеки (сейчас даже токен
   с `source: zabbix` создать можно, а слать вебхуки некуда).
6. Обогатить `escalation.triggered` полями `incident_title`, `severity`, `status`
   (incident уже кладёт их в `incident.created` — escalation может протаскивать их в
   состояние) и довести email до требований спеки.
7. Реализовать `harden-auth-shell` (план и дельта готовы, задачи не начаты).
8. Удалить мёртвые хуки `useOnCallNow`/`useScheduleWindow` или починить второй на `/shifts`.

**P2 — синхронизация спецификаций (через изменения OpenSpec):**

9. Обновить шесть UI-спек под фактические контракты (вердикты раздела 4: пункты 3.1-3.5, 3.7).
10. Обновить бэкенд-спеки: пути ingestion, 503, Redis-lookup токена, `timeout_seconds`,
    модель политики без `schedule_id`, немедленный tier-1 триггер, Stop на resolved,
    недокументированные эндпоинты.
11. Зафиксировать решения по пробелам: severity «от первого алерта», поведение
    переоткрытия, отсутствие каскада удаления тенанта, права на `POST /tenants`.
