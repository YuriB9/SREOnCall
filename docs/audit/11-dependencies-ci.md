# Аудит SREOnCall — Область 11: Зависимости / CI

Дата: 2026-06-13
Область: go.work, уязвимости, lint-конфиг, пайплайны.
Применённые скилы: `golang-dependency-management`, `golang-lint`, `golang-continuous-integration`.

Набор зависимостей дисциплинированный (маленький, вычитанный, актуальный), `go.sum` закоммичен. Но **`govulncheck` запущен в рамках аудита и нашёл две достижимые из кода уязвимости — обе в пути аутентификации.** Это не гипотеза: сканер подтвердил, что код их вызывает. Они не были замечены ровно потому, что **CI-пайплайна нет** — ничто не запускает ни `govulncheck`, ни линтер, ни `go vet`, ни `-race`, ни автообновление зависимостей.

## Детализация

### DC1 — Две достижимые уязвимости в пути аутентификации — **major**

`govulncheck ./...` (запущен на `pkg` и на сервисном модуле) сообщил: **«Your code is affected by 2 vulnerabilities from 2 modules»** — то есть это вызываемый, достижимый код, а не просто наличие в дереве:

**1. GO-2025-3553 — `github.com/golang-jwt/jwt/v5@v5.2.1`** (исправлено в **v5.2.2**).
Достижимо из `auth.Middleware → jwt.ParseWithClaims` ([pkg/auth/auth.go:114](pkg/auth/auth.go#L114)). Уязвимость парсинга JWT (чрезмерная аллокация при разборе специально сформированного токена → DoS). Поскольку валидация токена стоит на входе **каждого** защищённого запроса, это прямой вектор отказа в обслуживании неаутентифицированным запросом.

**2. GO-2025-3376 — `github.com/MicahParks/jwkset@v0.5.19`** (исправлено в **v0.6.0**; транзитивно через `keyfunc/v3@v3.3.5`).
Достижимо из `auth.Middleware → keyfunc.NewDefault` ([pkg/auth/auth.go:94](pkg/auth/auth.go#L94)). Суть: HTTP-клиент JWK Set при refresh только перезаписывает и **дополняет** локальный кэш ключей, но не удаляет устаревшие. Практически это значит, что **отозванный/ротированный в Keycloak ключ может остаться валидным** в кэше сервиса — ослабление гарантий ротации ключей в пути аутентификации. Прямо усиливает находки области 5 (S-кластер, валидация JWT).

Обе закрываются бампом версий:

```bash
# в каждом модуле, где используется auth (pkg + сервисы)
go get github.com/golang-jwt/jwt/v5@v5.2.2
go get github.com/MicahParks/jwkset@v0.6.0   # либо обновить keyfunc/v3 до версии, тянущей jwkset>=0.6.0
go mod tidy
go run golang.org/x/vuln/cmd/govulncheck@latest ./...   # подтвердить, что чисто
```

Это ровно тот класс проблем, который `govulncheck` в CI ловит в день появления адвизори (DC2). Приоритет — немедленный: достижимый код, путь auth, фикс однострочный.

> `govulncheck` дополнительно отметил уязвимости в импортируемых пакетах/модулях, которые **код не вызывает** — их можно подчистить при общем апдейте, но они не достижимы и не срочны.

### DC2 — Нет CI-пайплайна — **major**

В репозитории нет `.github/workflows/` (найдены только assets внутри скилов). Значит, не выполняется автоматически ничего из обязательного по `golang-continuous-integration`:

- **`govulncheck`** — и именно поэтому DC1 пролежал незамеченным;
- **`golangci-lint`** / **`go vet`** — статанализ;
- **`go test -race -shuffle=on`** — гонки и зависимости тестов (ср. область 2 — конкурентный код вообще не гоняется под `-race`);
- **`go mod tidy && git diff --exit-code`** — честность go.mod;
- **SAST** (`gosec`/CodeQL) — пересекается с областью 5;
- интеграционные/e2e-гейты (ср. T1/T2 области 8 — тесты есть, но никем не запускаются).

Это та же первопричина, что T1 (область 8), но в разрезе зависимостей и безопасности: без автоматического гейта уязвимости и регрессии обнаруживаются только вручную и случайно.

**Фикс.** GitHub Actions с матрицей по модулям `go.work`, порядок из скила: `test` (`-race -shuffle=on`, `go mod tidy`-проверка) → `lint` (`golangci-lint`) → `security` (`govulncheck`, `gosec`/CodeQL) → `deps` (Dependabot/Renovate). Пины major-версий экшенов, `permissions:` по least-privilege, `fail-fast: false` в матрице.

### DC3 — Нет конфигурации линтера — **minor**

`.golangci.yml` отсутствует во всём репозитории. Даже при ручном запуске `golangci-lint` работает на дефолтном наборе без проектных правил, поэтому накопленные минорные находки прочих областей **никем не энфорсятся**: ключ логов `err`/`error` (E5), package-комментарии (N1), стуттер `pkg.Pkg` (N2, через `revive`), `[]interface{}`→`[]any` (через `modernize`), непокрытые ошибки `_ =` (E4, через `errcheck`).

**Фикс.** Добавить `.golangci.yml` (рекомендованный набор — из скила `golang-lint`: `errcheck`, `govet`, `staticcheck`, `revive`, `gosec`, `bodyclose`, `sqlclosecheck`, `nilerr`, `modernize`, `misspell`, `errname`, `paralleltest`). Бо́льшая часть «минорных» находок всего аудита закрывается включением линтера в CI — это главный рычаг по объёму.

---

### DC4 — Нет автообновления зависимостей и пиннинга инструментов — **minor**

- **Нет Dependabot/Renovate.** Патчи безопасности (как DC1) не всплывают автоматически — обновление зависимостей целиком на ручной инициативе. Renovate предпочтителен для монорепо на `go.work` (умеет `gomodTidy`, группировку, multi-module).
- **Нет `tool`-директив в go.mod** (Go 1.24+) и нет `tools.go`. `golangci-lint`/`govulncheck` не запинены по версии → нерепроизводимые прогоны.

**Фикс.**
```bash
go get -tool github.com/golangci/golangci-lint/v2/cmd/golangci-lint@latest
go get -tool golang.org/x/vuln/cmd/govulncheck@latest
```
и `renovate.json`/`dependabot.yml` (шаблоны — в скиле `golang-continuous-integration`), с группировкой minor/patch и отдельными PR на major.

### DC5 — Нет Dockerfile в репозитории — **minor**

В `deploy/k8s/*/deployment.yaml` образы только **ссылаются**, но `Dockerfile` в репозитории нет. Следствия: сборка образа не воспроизводима из исходников, в пайплайн нельзя встроить сканирование образа (Trivy) и генерацию SBOM/provenance, посмортность контейнера (multi-stage, non-root, пины base-image) не контролируется. Возможно, сборка живёт во внешней системе — поэтому minor, но это слепое пятно цепочки поставки.

**Фикс.** Добавить multi-stage `Dockerfile` на сервис (distroless/non-root, пин base) и `docker.yml` с Trivy + SBOM (шаблон — в скиле).

---

### DC6 — `tests/e2e/go.sum` отсутствует — **trivial**

У модуля `tests/e2e` есть `go.mod`, но нет `go.sum` (у остальных 6 модулей есть). Если e2e тянет внешние зависимости — `go mod verify` для него не сработает, и сборка нерепродуцируема. Если зависимостей нет (только stdlib) — это нормально.

**Фикс.** Проверить (`cd tests/e2e && go mod tidy`); если появятся записи — закоммитить `go.sum`.

## Что сделано хорошо (для контекста)

- **`go.sum` закоммичен для всех 6 собираемых модулей** (`pkg` + 5 сервисов) — криптографические контрольные суммы на месте, `go mod verify` работает, защита от подмены в proxy есть.
- **Набор зависимостей образцово дисциплинирован:** chi/v5 `5.3.0`, pgx/v5 `5.10.0`, go-redis/v9 `9.20.0`, amqp091-go `1.11.0`, golang-migrate/v4 `4.17.1`, google/uuid `1.6.0`, golang-jwt/jwt/v5, keyfunc/v3 — всё от авторитетных организаций, без обскурных пакетов, без ORM. Видно соблюдение принципа «сначала проверь, покрывает ли stdlib» — минимальная поверхность зависимостей.
- **`go`-директивы согласованы** (`go 1.26.4` во всех модулях + `go.work`, `toolchain go1.26.4`) — версионного скоса между модулями нет.
- **`go.work` + per-module `replace`** настроены корректно (область 1) — модули собираются и в воркспейсе, и автономно.
- Версии зависимостей **свежие** — отставание небольшое; апдейт до фикса DC1 не тянет за собой мажорных миграций.

## Рекомендованный порядок исправлений

1. **DC1** — немедленно: `go get jwt/v5@v5.2.2` + `jwkset@v0.6.0`, `go mod tidy`, перепроверить `govulncheck`. Достижимые CVE в auth, однострочный фикс.
2. **DC2** — поднять CI (test+lint+security+deps): включает `govulncheck` как постоянный гейт, чтобы следующий DC1 ловился автоматически. Один пайплайн закрывает рычаги областей 4–11.
3. **DC3 + DC4** — `.golangci.yml` + `tool`-директивы + Renovate: автоматизирует львиную долю «минорных» находок всего аудита.
4. **DC5 + DC6** — Dockerfile с Trivy/SBOM; проверить e2e `go.sum`.

> Кросс-ссылки: DC1 — прямое продолжение S-кластера (область 5, валидация JWT/ключи); DC2 — тот же корень, что T1 (область 8), но в разрезе vuln/lint; DC3 закрывает энфорсментом E5 (область 4), N1/N2/N5 (область 9), E4; DC2 даёт `-race` для C-кластера (область 2) и бенч-гейт для области 10. CI — это общий рычаг, которого не хватает большинству областей аудита.
