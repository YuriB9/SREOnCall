# Протокол запуска OpenSpec-чейнджа в новой сессии

Каждый чейндж из [00-roadmap.md](00-roadmap.md) реализуется в **отдельной холодной сессии** Claude.
Холодная сессия не помнит аудит — поэтому вся память живёт в коммитнутых файлах:
`docs/audit/*` (находки с `file:line`), `00-roadmap.md` (план + зависимости + статус),
`openspec/specs/*` (capability-спеки), `openspec/config.yaml` (контекст проекта + rules).

Этот протокол гарантирует, что ничего не потеряется на стыке сессий.

**Репозиторий:** `https://github.com/YuriB9/SREOnCall.git` · интеграционная ветка — `main`.
**Каждый чейндж = отдельная ветка**, которая после архивации **мержится в `main`**.
**GitHub Actions ещё ни разу не настраивался** — его заводит CH01. До CH01 нет CI-гейта:
проверки гоняются вручную, мерж в `main` прямой. После CH01 — мерж через PR, когда Actions зелёный.

## 1. Промпт для старта (копировать в новую сессию)

Подставь номер и слаг из roadmap. Больше ничего вписывать не нужно — все данные сессия достанет сама.

```
Реализуем находки аудита через OpenSpec, по одному чейнджу за сессию.
Сейчас делаем CH<NN> — <slug> из docs/audit/00-roadmap.md.

Выполни протокол ПЕРЕД любыми изменениями (docs/audit/CHANGE-KICKOFF.md):

1. Прочитай docs/audit/00-roadmap.md → строку CH<NN>: scope, находки,
   depends-on, фаза, риск. Объём чейнджа = строго находки CH<NN> из
   «Матрицы покрытия». Соседние находки НЕ трогаем — это другие чейнджи.
2. Прочитай разделы docs/audit/<NN>-*.md, на которые ссылается CH<NN> —
   там точные находки с file:line и предложенными фиксами (источник истины).
3. PRE-FLIGHT зависимостей: проверь, что каждый depends-on чейндж уже
   заархивирован в openspec/changes/archive/ И влит в main. Если нет —
   СТОП, сообщи мне.
4. ВЕТКА: git checkout main && git pull, затем git checkout -b
   ch<NN>-<slug>. Вся работа чейнджа — только в этой ветке.
5. Прочитай openspec/config.yaml (контекст + rules) и затронутые
   capability-спеки в openspec/specs/.
6. Создай чейндж через opsx (/opsx:propose или /opsx:new), имя
   <slug>. Артефакты по rules репозитория: на русском;
   в proposal Impact перечисляет затронутые сервисы и события RabbitMQ;
   изменения API/событий помечай **BREAKING**; capability-дельта только
   если меняется наблюдаемое поведение продуктовой capability (иначе —
   infra/tooling без дельты); ADR — только для значимого арх-решения.
7. В tasks.md привяжи каждую задачу к находке и file:line
   (например: «C1 — supervisor-петля реконнекта, pkg/amqp/amqp.go»).
8. Реализуй → /opsx:verify → прогон проверок (см. раздел 3).
9. /opsx:archive, обнови статус CH<NN> в docs/audit/00-roadmap.md
   (дашборд + строка чейнджа).
10. ВЛЕЙ В MAIN: закоммить всё в ветку, затем влить ch<NN>-<slug> в main
    (см. раздел «Git-воркфлоу»: до CH01 — прямой merge --no-ff + push;
    после CH01 — PR, мерж по зелёному Actions). Сообщи мне результат.

Сначала покажи план (proposal + tasks), реализацию начинай после моего ОК.
```

## 2. Pre-flight чеклист (сессия проверяет до создания чейнджа)

- [ ] **Зависимости закрыты.** Все `depends-on` из roadmap-строки заархивированы в `openspec/changes/archive/`. Критический путь: `CH01 → CH05 → CH07 → CH10 → CH11 → CH12`. Не начинать CH07 без CH05, CH10 без CH07, CH08-CAS без понимания CH07-requeue.
- [ ] **Объём заморожен.** Берём ровно находки из «Матрицы покрытия» roadmap для этого CH. Если по дороге всплывает соседняя находка — записать в заметку для её чейнджа, **не расширять текущий**.
- [ ] **Прочитан исходник аудита.** Не работать «по памяти промпта» — открыть `docs/audit/<NN>-*.md` и взять оттуда `file:line`. Аудит писался по снимку кода — **перепроверить, что строки/символы ещё существуют** (код мог измениться предыдущими чейнджами).
- [ ] **Понятна capability-привязка** (см. таблицу ниже): меняем ли мы гарантию продуктовой capability (→ MODIFIED спек) или это чистый infra/tooling-рефактор (→ без capability-дельты, как было в `harden-auth-shell`).

## 3. Проверки перед архивацией (Definition of Done)

Прогнать то, что относится к изменению (не обязательно всё):

| Что менялось | Команды |
|--------------|---------|
| Любой Go-код | `go build ./...`, `go vet ./...`, `go test ./...` |
| Конкурентность (CH07, CH16) | `go test -race ./...` |
| Поведение консьюмеров/хендлеров/store (CH07–CH10) | `go test -tags integration ./...` (нужны Postgres/Redis/RabbitMQ из docker-compose) |
| Зависимости/auth (CH02, CH03) | `govulncheck ./...` — должно быть чисто |
| Любой чейндж после CH01 | `golangci-lint run` |
| Контракты событий (CH05, CH13) | проверить, что publisher и consumer используют один тип; e2e при наличии |

Затем: `/opsx:verify` → `/opsx:archive` (архивация синхронизирует delta-спеки в `openspec/specs/`).

## 3a. Git-воркфлоу (ветка на чейндж → мерж в main)

Репозиторий: `https://github.com/YuriB9/SREOnCall.git`. Интеграционная ветка — `main`.
Один чейндж живёт в одной ветке и вливается в `main` только после `/opsx:archive`.

**Старт ветки** (шаг 4 промпта):
```bash
git checkout main && git pull
git checkout -b ch<NN>-<slug>      # напр. ch07-consumer-resilience
```

**В процессе:** коммитить осмысленными порциями в ветку; артефакты OpenSpec и код — вместе.

**Финал — влить в main:**

- **До того как CH01 сделан (Actions ещё нет):** мерж прямой, после ручных проверок из раздела 3.
  ```bash
  git checkout main && git pull
  git merge --no-ff ch<NN>-<slug>      # --no-ff: сохранить факт чейнджа в истории
  git push origin main
  git branch -d ch<NN>-<slug>          # и удалить удалённую, если пушилась
  ```
- **После CH01 (Actions настроен):** мерж через PR.
  ```bash
  git push -u origin ch<NN>-<slug>
  gh pr create --base main --title "CH<NN>: <slug>" --body "<ссылки на docs/audit и change>"
  # дождаться зелёного Actions → смержить
  gh pr merge --merge --delete-branch
  ```

**Правила:**
- Не начинать ветку чейнджа, пока его `depends-on` не **влиты в main** (а не только заархивированы) — иначе ветка не увидит нужного кода.
- Ветвиться всегда от свежего `main` (`git pull` перед `checkout -b`).
- Один чейндж = одна ветка = один мерж. Не смешивать находки разных CH в одной ветке.
- `--no-ff` (или squash-merge в PR) — чтобы в истории `main` был виден отдельный чейндж.

> Проверено (2026-06-13): `origin` = `https://github.com/YuriB9/SREOnCall.git`; интеграционная
> ветка — `main` (локальная и `origin/main` синхронны); ветки `master` в репозитории нет.
> Историческая модель ветвления — `epicN/sprintM` / `backend/sprintM` / `frontend/sprintM`,
> всё вливается в `main`. Аудит-чейнджи ветвим как `ch<NN>-<slug>` от `main` и мержим в `main`.

## 4. Хэндофф — что обновить, чтобы следующая сессия не потерялась

После архивации **в той же сессии**:

1. **Обнови `docs/audit/00-roadmap.md`**: пометь CH<NN> как `✅ done` (добавь статус-маркер в строку чейнджа). Это сигнал следующей сессии, что зависимость закрыта.
2. Если по ходу нашлась **новая** проблема вне объёма — допиши строку-заметку в roadmap (раздел «Backlog находок») или в соответствующий `docs/audit/<NN>-*.md`, не реализуя.
3. Если изменился публичный контракт (событие/API/`pkg`-сигнатура), которым пользуются **другие** запланированные чейнджи — отметь это в их строке roadmap (например, «CH11 теперь опирается на метрики из CH07»).

> Память между сессиями = коммит. Всё, что не записано в `docs/audit/*` или `openspec/`, в следующей сессии не существует.

## 5. Capability-привязка по чейнджам (стартовая гипотеза — подтвердить против `openspec/specs/`)

Многие чейнджи — **кросс-каттинг/инфра** (`pkg/*`, CI), у них нет продуктовой capability и дельта-спек не нужен (прецедент: `harden-auth-shell` → «Новых capability не вводится»). Меняем спек только там, где меняется **наблюдаемая гарантия** фичи.

| Чейндж | Тип | Вероятная capability (MODIFIED) |
|--------|-----|----------------------------------|
| CH01, CH02, CH19 | infra/tooling | нет (без дельты) |
| CH03 `harden-auth` | infra (pkg/auth) | нет; задокументировать требования в Impact (как harden-auth-shell) |
| CH04 `guard-webhook-ssrf` | поведение | `notification-dispatch`, `tenant-settings-ui`/`tenant-management` (валидация webhook URL) |
| CH05 `bus-contracts` | infra (рефактор без смены поведения) | нет дельты, но **BREAKING-внимание** если меняются JSON-теги |
| CH06 `config+httpclient` | infra | нет |
| CH07 `consumer-resilience` | поведение (надёжность доставки) | `alert-ingestion`, `incident-management`, `escalation-policies`, `notification-dispatch` — гарантии at-least-once/восстановления |
| CH08 `db-atomicity` | поведение (переходы состояний) | `incident-management` (статусы), `escalation-policies` (нет двойной эскалации) |
| CH09 `store+pool` | infra | нет |
| CH10 `httpserver+readiness` | infra + поведение проб | нет продуктовой; возможно новая `observability`-capability или без дельты |
| CH11 `metrics+alerts` | observability | новая capability `observability`/`monitoring` **или** без дельты (решить с владельцем) |
| CH12 `log-correlation` | infra | нет |
| CH13 `distributed-tracing` | infra | нет (или `observability`) |
| CH14 `bus-publish-perf` | perf (без смены поведения) | нет дельты; benchstat в коммите |
| CH15 `ingestion-throughput` | perf | нет дельты; benchstat; следить, чтобы дедуп/порядок не сломались |
| CH16 `tenantcache-singleflight` | perf/correctness | нет (или `notification-dispatch`) |
| CH17 `test-hardening` | infra/tests | нет |
| CH18 `docs-and-style` | стиль/доки | нет |

При сомнении «новая capability или нет» — спросить владельца проекта, а не вводить спек по своей инициативе (правило ADR/спека в `config.yaml`: только значимые решения).

## 6. Грабли, специфичные для этого репо

- **Русский язык** в `proposal.md`/`spec.md`/`design.md`/ADR (правило `config.yaml`). UI-тексты тоже на русском.
- **`go.work`-монорепо:** правки в `pkg/*` затрагивают все сервисы. После смены `pkg`-сигнатуры — собрать **все** модули (`go build ./...` в каждом или из корня воркспейса), не только один сервис. `go mod tidy` в каждом задетом модуле.
- **Impact обязан перечислять сервисы и события RabbitMQ** — это явное правило `config.yaml` для `proposal`.
- **BREAKING для событий шины:** смена JSON-тегов/полей `Envelope`/payload — breaking для консьюмеров; пометить и описать миграцию (старые сообщения в очереди!).
- **Спек MODIFIED копируется целиком** из `openspec/specs/<capability>/spec.md` (правило `config.yaml`), затем правится — нельзя писать только дифф.
- **Каждое требование спека = хотя бы один сценарий `#### Scenario` WHEN/THEN**.
- **CI ещё нет до CH01** — пока он не сделан, проверки (`-race`, `govulncheck`, lint) гоняются **вручную** в каждой сессии. После CH01 — это гейт PR.
- **Сверка спек↔код** ведётся в `docs/spec-vs-code-audit.md` — при изменении гарантий обновить и его, если релевантно.

## 7. Порядок и параллелизм (из roadmap)

- **Сразу после CH01** параллелятся: CH02, CH03, CH04, CH05, CH06, CH09, CH16, CH18, CH19.
- **Критический путь:** CH01 → CH05 → CH07 → CH10 → CH11 → CH12.
