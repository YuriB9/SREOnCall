# Аудит SREOnCall — Область 7: Наблюдаемость

Дата: 2026-06-13
Область: `slog`, prometheus-метрики, покрытие сигналов наблюдаемости.
Применённый скил: `golang-observability`.

Базовый фундамент заложен правильно: структурное JSON-логирование через `slog`, HTTP-метрики на гистограмме (а не Summary), доменные метрики дедупликации, авто-экспорт рантайм-метрик Go. Но для системы **on-call/алертинга из 5 сервисов** покрытие узкое: наблюдается только HTTP-фронт, а сам событийный конвейер (ради которого система и существует) — слеп. Пробы здоровья декоративные, распределённой трассировки нет, логи не коррелируются с запросами, алертов/дашбордов в репозитории нет.

Особо важный контекст: в области 2 найдено, что консьюмеры **молча умирают** при разрыве RabbitMQ (C1). Наблюдаемость — это ровно тот слой, который должен такое ловить; сейчас он этого не делает (O1, O2).

## Детализация

### O1 — Пробы здоровья ничего не проверяют — **major**

Во всех пяти сервисах liveness и readiness — это статический `200`:

```go
r.Get("/healthz", func(w http.ResponseWriter, _ *http.Request) { w.WriteHeader(http.StatusOK) })
r.Get("/readyz",  func(w http.ResponseWriter, _ *http.Request) { w.WriteHeader(http.StatusOK) })
```

([incident/main.go:93-94](services/incident/cmd/server/main.go#L93-L94), идентично в остальных). Пробы возвращают «здоров/готов», пока жив сам HTTP-листенер, и **не отражают состояние зависимостей**:

- **`/readyz` не проверяет БД/Redis/AMQP.** Если Postgres недоступен, под всё равно «ready» → k8s продолжает слать на него трафик, который падает на каждом запросе. Readiness-проба, не зависящая от backing-сервисов, не выполняет свою единственную задачу.
- **Главное (связка с C1, область 2): `/readyz` не отражает живость консьюмера.** Когда консьюмер молча умирает после разрыва RabbitMQ, HTTP-часть жива → readyz=200 → ни k8s, ни оператор не видят, что конвейер встал. Это превращает скрытый сбой в необнаружимый.

**Фикс.** Разделить пробы:
- `/healthz` (liveness) — лёгкая, без внешних зависимостей (как сейчас) — ОК.
- `/readyz` (readiness) — проверять критичные зависимости: `pool.Ping(ctx)`, `rdb.Ping(ctx)`, состояние AMQP-соединения и флаг «консьюмер запущен и не падал» (через `amqp091.Connection.NotifyClose` или атомарный признак, выставляемый supervisor-петлёй из фикса C1). Возвращать `503`, когда любая критичная зависимость недоступна.

### O2 — Конвейер событий без метрик — **major**

Инвентаризация метрик в проекте:
- `pkg/metrics`: `http_requests_total`, `http_request_duration_seconds` — только HTTP-фронт.
- `ingestion/dedup`: `dedupHits`, `dedupMisses` — единственная доменная метрика (и хороший образец).
- Авто: рантайм/процессные метрики Go (через `promhttp.Handler()` на дефолтном регистре) — есть.

Чего **нет** вообще — метрик самого продукта. Для платформы on-call это критичный пробел: невозможно построить SLO и алерты на бизнес-поток. Не инструментированы:
- **ingestion**: принято алертов, нормализовано, отброшено как дубль (есть только hit/miss дедупа, но не «алертов получено»), ошибок публикации в шину;
- **incident**: создано/обновлено/закрыто инцидентов, ошибок группировки;
- **escalation**: эскалаций запущено/продвинуто/исчерпано, провалов `GetOnCall`, размер backlog'а просроченных state'ов (`ListExpiredStates`);
- **notification**: уведомлений отправлено/провалено по каналам (email/mattermost), сработка rate-limit, ретраи;
- **шина (все консьюмеры)**: сообщений consumed/ack/nack/requeue, длительность обработки, publish-ошибок и ретраев `pkg/amqp`.

Без этого нельзя ответить на базовые операционные вопросы on-call системы: «доходят ли уведомления?», «не растёт ли очередь?», «сколько эскалаций исчерпалось без ответа?». В связке с C1 нет даже метрики «консьюмер жив / сообщений обработано за минуту», по падению которой можно было бы алертить на тихую смерть.

Отдельно: метрики пула pgx (`pgxpool.Stat()` — acquired/idle/total, время ожидания) не экспортируются (ср. D4, область 3).

**Фикс.** Ввести доменные метрики по образцу `dedup` (counter на операции/ошибки, histogram на длительность обработки сообщения, gauge на backlog). Минимальный набор «золотых сигналов» для шины: `<service>_messages_processed_total{result=ack|nack|requeue}`, `<service>_message_processing_seconds`, `<exchange>_publish_total{result}`. По соглашению из скила — писать PromQL и alert-правила комментарием над объявлением метрики.

### O3 — Нет распределённой трассировки — **major**

Поиск `otel`/`opentelemetry`/`trace` по коду пуст. При этом топология — классическая распределённая: алерт проходит `ingestion → (alerts) → incident → (incidents) → escalation → (escalations) → notification`, четыре сервиса и три обмена RabbitMQ. Без трассировки:
- нельзя проследить путь одного алерта/инцидента сквозь сервисы и понять, где он «застрял» или потерялся;
- нельзя разложить латентность end-to-end (HTTP → БД → publish → consume → отправка);
- envelope (`pkg/amqp.Envelope`) несёт `id` (uuid), но **не несёт trace-context**, поэтому даже корреляция по ID через шину невозможна автоматически.

По скилу (п.7–9) для нового микросервисного проекта трассировку настраивают рано. Это build-out, а не баг, но для системы такого профиля — главный отсутствующий сигнал.

**Фикс.** Подключить OpenTelemetry SDK: `otelhttp` на входящие/исходящие HTTP, span'ы на методы стора и вызовы клиентов (`schedclient`/`incclient`/`keycloak`), и — ключевое для этого проекта — **пробрасывать trace-context через AMQP-конверт**: класть `traceparent` в `Envelope` (новое поле или AMQP headers) на `Wrap`, восстанавливать на `Unwrap`/в консьюмере. Тогда трасса не рвётся на границе шины. Контекст уже корректно пробрасывается во все слои (см. область 3), так что почва готова.

### O4 — Логи без корреляции с запросом/трассой — **minor**

Контекстных логов нет: `InfoContext`/`ErrorContext`/`WarnContext` — 0 вхождений. Все вызовы вида `logger.Error("...", "err", err)` пишут без `ctx`, поэтому в записи не попадают ни `request_id`, ни (в будущем) `trace_id`. При этом `chiMiddleware.RequestID` ставит request-id в контекст — но только в ingestion и incident (O6), и **ни один лог его не извлекает и не пишет**. То есть идентификатор есть, а в логах его нет — корреляция невозможна.

**Фикс.** Перейти на `slog.*Context(ctx, ...)` и handler, который достаёт `request_id`/`trace_id` из контекста и добавляет в каждую запись (либо `logger.With("request_id", ...)` в middleware, прокинутый в `ctx`). После подключения O3 — мост `otelslog` для авто-инъекции `trace_id`/`span_id`.

### O5 — Нет алертов, дашбордов и scrape-конфига в репозитории — **minor**

В `deploy/k8s/` есть манифесты сервисов и ingress, но нет ни `PrometheusRule`/alert-правил, ни `ServiceMonitor`/scrape-конфига, ни Grafana-дашбордов (поиск по `deploy`/`docs` ничего не дал, кроме ADR про дедуп). `/metrics` экспонируется каждым сервисом, но в репозитории ничто не описывает, кто его скрейпит и какие условия поднимают алерт. По «Definition of Done» скила метрика не считается готовой, пока её PromQL не заведён в дашборд и правило алертинга.

**Фикс.** Добавить в `deploy/` `ServiceMonitor` (или scrape-аннотации), `PrometheusRule` с «золотыми сигналами» (latency P99, error-rate, отсутствие обработки сообщений, насыщение пула), и базовые дашборды. Готовые правила для Postgres/Redis/RabbitMQ — из [awesome-prometheus-alerts](https://samber.github.io/awesome-prometheus-alerts/).

---

### O6 — Непоследовательный RequestID middleware — **minor**

`r.Use(chiMiddleware.RequestID)` стоит только в ingestion ([main.go:91](services/ingestion/cmd/server/main.go#L91)) и incident; escalation, notification и scheduling его не подключают. Это часть общего разнобоя middleware (ср. F4 области 1, E1 области 4: `Recoverer` тоже не везде). Даже там, где request-id ставится, он не используется (O4) — но без него корреляция невозможна в принципе.

**Фикс.** Единый набор middleware (`RequestID` + `Recoverer` + metrics + request-log) в общем `pkg/httpserver`, применяемый всеми сервисами одинаково.

## Что сделано хорошо (для контекста)

- **Структурное JSON-логирование через `slog`** с настраиваемым уровнем и установкой default ([pkg/logger](pkg/logger/logger.go)) — правильный фундамент (исключение — scheduling, который строит логгер мимо `pkg/logger`, F5 области 1).
- **Латентность на Histogram, а не Summary** ([metrics.go:17-20](pkg/metrics/metrics.go#L17-L20)) с `DefBuckets` — корректный выбор: агрегируется между инстансами, поддерживает `histogram_quantile`. Ровно как рекомендует скил.
- **Доменные метрики дедупликации** (`dedupHits`/`dedupMisses`) — хороший образец инструментирования, который надо растиражировать на остальной конвейер (O2).
- **Рантайм- и процессные метрики Go** экспонируются автоматически через `promhttp.Handler()` на дефолтном регистре — GC, heap, goroutines, FD видны без доп. кода.
- **Single-handling rule в логах соблюдён** (логируем или возвращаем, не оба — см. область 4): логи не зашумлены дубль-записями одной ошибки.

## Рекомендованный порядок исправлений

1. **O1** — content-aware `/readyz` (БД/Redis/AMQP + живость консьюмера): напрямую закрывает «слепое пятно» вокруг C1, минимальный объём.
2. **O2** — доменные метрики конвейера и шины: даёт основу для SLO и алертов; начать с «сообщений обработано/ack/nack» и «уведомлений отправлено/провалено».
3. **O5** — завести `ServiceMonitor` + `PrometheusRule` (хотя бы золотые сигналы), чтобы метрики из O1/O2 реально кого-то будили.
4. **O3** — OpenTelemetry с проносом trace-context через AMQP-конверт: больший объём, но единственный способ видеть поток end-to-end.
5. **O4 + O6** — единый middleware-набор и `*Context`-логирование с `request_id`/`trace_id`.

> Кросс-ссылки: O1/O2 — наблюдательная половина C1 (область 2, тихая смерть консьюмера); pgxpool-метрики из O2 опираются на конфигурацию пула D4 (область 3); единый middleware (O6) собирается вместе с `Recoverer` (E1) и auth-wiring (S1) в общий `pkg/httpserver` (F4 области 1); ключ ошибки в логах `err`/`error` (E5) — там же.
