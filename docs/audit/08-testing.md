# Аудит SREOnCall — Область 8: Тесты

Дата: 2026-06-13
Область: покрытие, интеграционные тесты, table-driven, goleak, качество.
Применённый скил: `golang-testing`.

Тестов написано прилично — 24 файла, ~5161 строк, у каждого сервиса есть и хендлерные, и консьюмерные сценарии, есть e2e. Механика разделения через build-теги выстроена аккуратно. Но при ближайшем рассмотрении **реальная защита dev-loop'а тоньше, чем кажется по объёму**: бо́льшая часть поведенческих тестов спрятана за тегом `integration`, хотя инфраструктуры не требует, а CI-пайплайна нет вообще — ничто не запускает тесты, `-race` и линтеры автоматически. Именно поэтому реальные баги из областей 2–3 (D1/D2/C3) прошли мимо: код, где они живут, по умолчанию не тестируется.

## Детализация

### T1 — Нет CI-пайплайна — **major**

В репозитории нет ни `.github/workflows/`, ни корневого `Makefile`, ни `.golangci.yml` (единственные найденные — это **assets внутри скилов**, не относящиеся к проекту). Значит, ни один из 5161 строк тестов не выполняется автоматически: нет `go test ./...`, нет `go test -race`, нет `go test -tags integration`, нет e2e-гейта, нет `golangci-lint`, нет `govulncheck`. Любая регрессия ловится только если разработчик вручную вспомнит прогнать нужную команду с нужным тегом.

Это первопричина того, почему дефекты из областей 2–3 (двойная эскалация D1, нетранзакционные записи D2, обрыв по отмене C3) не были пойманы: код store/consumer покрыт тестами, которые **по умолчанию не запускаются** (T2) и в CI не запускаются вовсе. Для системы из пяти сервисов отсутствие автоматического гейта качества — самый дорогой процессный пробел в этой области.

**Фикс.** Завести GitHub Actions с матрицей по модулям `go.work`: `go build ./...`, `go test -race ./...` (юниты), `go test -tags integration ./...` на сервисах Postgres/Redis/RabbitMQ (через `services:` в workflow), `golangci-lint run`, `govulncheck ./...`. e2e — отдельным джобом по тегу/расписанию. Конфигурацию линтера взять из скила `golang-lint` (`.golangci.yml`).

### T2 — Поведенческие тесты на стабах спрятаны за тегом `integration` — **major**

Тег `integration` подразумевает тесты с реальными внешними зависимостями (по `golang-testing` — настоящие БД/очереди под build-тегом). Здесь же из 9 файлов `*_integration_test.go` **семь явно используют in-memory стабы** и сами это декларируют:

```go
//go:build integration
// Uses in-memory stubs — no external services required.   // incident/consumer_integration_test.go:1-4
```

Реальную инфраструктуру требуют только два — `scheduling/store` и `scheduling/tokenindex` (там тег уместен). Остальные (консьюмеры incident/escalation/notification, хендлеры incident/escalation/scheduling, webhook ingestion) — это, по сути, **юнит-тесты с фейками**, ошибочно помеченные `integration`.

Последствие: `go test ./...` (естественная команда, и та, что обычно стоит в pre-commit/CI) их **не компилирует и не запускает**. По умолчанию выполняется лишь тонкий срез — `escalator`, `notifier`, `rotation`, `statemachine`, `dedup`, `normalize`(alertmanager/grafana), `auth`, `tenant`, мелкие клиенты. Вся логика консьюмеров и хендлеров — за выключенным по умолчанию тегом, который вдобавок никем не запускается (T1). Объём тестов вводит в заблуждение: эффективное покрытие дефолтного прогона кратно ниже, чем 5161 строка.

**Фикс.** Снять тег `integration` с тестов на стабах — пусть они станут обычными юнит-тестами и защищают dev-loop в каждом `go test ./...`. Тег оставить только там, где реально нужны внешние сервисы (`scheduling/store`, `tokenindex`), и расширить такими настоящими integration-тестами покрытие критичных store-операций (D1/D2). Если цель тега была «быстрые vs медленные» — это решается `testing.Short()`/`-short`, а не исключением из сборки.

### T3 — Нет детекции утечек горутин (goleak) — **minor**

Ни одного `TestMain` с `goleak.VerifyTestMain`, ни одного `defer goleak.VerifyNone(t)` (поиск пуст). Пакеты с горутинами — `consumer` (цикл `Run`), `monitor` (тикер), `pkg/amqp` (reconnect) — тестируются без проверки на утечки. Это особенно болезненно на фоне областей 2: там найдено, что у консьюмеров нет ни graceful-drain (C2), ни recover (E2), ни supervisor-рестарта (C1). `goleak` в консьюмерных тестах с большой вероятностью подсветил бы зависшую после отмены горутину `Run` или незакрытый канал.

**Фикс.** Добавить `TestMain` с `goleak.VerifyTestMain(m)` в пакеты `consumer`, `monitor`, `pkg/amqp` (и в любые тесты, поднимающие `Run` в горутине). Для известных фоновых горутин библиотек — `goleak.IgnoreCurrent()`.

### T4 — Дешёвые и важные юниты без покрытия — **minor**

Файлы без соседнего `_test.go`, которые стоит покрыть в первую очередь (а не «всё подряд»):

- **`pkg/amqp/envelope.go`** — `Wrap`/`Unwrap` это межсервисный контракт шины (тот самый, что размазан копипастой, F1 области 1). Тривиально юнит-тестируется: round-trip, версия конверта, ошибка на битом payload, поведение `Unwrap` при неизвестном типе. Высокая ценность, минимальная цена.
- **`escalation/store.go`** — здесь живёт `ListExpiredStates` с `FOR UPDATE SKIP LOCKED` (баг D1) и `UpdateEscalationState` без guard'а. Ни юнита, ни integration-теста на store эскалации нет вообще.
- **`escalation/monitor.go`** — тикер-воркер, триггер D1; не покрыт.
- **`notification/ratelimit`** (security-relevant, S6/область 5), **`tenantcache`** (конкурентность C7/область 2), **`dispatcher` email/mattermost** (ретраи с неотменяемым `time.Sleep`, C5) — не покрыты.
- **`scheduling/keycloak/client.go`** — клиент Admin API, не покрыт.

**Фикс.** Начать с `pkg/amqp.Envelope` (контракт) и `tenantcache` (поведение TTL + конкурентный доступ под `-race`), затем store-операции эскалации настоящим integration-тестом, проверяющим атомарность под параллелизмом (это же закроет регресс по D1).

### T5 — Нет параллелизма и table-driven там, где он идиоматичен — **minor**

`t.Parallel()` не используется нигде (поиск пуст) — все тесты последовательны, хотя юнит-тесты здесь чистая логика (мгновенные) и независимы. Кроме того, два явных кандидата на table-driven написаны как набор отдельных функций без именованных подтестов:

- `TestParseISO8601Duration` ([rotation_test.go:25](services/scheduling/internal/rotation/rotation_test.go#L25)) — парсер длительностей с множеством входов;
- `TestValidate_AllowedTransitions`/`TestValidate_ForbiddenTransitions` ([statemachine_test.go:10](services/incident/internal/statemachine/statemachine_test.go#L10)) — матрица переходов.

Для парсера и матрицы переходов table-driven с `t.Run(tt.name, …)` даёт поимённый вывод упавшего кейса и облегчает добавление случаев. Это не дефект корректности — стиль «одна функция на сценарий» (как в `escalator_test`) для различных поведений вполне легитимен; замечание касается именно матричных кейсов и отсутствия `t.Parallel()`.

**Фикс.** Перевести `ParseISO8601Duration` и матрицу стейт-машины на table-driven с именованными подтестами и `t.Parallel()`; в остальных независимых юнит-тестах добавить `t.Parallel()`.

## Что сделано хорошо (для контекста)

- **Build-теги расставлены аккуратно и единообразно**: `//go:build integration` на всех `*_integration_test.go`, `//go:build e2e` на e2e — механика разделения чистая (проблема T2 — в семантике тега, не в его применении).
- **Широта сценариев**: у каждого сервиса есть тесты хендлеров и консьюмеров; есть полноценный e2e (332 строки, поток «тенант → алерт → инцидент»).
- **Mock-by-interface** — `mockSchedClient`, `mockPublisher`, `memStore` реализуют интерфейсы потребителя, а не подменяют конкретные типы; это ровно рекомендуемый паттерн (мокать интерфейсы).
- **Содержательные edge-кейсы**, а не покрытие ради покрытия: DST-переход в ротации (`TestOnCallAt_DST`), запрещённые переходы стейт-машины, ветки exhaust/advance/stop эскалатора, разбор JWT в `auth_test`.
- **Чистое stdlib-`testing`** без testify — согласуется с конвенцией проекта; ассершены явные, не тривиальные.
- **Настоящие integration-тесты для реально инфраструктурного кода** (`scheduling/store`, `tokenindex`) существуют и тегированы корректно — есть от чего отталкиваться, расширяя покрытие store-слоя.

## Рекомендованный порядок исправлений

1. **T1** — поднять CI (`go test -race`, integration с сервисами, lint, govulncheck): включает автоматический гейт, без которого остальное не имеет рычага.
2. **T2** — снять `integration` с тестов на стабах → дефолтный `go test ./...` снова что-то охраняет; одновременно это резко поднимает эффективное покрытие дев-цикла.
3. **T3** — `goleak` в `consumer`/`monitor`/`pkg/amqp`: ловит регрессы из C1/C2 на уровне тестов.
4. **T4** — покрыть `pkg/amqp.Envelope`, `tenantcache`, store эскалации (последнее — настоящим integration-тестом на атомарность, регресс-гард для D1).
5. **T5** — `t.Parallel()` + table-driven для парсера/матрицы.

> Кросс-ссылки: T1 — общий рычаг для областей 4–7 (lint поймает E5-ключи логов, `govulncheck` — депенденси, `-race` — конкурентность C-области); T2/T4 — почему баги D1/D2/C3 не были пойманы; T3 — тестовая проекция C1/C2 (горутины-консьюмеры). Конфигурация CI и линтеров детальнее — в следующей области (инструменты/CI).
