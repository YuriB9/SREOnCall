## Детализация

### R1 — Нет рендер-тестового харнесса — **minor**

`devDependencies` содержат vitest, но не содержат `@testing-library/react`,
`@testing-library/user-event`, `@testing-library/jest-dom`, `jsdom`
([package.json:32-51](../../frontend/package.json#L32-L51)). `vite.config.ts` **не имеет** `test`-секции
([vite.config.ts](../../frontend/vite.config.ts)) — значит окружение по умолчанию `node`, без DOM. Текущий
`parseGroups.test.ts` это устраивает (чистая функция), но любой `render(<Form/>)` упадёт без
`environment: 'jsdom'` и setup-файла с jest-dom-матчерами.

**Фикс (чейндж FE01, энейблер).** Добавить dev-зависимости
`@testing-library/react` + `@testing-library/user-event` + `@testing-library/jest-dom` +
`jsdom`; в `vite.config.ts` — `test: { environment: 'jsdom', globals: true, setupFiles:
['./src/test/setup.ts'] }`; setup-файл импортирует `@testing-library/jest-dom`. Написать
1 smoke-render-тест как образец. Это **аналог CH01 бэкенда** — сетка, на которой держатся
последующие тесты, ставится первой.

### R2 — Инварианты форм без покрытия — **minor**

Сейчас тестировать в формах нечего: валидация — это россыпь `if` и булевых `canSubmit`,
которые проще прочитать, чем застейджить в тесте. После FE02/FE03 правила станут zod-схемами,
а поведение — наблюдаемым через DOM (поле краснеет, submit заблокирован). Именно тогда тесты
дают ценность и **закрывают конкретные баги**:

- `ScheduleFormModal`: ввод `end ≤ start` показывает ошибку поля и блокирует «Добавить замену»
  (регресс-гард для F2);
- `ProfilePage`: включение канала без значения показывает предупреждение и не сохраняет (F5);
- `NotificationConfigPage`: невалидный email/url подсвечивает поле, валидный — проходит.

**Фикс.** Не отдельный чейндж: каждый форм-чейндж (FE02, FE03) **обязан** приносить
RTL-тесты на инварианты мигрированной формы (Definition of Done). FE01 даёт харнесс, FE02
пишет первые 2–3 теста как образец на `NotificationConfigPage`.

## Рекомендованный порядок (детали — в roadmap)

1. **FE01** — поставить RTL-харнесс (deps + `test`-блок + setup + 1 smoke-тест). Энейблер,
   делается **до** форм-миграции, чтобы у её тестов был дом.
2. **FE02/FE03** — каждая мигрированная форма приносит RTL-тесты на свои инварианты (DoD).

> RTL и RHF/zod — связка: RTL без миграции форм почти нечего тестировать; миграция форм без
> RTL теряет страховку от регресса по тем самым багам, ради которых затевалась. Поэтому FE01
> (харнесс) идёт первым энейблером, а тесты пишутся вместе с каждой формой.
