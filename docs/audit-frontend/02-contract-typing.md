# Аудит SREOnCall Frontend — Область 2: Типобезопасность контракта фронт↔Go

Дата: 2026-06-14
Область: синхронизация TS-типов ответов с Go-бэкендом, рантайм-валидация ответов.
Вердикт: **openapi-typescript — НЕ делаем сейчас.** Точечная zod-валидация ответов — опционально
в рамках форм-чейнджей.

[api/types.ts](../../frontend/src/api/types.ts) — 187 строк **вручную** поддерживаемых TS-типов ответов пяти
Go-сервисов. Ответы axios типизируются дженериком (`apiClient.get<UserContacts>(...)`,
[profile.ts:30](../../frontend/src/api/profile.ts#L30)) — это **обещание компилятору без рантайм-проверки**:
если Go вернёт другую форму, TS будет считать данные валидными, а упадёт оно как `undefined`
глубже по стеку. Это реальный, но **латентный** риск — он не горит.

## Решение 1 — openapi-typescript: НЕ делаем (сейчас)

**Что это дало бы.** Генерация `types.ts` из OpenAPI-спеки → один источник правды, типы
фронта следуют за контрактом автоматически, рассинхрон C1 исчезает структурно.

**Почему не сейчас — фактическая блокировка.** `openapi-typescript` генерит типы **из спеки.
Спеки нет.** Сервисы — на `go-chi/chi`, swag-аннотаций (`@Summary`/`@Router`) в коде ноль
(проверено grep'ом по `.go`). Чтобы получить вход для генератора, нужно:

1. навесить аннотации на **пять** сервисов (incident, scheduling, escalation, notification,
   ingestion) или вручную писать/поддерживать `openapi.yaml`;
2. встроить генерацию спеки в Go-CI и держать аннотации синхронными с хендлерами;
3. только потом подключить `openapi-typescript` на фронте.

Это **бэкенд-работа на 5 сервисов** + новое постоянное обязательство (аннотации, которые
сами могут разойтись с кодом) ради замены 187 строк, которые сейчас не вызывают инцидентов.
Соотношение цена/боль не оправдано. Каждая лишняя зависимость и пайплайн — это обязательство.

**Реалистичный путь под эту архитектуру.** Оставить ручной `types.ts` как есть — он аккуратный,
сгруппирован по сервисам и читается как контракт. Это нормальное состояние, а не долг.

**Когда пересмотреть.** Если OpenAPI-спека появится на бэкенде по **другим** причинам
(внешние потребители API, кодоген Go-клиентов, публичная документация) — тогда
`openapi-typescript` становится почти бесплатным выигрышем и заменяет и `types.ts`, и любые
ручные zod-схемы. Это «если/когда», а не «сейчас». Решение фиксируется здесь именно чтобы
**агенты не предлагали openapi-typescript повторно** без выполнения предусловия (спека на бэке).

## Решение 2 — zod-валидация ответов: опционально, точечно

`apiClient.get<T>()` лжёт компилятору (C2). Один способ закрыть это **без** бэкенд-работы —
рантайм-`parse` ответа zod-схемой в `queryFn`:

```ts
// было: const { data } = await apiClient.get<UserContacts>(url); return data
// стало: return UserContactsSchema.parse((await apiClient.get(url)).data)
```

Тогда рассинхрон с Go падает **в точке запроса** с понятным сообщением, а не как `undefined`
тремя слоями выше.

**Но это решение с границами:**
- **Не** переписывать весь api-слой на `parse` разом — это превратило бы латентный риск в
  лавину работы и связало бы релиз фронта с каждым полем Go.
- Применять **точечно** на 1–2 критичных ответах, которые рендерятся в формы/детали и где
  тихий `undefined` болезненнее всего (кандидаты: `UserContacts`, `NotificationConfig`).
- Переиспользует zod, который **и так вводится** для форм (FE02) — нулевая новая зависимость.
- Схема ответа ≠ схема запроса (ответ шире: серверные `id`/`created_at`), но примитивы из
  `lib/validators.ts` общие.

Поэтому это **не отдельный чейндж**, а опциональная под-задача внутри FE02/FE03 — добавить
`*ResponseSchema` для того же ресурса, форму которого мигрируем. Если по ходу окажется, что
ценность мала — не делать, и это ок.

## Что сделано хорошо (для контекста)

- **`types.ts` дисциплинирован**: сгруппирован комментариями по сервисам, union-литералы для
  статусов/severity ([api/types.ts:3-4](../../frontend/src/api/types.ts#L3-L4)), `?`-опциональность отражает
  реальную форму ответов. Это **хороший** ручной контракт, а не свалка.
- **api-слой модульный**: файл на сервис, общий `client.ts`, query-ключи и типы ошибок рядом
  с запросами ([profile.ts](../../frontend/src/api/profile.ts)). Точечный `parse` встанет сюда без переустройства.
- **Деградация на 404 уже осмысленная**: `useUserContacts` ловит «нет контактов» как `null`
  ([profile.ts:34-37](../../frontend/src/api/profile.ts#L34-L37)) — zod-`parse` это поведение не ломает (парсить только
  успешный ответ).
