# Протокол запуска фронтенд-чейнджа в новой сессии

Каждый чейндж из [00-roadmap.md](00-roadmap.md) реализуется в **отдельной холодной сессии** Claude.
Холодная сессия не помнит аудит — вся память живёт в коммитнутых файлах:
`docs/audit-frontend/*` (находки с `file:line`), `00-roadmap.md` (план + зависимости + статус),
`openspec/specs/*` (capability-спеки), `openspec/config.yaml` (контекст проекта + rules).

Параллель к бэкенд-протоколу `docs/audit/CHANGE-KICKOFF.md`. Отличия фронта отмечены ниже.

**Репозиторий:** `https://github.com/YuriB9/SREOnCall.git` · интеграционная ветка — `main`.
**Каждый чейндж = отдельная ветка**, после архивации мержится в `main` через PR (Actions настроен с CH01).
**Рабочая директория фронта:** `frontend/` (свой `package.json`, отдельно от Go-модулей).

---

## 1. Промпт для старта (копировать в новую сессию)

Подставь номер и слаг из roadmap.

```
Реализуем находки фронтенд-аудита через OpenSpec, по одному чейнджу за сессию.
Сейчас делаем FE<NN> — <slug> из docs/audit-frontend/00-roadmap.md.

Выполни протокол ПЕРЕД любыми изменениями (docs/audit-frontend/CHANGE-KICKOFF.md):

1. Прочитай docs/audit-frontend/00-roadmap.md → строку FE<NN>: scope,
   находки, depends-on, риск. Объём = строго находки FE<NN> из «Матрицы
   покрытия». Соседние находки НЕ трогаем — это другие чейнджи.
2. Прочитай разделы docs/audit-frontend/<NN>-*.md, на которые ссылается
   FE<NN> — там точные находки с file:line и предложенными фиксами.
3. PRE-FLIGHT зависимостей: каждый depends-on уже заархивирован в
   openspec/changes/archive/ И влит в main. Если нет — СТОП, сообщи мне.
4. ВЕТКА: git checkout main && git pull, затем git checkout -b
   fe<NN>-<slug>. Вся работа — только в этой ветке.
5. Прочитай openspec/config.yaml (контекст + rules) и затронутые
   capability-спеки в openspec/specs/.
6. Создай чейндж через opsx (/opsx:propose или /opsx:new), имя <slug>.
   Артефакты на русском; capability-дельта только если меняется
   наблюдаемое поведение продуктовой capability; ADR — только для
   значимого арх-решения (для FE02 — да, см. ниже).
7. В tasks.md привяжи каждую задачу к находке и file:line
   (например: «F2 — end>start в override, frontend/src/pages/ScheduleFormModal.tsx»).
8. Реализуй → /opsx:verify → прогон проверок (раздел 3).
9. /opsx:archive, обнови статус FE<NN> в docs/audit-frontend/00-roadmap.md
   (дашборд + строка чейнджа).
10. ВЛЕЙ В MAIN через PR (Actions зелёный). Сообщи результат.

Сначала покажи план (proposal + tasks), реализацию начинай после моего ОК.
```

## 2. Pre-flight чеклист

- [ ] **Зависимости закрыты.** Все `depends-on` из roadmap-строки заархивированы И влиты в `main`.
      Критический путь фронта: `FE01 → FE02 → FE03`. Не начинать FE02 без FE01, FE03 без FE02.
- [ ] **Объём заморожен.** Берём ровно находки из «Матрицы покрытия» для этого FE. Соседняя находка
      по дороге → записать в Backlog roadmap, **не расширять текущий**.
- [ ] **Прочитан исходник аудита.** Открыть `docs/audit-frontend/<NN>-*.md`, взять `file:line`
      оттуда. Аудит писался по снимку кода (2026-06-14) — **перепроверить, что строки/символы ещё
      существуют** (предыдущий FE-чейндж мог сдвинуть их).
- [ ] **UX-инварианты не потеряны.** Перед миграцией формы выписать её UX-качества (раздел «Что
      сделано хорошо» в `01`) — они критерий приёмки, а не «перепишем с нуля».

## 3. Проверки перед архивацией (Definition of Done)

Все команды — из директории `frontend/`:

| Что менялось | Команды |
|--------------|---------|
| Любой TS/TSX | `npm run lint`, `npx tsc -b` (или `npm run build`) |
| Любой чейндж | `npm run test` (vitest) — должен быть зелёным |
| Миграция формы (FE02/FE03) | RTL-тесты на инварианты формы **обязательны** (DoD, см. область 3) |
| FE03 / ScheduleFormModal | **обязательно** регресс-тест: инвертированный override (`end ≤ start`) блокируется с ошибкой поля (баг F2) |
| Поведение формы | ручная проверка в `npm run dev` (сохранение, валидация, тогглы) |

Затем: `/opsx:verify` → `/opsx:archive`.

> **CI-гейт.** На бэке Actions настроен (CH01). Убедись, что workflow гоняет и фронтовые
> проверки (`npm ci && npm run lint && npm run build && npm run test` в `frontend/`). Если нет —
> это отдельная находка для FE01/Backlog, отметь в roadmap.

## 4. Хэндофф — что обновить

После архивации **в той же сессии**:

1. **Обнови `docs/audit-frontend/00-roadmap.md`**: пометь FE<NN> как `✅ done` (дашборд + строка),
   обнови «Прогресс: N / 3».
2. Новая проблема вне объёма → строка в «Backlog находок» roadmap или в соответствующий
   `docs/audit-frontend/<NN>-*.md`, **не реализуя**.
3. Если FE<NN> ввёл переиспользуемый артефакт (`form.tsx`, `lib/validators.ts`, тест-setup),
   на который опирается следующий чейндж — отметь это в его строке roadmap (как FE03 опирается
   на инфраструктуру FE02).

> Память между сессиями = коммит. Не записанное в `docs/audit-frontend/*` или `openspec/` в
> следующей сессии не существует.

## 5. Capability-привязка (стартовая гипотеза — подтвердить против `openspec/specs/`)

Форм-чейнджи **меняют UI, но обычно не меняют наблюдаемую гарантию продуктовой capability**
(API/события не трогаются — формы шлют тот же payload на те же эндпоинты). Скорее всего — без
дельты, как infra/UI-рефактор. Исключение — если миграция меняет **что именно** валидируется на
клиенте так, что это документировано в спеке (напр. tenant-settings-ui про webhook URL — но это
уже закрыто CH04 на бэке).

| Чейндж | Тип | Вероятная capability |
|--------|-----|----------------------|
| FE01 | infra/tooling (тест-харнесс) | нет дельты |
| FE02 | UI-рефактор (валидация форм) | нет дельты (payload не меняется); **ADR** на подход RHF/zod |
| FE03 | UI-рефактор + фикс клиентского инварианта F2 | нет дельты или уточнение `tenant-settings`/`scheduling`-UI — решить с владельцем |

При сомнении «новая capability или нет» — спросить владельца, не вводить спек по своей инициативе.

## 6. Грабли, специфичные для фронта

- **Рабочая директория — `frontend/`.** `package.json`, `node_modules`, vitest, eslint — все там.
  Go-`go.work` к фронту отношения не имеет; не путать команды.
- **UI-тексты и сообщения валидации — на русском** (правило проекта). zod-`message` тоже русские.
- **shadcn `form.tsx` ещё не существует** — его вводит FE02 (`src/components/ui/form.tsx`).
  Не выдумывать обёртку с нуля, если shadcn-версия подходит.
- **`@`-алиас** настроен в `vite.config.ts` — импорты в тестах те же, что в приложении.
- **Не растерять UX**: висячие ссылки (PolicyEditor), маскировку webhook (NotificationConfig),
  типизированные серверные ошибки (ScheduleFormModal) — это явный критерий приёмки, см. `01`.
- **`zod` — прямая зависимость только после FE02.** Сейчас она есть в `package-lock`
  **транзитивно** (тянется чужой зависимостью), но не в `dependencies`. FE02 добавляет её явно.
- **openapi-typescript предлагать нельзя** без предусловия (OpenAPI-спека на бэке) — закрыто в
  [02-contract-typing.md](02-contract-typing.md).

## 7. Связь с бэкенд-аудитом

- Нумерация **FE** не пересекается с **CH** (`docs/audit/`). Бэкенд и фронт ведутся параллельно.
- CH04 (`guard-webhook-ssrf`) уже задал серверную валидацию webhook URL (https + блок приватных).
  Клиентская валидация url в `NotificationConfigPage` (FE02) — это UX-зеркало, **не** замена
  серверной проверки. Не ослаблять серверную в угоду клиентской.
