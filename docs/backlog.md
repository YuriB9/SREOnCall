# Backlog-кандидаты

Идеи, осознанно удалённые из спецификаций или отложенные при синхронизации
«спека = код» (change `sync-specs-with-implementation`, июнь 2026). Для возврата
любой из них нужен отдельный OpenSpec-change с ADDED/MODIFIED-дельтой и реализацией.

- **Фильтр инцидентов по источнику** (`?source=`) — нет ни в API, ни в UI;
  удалён из требований дашборда.
- **Фильтр инцидентов по лейблам** (`?label=k:v`) — store поддерживает
  `ListFilter.Label`, но handler параметр не разбирает и UI его не имеет;
  из спеки удалён.
- **Объединённый таймлайн инцидента** — подмешивание событий эскалации
  (escalation history) в вкладку «История» панели деталей.
- **Агрегированный виджет «кто дежурит»** — одна сводка по всем расписаниям
  вместо карточек per-schedule (+ текущий уровень эскалации, если появится
  источник данных).
- **`PUT /policies/{id}`** — атомарное редактирование политики эскалации вместо
  замены POST → перенос default → DELETE (известное ограничение: не атомарно).
- **Пересчёт severity инцидента** по максимуму severity привязанных алертов
  (сейчас — от первого алерта, без пересчёта).
- **Каскадное удаление тенанта** — внутрисервисная очистка scheduling
  (расписания, переопределения, вебхук-токены + Redis-индекс, конфигурация
  уведомлений) и межсервисный каскад (инциденты, эскалации, журнал
  уведомлений). Сейчас DELETE тенанта удаляет только запись тенанта;
  рекомендация — отзывать токены перед удалением.
