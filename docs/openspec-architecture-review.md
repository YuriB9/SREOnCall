# Анализ архитектуры спецификаций OpenSpec (SREOnCall)

> Дата анализа: 2026-06-10. Объект анализа: каталог `openspec/` — 12 главных спецификаций,
> активное изменение `harden-auth-shell`, два архивных изменения.

## 1. Обзор структуры

```text
openspec/
├── config.yaml                  # схема spec-driven; context и rules не заполнены
├── specs/                       # 12 capability-спецификаций (~1100 строк)
│   ├── бэкенд: alert-ingestion, incident-management, escalation-policies,
│   │          oncall-scheduling, notification-dispatch, tenant-management
│   └── фронтенд: frontend-shell, incident-dashboard-ui, oncall-schedule-ui,
│              escalation-policy-ui, tenant-settings-ui, user-profile-ui
└── changes/
    ├── harden-auth-shell/       # активное изменение (все задачи не выполнены)
    └── archive/
        ├── 2026-06-07-sre-oncall-platform/    # бэкенд-платформа
        └── 2026-06-08-sre-oncall-frontend/    # фронтенд
```

Архитектура описываемой системы: 5 Go-микросервисов (ingestion → incident → escalation →
notification, плюс scheduling), связанных через RabbitMQ; PostgreSQL, Redis; SPA-фронтенд
с OIDC через Keycloak; мультитенантность на основе Keycloak-групп.

Поток событий: `alert.received` → `incident.created/updated` → `escalation.triggered/exhausted` → уведомления.

## 2. Сильные стороны

- **Дисциплина формата.** Все спеки следуют единому шаблону Requirement → Scenario
  (WHEN/THEN), с нормативными «ДОЛЖЕН/НЕ ДОЛЖЕН». Сценарии конкретны (коды HTTP, имена
  очередей, ключи `localStorage`).
- **Мультитенантность продумана сквозно**: изоляция по `tenant_id` в дедупликации,
  fingerprint, контактах, конфигурации уведомлений; отдельное требование изоляции данных.
- **Хорошие продуманные мелочи**: ротация webhook-токенов без простоя (несколько активных
  токенов на источник), одноразовый показ plaintext-токена с обязательным чекбоксом,
  отсутствие звука при скрытой вкладке, DST-сценарий в Gantt.
- **Процесс OpenSpec используется корректно**: завершённые изменения заархивированы с
  датами, дельта-спека `harden-auth-shell` меняет только два требования и помечена
  `MODIFIED`; в design.md зафиксированы альтернативы и причины отказа от них.
- **`harden-auth-shell` — образцовый пример**: аудит выявил реальное расхождение
  фронт/бэкенд (`parseGroups` vs `IsMember`) и конфигурационную связку (groups-маппер в
  обоих токенах), и изменение закрывает именно их.

## 3. Баги: противоречия между спецификациями

Главная системная проблема: **UI-спеки писались отдельной волной (изменение
`sre-oncall-frontend`) и ссылаются на эндпоинты, которых нет в бэкенд-спеках, либо
называют их иначе**. Если фронтенд реализован по своим спекам — он не работает с таким
бэкендом; если работает — спеки лгут. Ниже конкретика.

### 3.1 Несуществующий эндпоинт «кто дежурит» для виджета — критично

`oncall-schedule-ui/spec.md:11` требует `GET /api/schedules/v1/{tenant}/oncall?at=now`
(тенант-уровневый, без расписания), но `oncall-scheduling/spec.md:39` определяет только
`GET .../schedules/{id}/oncall` (на конкретное расписание). Тенант-уровневой агрегации
«кто сейчас дежурит по всем расписаниям» в бэкенде нет. Также `?at=now` не соответствует
контракту `?at=<RFC3339>`.

Тот же виджет должен показывать «текущий уровень эскалации» — но состояние эскалации
существует только per-incident (`escalation-policies/spec.md:90`); источника данных для
виджета нет.

### 3.2 Gantt запрашивает не тот эндпоинт

`oncall-schedule-ui/spec.md:27`: окно смен запрашивается через
`GET .../schedules/{id}/oncall?from=&to=` — но `/oncall` принимает только `?at=`.
Для окна смен существует `GET .../schedules/{id}/shifts?from=&to=`
(`oncall-scheduling/spec.md:76`). Кроме того, формулировка «для каждой ротации» намекает
на N+1 запросов, тогда как `/shifts` возвращает все смены расписания одним вызовом.

### 3.3 Профиль пользователя обращается не к тому сервису

`user-profile-ui/spec.md:11`: контакты читаются/пишутся через
`GET/PUT /api/schedules/v1/users/me/contacts` (scheduling, без тенанта). Но владелец
контактов — notification-сервис: `PUT /api/notifications/v1/{tenant}/contacts/{userId}`,
и данные привязаны к паре `(user_id, tenant_id)` (`notification-dispatch/spec.md:97`).
Конфликт двойной: не тот сервис **и** потеря тенантного измерения — глобальная страница
`/profile` не может однозначно определить, контакты какого тенанта она редактирует.

### 3.4 Назначение политики по умолчанию: два несовместимых контракта

UI: `PATCH /api/escalations/v1/{tenant}/policies/{id}` с `{"is_default": true}`
(`escalation-policy-ui/spec.md:63`). Бэкенд: `PUT /api/escalations/v1/{tenant}/default-policy`
с `{"policy_id": "<uuid>"}` (`escalation-policies/spec.md:35`). Поле `is_default` в модели
политики бэкенда вообще отсутствует.

### 3.5 Настройки тенанта: расходятся и пути, и методы

| Что | tenant-settings-ui | tenant-management |
|---|---|---|
| Токены | `GET/POST/DELETE /api/schedules/v1/{tenant}/webhook-tokens` | `.../v1/tenants/{slug}/webhook-tokens` |
| Участники | `GET /api/schedules/v1/{tenant}/members` | `.../v1/tenants/{slug}/members` |
| Конфиг уведомлений | `GET` + **`PATCH`** `/api/schedules/v1/{tenant}/notification-config` | **`PUT`** `.../v1/tenants/{slug}/notification-config` |

В UI-путях пропущен сегмент `tenants/`, метод сохранения конфига — PATCH против PUT.
Также при создании токена UI просит «метку источника» как свободный текст
(`tenant-settings-ui/spec.md:27`), тогда как бэкенд требует enum `source`
(alertmanager | grafana | zabbix).

### 3.6 Маскированный webhook URL затирает реальный при сохранении

`tenant-management/spec.md:92`: GET конфига возвращает `mattermost_webhook_url`
**маскированным** (только домен). Форма UI (`tenant-settings-ui/spec.md:53`) загружает
конфиг через GET, показывает поля и сохраняет всю форму. Если пользователь поменял только
`mattermost_channel`, при сохранении уйдёт маскированное значение URL и перезапишет
рабочий вебхук. Ни одна из спек этот сценарий не оговаривает.

### 3.7 Фильтр по источнику, которого нет

Дашборд требует фильтрацию инцидентов по источнику (`incident-dashboard-ui/spec.md:11`),
но `GET .../incidents` поддерживает только `status`, `severity`, `label`, `from_time`,
`to_time` (`incident-management/spec.md:140`). Параметра `source` нет — как и поля
«источник» в модели инцидента вообще (инцидент может группировать алерты разных источников?).

### 3.8 Ingestion: «ответить 200 и поставить в очередь» против «вернуть 500»

`alert-ingestion/spec.md:32` — на валидный payload сервис «возвращает HTTP 200 и ставит
каждый алерт в очередь на обработку» (асинхронная модель). Но `spec.md:119` — при
недоступности RabbitMQ после 3 ретраев с экспоненциальной задержкой сервис «возвращает
HTTP 500 вызывающей стороне» (синхронная модель). Оба требования одновременно невыполнимы:
если 200 уже отдан, 500 вернуть некому; если публикация синхронна, ретраи с backoff
упрутся в таймаут вебхука Alertmanager (~10 с по умолчанию).

### 3.9 Поиск webhook-токена: таблица БД или Redis?

`alert-ingestion/spec.md:11` — ingestion делает «lookup токена в таблице
`tenant_webhook_tokens`». `tenant-management/spec.md:59` — scheduling поддерживает индекс
`oncall:tokens:{hash}` в Redis именно для этого. Либо ingestion читает чужую таблицу
напрямую (anti-pattern «shared database» между микросервисами), либо использует Redis —
спеки утверждают разное. Сценарий отзыва токена («дальнейшие вебхуки возвращают 401»)
зависит от того, какой путь канонический.

### 3.10 Уведомление tier 1: notification ожидает событие, которое escalation не обещает

`notification-dispatch/spec.md:13`: «первичное оповещение при создании инцидента
покрывается событием `escalation.triggered` для tier 1, публикуется escalation
**сразу после авто-назначения** политики». Но в `escalation-policies` сценарий
авто-назначения (`spec.md:43-47`) говорит лишь «отслеживание стартует с 1-го уровня»,
а событие `escalation.triggered` публикуется только **по истечении таймаута**
(`spec.md:67-74`). Если escalation реализован строго по своей спеке, дежурный получает
первое уведомление не при создании инцидента, а спустя `timeout_minutes` — для
on-call-платформы это критичный по смыслу разрыв.

## 4. Пробелы (недоопределённое поведение)

1. **Severity инцидента не определена.** Алерт имеет `severity` в канонической схеме, UI
   показывает колонку «Критичность» и фильтр, list-API фильтрует по `severity` — но в
   `incident-management` нигде не сказано, как инцидент получает severity (от первого
   алерта? максимум по группе? что при частичном resolve?).
2. **Payload `escalation.triggered` недостаточен для уведомления.** Email обязан содержать
   заголовок, severity и статус инцидента (`notification-dispatch/spec.md:31`), но событие
   несёт только `incident_id`, `tenant_id`, `tier`, `oncall_user_id`, `oncall_username`.
   Либо нужен синхронный вызов notification → incident (нигде не описан), либо обогащение
   payload (противоречит заявленному «notification не делает синхронных вызовов»).
3. **Эскалация и жизненный цикл инцидента рассинхронизированы.** Эскалация останавливается
   только при `acknowledged`. Не определено: остановка при `resolved` без подтверждения
   (сейчас по букве спеки эскалация продолжится по закрытому инциденту); перезапуск при
   повторном открытии инцидента (`open` ← `resolved`).
4. **Resolved-алерт без открытого инцидента.** Incident-сервис описывает обработку
   resolved-алерта только при наличии совпадающего открытого инцидента. Поведение для
   «осиротевшего» resolved (инцидент уже закрыт вручную, или firing был дедуплицирован) не задано.
5. **Вкладка «История» показывает события эскалации**, но журнал инцидента фиксирует
   только статусы, лейблы и комментарии (`incident-management/spec.md:124`). Откуда UI
   возьмёт события эскалации в едином таймлайне — не определено (escalation отдаёт своё
   состояние отдельным эндпоинтом).
6. **Каскадное удаление тенанта** (`tenant-management/spec.md:33`) обещает удалить «все
   связанные данные», но инциденты, эскалации и журнал уведомлений живут в других
   сервисах/БД. Механизм межсервисного каскада (событие `tenant.deleted`? сага?) не описан.
7. **Кто вправе создавать тенантов** — POST `/tenants` без указания требуемой роли
   (любой аутентифицированный? платформенный админ?). Учитывая, что members управляются
   только через Keycloak, создание тенанта без создания Keycloak-группы даёт «мёртвый» тенант.
8. **`config.yaml` пуст** — секции `context` и `rules` остались шаблонными комментариями,
   хотя у проекта есть очевидный контекст (Go-микросервисы, RabbitMQ, мультитенантность,
   русскоязычные спеки) и устоявшиеся конвенции оформления.

## 5. Рекомендации

### Приоритет 1 — устранить расхождения контрактов (раздел 3)

Это не стилистика: каждая пара «UI-спека ↔ бэкенд-спека» из раздела 3 описывает
неработающую интеграцию. Рекомендуемый порядок:

1. Сверить с **реальным кодом** (`services/`, `frontend/src/`), какой контракт фактически
   реализован, и привести проигравшую спеку к реальности (через изменение OpenSpec, как
   сделано в `harden-auth-shell`). Особенно 3.1, 3.3, 3.4, 3.5 — там выбор «чей контракт
   канонический» определяет код двух сторон.
2. Для 3.10 (tier-1 уведомление) — дополнить `escalation-policies` явным требованием:
   при назначении политики (авто или ручном) немедленно публикуется `escalation.triggered`
   с `tier: 1`. Это самый дешёвый способ закрыть разрыв.
3. Для 3.8 — выбрать модель: либо синхронная публикация с коротким таймаутом и честным 500,
   либо 200 + локальный буфер/WAL с фоновым republish. Зафиксировать одну.

### Приоритет 2 — закрыть пробелы поведения (раздел 4)

Минимум: определить severity инцидента (предложение: максимальная severity среди
привязанных firing-алертов, пересчёт при изменении состава), связать эскалацию с
`resolved`/re-open, описать обогащение payload эскалационных событий полями
`incident_title`, `incident_severity`, `incident_status`.

### Приоритет 3 — процессные улучшения

- **Заполнить `config.yaml`**: tech stack, конвенции (русский язык, формат WHEN/THEN,
  RFC-2119), правила для proposal/tasks. Это напрямую улучшит качество будущих
  AI-генерируемых артефактов.
- **Завести единый реестр API-контрактов.** Корень всех багов раздела 3 — дублирование
  URL-путей в двух независимых спеках. Варианты: (а) capability-спека «api-contracts» с
  таблицей эндпоинтов, на которую ссылаются и бэкенд-, и UI-спеки; (б) OpenAPI-файлы в
  `docs/` как источник истины, спеки описывают только поведение. Любой вариант лучше
  копипасты путей.
- **Аналогично — реестр событий шины** (exchange, routing keys, схемы payload):
  `alert.received`, `incident.created/updated`, `escalation.triggered/exhausted` сейчас
  описаны фрагментами в четырёх спеках; несоответствие payload (п. 4.2) — прямое следствие.
- **Чек-лист кросс-ревью для UI-изменений**: «каждый упомянутый эндпоинт существует в
  бэкенд-спеке с тем же путём, методом и схемой». Аудит из `harden-auth-shell` показал,
  что такой кросс-аудит ловит реальные баги — его стоит сделать регулярной практикой
  (и прогнать по разделу 3 этого документа как по готовому списку).
- **Довести `harden-auth-shell` до конца**: все задачи в `tasks.md` не отмечены, дельта
  не синхронизирована в `specs/frontend-shell/spec.md`. Пока изменение висит, главная
  спека описывает поведение (мгновенный 30-секундный отсчёт по любому `silentRenewError`,
  узкая трактовка групп), признанное дефектным.

### Возможные оптимизации описанной системы (за рамками консистентности)

- **Поллинг 12 с на дашборде** (`incident-dashboard-ui/spec.md:11`) — при росте числа
  открытых вкладок это заметная нагрузка; SSE/WebSocket от incident-сервиса (у него уже
  есть события `incident.created/updated` в RabbitMQ) дал бы и меньшую задержку для
  звуковых уведомлений. Как минимум — поддержать `If-None-Match`/`ETag`.
- **Дедупликация через SETNX теряет повторные firing**: алерт, отброшенный в окне 5 минут,
  не обновляет `last_seen` инцидента. Стоит рассмотреть SETEX с продлением + счётчик
  повторов, прокидываемый в инцидент.
- **Виджет «кто дежурит» + Gantt**: после исправления 3.1/3.2 одним эндпоинтом `/shifts`
  можно покрыть и Gantt, и мобильный список, и виджет (смена, активная сейчас), убрав
  лишние запросы.
