# On-Call Scheduling

## Purpose

Управление расписаниями дежурств и ротациями, вычисление текущего дежурного, поддержка разовых переопределений.

## Requirements

### Requirement: Создание и управление расписаниями

Сервис scheduling ДОЛЖЕН предоставлять CRUD-эндпоинты для расписаний дежурств по адресу `/api/schedules/v1/{tenant}/schedules`.

Расписание ДОЛЖНО включать: `name`, `tenant_id` (заполняется из URL), `timezone`, `rotation` (список ID пользователей в порядке ротации), `shift_duration` (длительность в формате ISO 8601, например `P7D`), `start_date`.

#### Scenario: Создание расписания

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules` участником тенанта с валидными полями
- **THEN** расписание сохраняется с `tenant_id` из URL и возвращается с HTTP 201 и сгенерированным `id`

#### Scenario: Создание расписания с отсутствующим обязательным полем

- **WHEN** выполняется POST без `rotation` или `shift_duration`
- **THEN** сервис возвращает HTTP 422 с ошибками на уровне полей

#### Scenario: Обновление расписания

- **WHEN** выполняется PATCH на `/api/schedules/v1/{tenant}/schedules/{id}` с изменёнными полями
- **THEN** обновляются только указанные поля; существующие смены пересчитываются из обновлённой конфигурации

#### Scenario: Удаление расписания

- **WHEN** выполняется DELETE на `/api/schedules/v1/{tenant}/schedules/{id}`
- **THEN** расписание и все связанные переопределения удаляются

---

### Requirement: Вычисление текущего дежурного

Сервис scheduling ДОЛЖЕН предоставлять GET `/api/schedules/v1/{tenant}/schedules/{id}/oncall`, возвращающий пользователя, который дежурит в данный момент.

#### Scenario: Вычисление текущего дежурного

- **WHEN** выполняется GET-запрос на `/api/schedules/v1/{tenant}/schedules/{id}/oncall`
- **THEN** сервис возвращает ID и отображаемое имя текущего дежурного на основании ротации расписания и активных переопределений

#### Scenario: Вычисление дежурного на заданное время

- **WHEN** GET-запрос содержит параметр `?at=<временная метка RFC3339>`
- **THEN** сервис возвращает, кто был бы дежурным в это конкретное время

---

### Requirement: Переопределения расписания

Сервис scheduling ДОЛЖЕН поддерживать разовые переопределения, заменяющие запланированного дежурного на конкретный временной интервал.

#### Scenario: Создание переопределения

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules/{id}/overrides` с `user_id`, `start_at` и `end_at`
- **THEN** переопределение сохраняется; в течение этого окна переопределённый пользователь возвращается как дежурный

#### Scenario: Переопределение имеет приоритет над ротацией

- **WHEN** текущее время попадает в активное окно переопределения
- **THEN** эндпоинт вычисления дежурного возвращает пользователя переопределения, а не пользователя из ротации

#### Scenario: Пересекающиеся переопределения отклоняются

- **WHEN** новое переопределение подаётся с временным окном, пересекающимся с существующим
- **THEN** сервис возвращает HTTP 409

---

### Requirement: Список предстоящих смен

Сервис scheduling ДОЛЖЕН предоставлять GET `/api/schedules/v1/{tenant}/schedules/{id}/shifts?from=<время>&to=<время>`, возвращающий все вычисленные смены в запрошенном окне.

#### Scenario: Список смен на следующие 7 дней

- **WHEN** GET-запрос задаёт 7-дневное окно
- **THEN** ответ содержит список смен с `user_id`, `start_at`, `end_at` и флагом `is_override`
