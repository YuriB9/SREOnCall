# Escalation Policies

## Purpose

Управление политиками эскалации, автоматический и ручной переход между уровнями, публикация событий эскалации с обогащёнными данными о дежурном.

## Requirements

### Requirement: Создание политик эскалации

Сервис escalation ДОЛЖЕН предоставлять CRUD-эндпоинты по адресу `/api/escalations/v1/{tenant}/policies` для управления политиками эскалации.

Политика ДОЛЖНА включать: `name`, `schedule_id` (основное расписание дежурств), `tiers` (упорядоченный список уровней эскалации).

Каждый уровень ДОЛЖЕН включать: `order` (целое число, начиная с 1), `timeout_minutes` (время ожидания перед переходом на следующий уровень), `notify_schedule_id` (расписание для уведомления на данном уровне). Уровень 1 ДОЛЖЕН использовать основной `schedule_id` политики.

#### Scenario: Создание политики с двумя уровнями

- **WHEN** выполняется POST на `/api/escalations/v1/{tenant}/policies` с двумя уровнями
- **THEN** политика сохраняется и возвращается с HTTP 201

#### Scenario: Политика без уровней отклоняется

- **WHEN** выполняется POST с пустым массивом `tiers`
- **THEN** сервис возвращает HTTP 422

---

### Requirement: Политика эскалации по умолчанию для тенанта

Сервис escalation ДОЛЖЕН хранить `default_escalation_policy_id` per-tenant в таблице `tenant_escalation_config` и автоматически назначать политику при появлении нового инцидента.

#### Scenario: Установка политики по умолчанию

- **WHEN** администратор выполняет PUT на `/api/escalations/v1/{tenant}/default-policy` с `{ "policy_id": "<uuid>" }`
- **THEN** `default_escalation_policy_id` сохраняется; политика должна принадлежать этому тенанту, иначе HTTP 422

#### Scenario: Снятие политики по умолчанию

- **WHEN** администратор выполняет DELETE на `/api/escalations/v1/{tenant}/default-policy`
- **THEN** запись удаляется; новые инциденты создаются без автоматической эскалации

#### Scenario: Автоматическое назначение при создании инцидента

- **WHEN** сервис escalation потребляет событие `incident.created` из очереди `incidents.escalation`
- **AND** для тенанта инцидента задан `default_escalation_policy_id`
- **THEN** политика автоматически привязывается к инциденту и отслеживание стартует с 1-го уровня (эквивалентно ручному POST на `/policy`)

#### Scenario: Тенант без политики по умолчанию

- **WHEN** событие `incident.created` получено, но `tenant_escalation_config` для тенанта отсутствует
- **THEN** запись игнорируется; эскалация может быть назначена вручную позднее

---

### Requirement: Привязка политики эскалации к инциденту

Сервис escalation ДОЛЖЕН позволять назначать политику эскалации инциденту, инициируя автоматическое отслеживание эскалации.

#### Scenario: Назначение политики инциденту

- **WHEN** выполняется POST на `/api/escalations/v1/{tenant}/incidents/{incidentId}/policy` с `policy_id`
- **THEN** политика привязывается и отслеживание эскалации начинается с 1-го уровня с таймаутом этого уровня

---

### Requirement: Срабатывание эскалации по таймауту

Сервис escalation ДОЛЖЕН отслеживать активные состояния эскалации и переходить на следующий уровень, если текущий таймаут истёк без подтверждения.

#### Scenario: Инцидент не подтверждён в течение таймаута

- **WHEN** инцидент с активной политикой эскалации не подтверждён в течение `timeout_minutes` текущего уровня
- **THEN** сервис escalation запрашивает scheduling (`GET /schedules/{notify_schedule_id}/oncall`) за текущим дежурным, переходит на следующий уровень и публикует событие `escalation.triggered` с полями `incident_id`, `tenant_id`, `tier`, `oncall_user_id`, `oncall_username`

#### Scenario: Инцидент подтверждён до истечения таймаута

- **WHEN** инцидент переходит в статус `acknowledged` до истечения таймаута уровня
- **THEN** переход эскалации для данного инцидента останавливается

#### Scenario: Все уровни исчерпаны

- **WHEN** все уровни были уведомлены, а инцидент так и не подтверждён
- **THEN** состояние эскалации устанавливается в `exhausted`, публикуется событие `escalation.exhausted`; дальнейшая автоматическая эскалация не происходит

---

### Requirement: Видимость состояния эскалации

Сервис escalation ДОЛЖЕН предоставлять GET `/api/escalations/v1/{tenant}/incidents/{incidentId}/state`, возвращающий текущий уровень эскалации, время до следующей эскалации и полную историю уровней.

#### Scenario: Запрос активного состояния эскалации

- **WHEN** выполняется GET-запрос для инцидента с активной эскалацией
- **THEN** ответ содержит `current_tier`, `next_escalation_at` и список прошедших переходов уровней с временными метками

---

### Requirement: Ручное переопределение эскалации

Сервис escalation ДОЛЖЕН позволять вручную продвигать уровень эскалации через POST `/api/escalations/v1/{tenant}/incidents/{incidentId}/escalate`.

#### Scenario: Ручной переход на следующий уровень

- **WHEN** выполняется POST на эндпоинт эскалации
- **THEN** эскалация немедленно переходит на следующий уровень и публикуется `escalation.triggered`
