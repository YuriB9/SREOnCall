# Frontend Shell

## Purpose

Оболочка SPA: OIDC-аутентификация через Keycloak, парсинг JWT-claims для мультитенантных ролей, маршрутизация с guards (TenantGuard, AdminGuard), глобальный макет с боковой навигацией, тёмная тема по умолчанию и звуковые уведомления.

## Requirements

### Requirement: OIDC-аутентификация через Keycloak

Система ДОЛЖНА (SHALL) аутентифицировать пользователей через Keycloak Authorization Code Flow с PKCE, используя `oidc-client-ts`. Access token ДОЛЖЕН (MUST) храниться в `sessionStorage`. Система ДОЛЖНА выполнять тихое обновление токена через скрытый iframe до истечения срока действия. Система ДОЛЖНА показывать баннер «Сессия истекает» с 30-секундным обратным отсчётом до выхода ТОЛЬКО когда обновление действительно не удаётся при приближении истечения токена, а не на единичный транзиентный сбой сети. При недоступности Keycloak в момент входа или возврата из Keycloak система ДОЛЖНА показывать понятный экран ошибки с возможностью повторить вход, НЕ ДОЛЖНА уходить в цикл редиректов и НЕ ДОЛЖНА оставлять пользователя на пустом экране.

#### Scenario: Неаутентифицированный пользователь открывает защищённый маршрут

- **WHEN** неаутентифицированный пользователь переходит на любой маршрут кроме `/callback`
- **THEN** система ДОЛЖНА перенаправить на endpoint авторизации Keycloak, сохранив исходный URL в OIDC-параметре `state`

#### Scenario: Успешный редирект после входа

- **WHEN** Keycloak перенаправляет обратно на `/callback` с валидным кодом авторизации
- **THEN** система ДОЛЖНА завершить PKCE-обмен, сохранить access token в `sessionStorage` и перенаправить пользователя на исходный URL назначения

#### Scenario: Тихое обновление токена

- **WHEN** до истечения access token остаётся менее 120 секунд
- **THEN** система ДОЛЖНА попытаться выполнить тихое обновление через iframe `check-session` Keycloak без прерывания работы пользователя

#### Scenario: Сбой тихого обновления при приближении истечения токена

- **WHEN** тихое обновление не удалось и до истечения access token осталось менее порога уведомления (либо после нескольких неудачных попыток подряд)
- **THEN** система ДОЛЖНА показать постоянный неотклоняемый баннер «Ваша сессия истекает. Вы будете выведены из системы через 30 секунд.» и перенаправить на вход через 30 секунд

#### Scenario: Транзиентный сбой тихого обновления при валидном токене

- **WHEN** тихое обновление не удалось из-за кратковременного сетевого сбоя, но access token ещё валиден (до истечения далеко)
- **THEN** система НЕ ДОЛЖНА показывать баннер выхода и НЕ ДОЛЖНА запускать 30-секундный отсчёт; последующее успешное обновление ДОЛЖНО сбросить состояние ошибки

#### Scenario: Keycloak недоступен в момент входа

- **WHEN** запуск входа (`signinRedirect`) завершается ошибкой из-за недоступности Keycloak
- **THEN** система ДОЛЖНА отрендерить экран «Сервис авторизации недоступен» с кнопкой повтора, а не оставить пользователя на пустом экране

#### Scenario: Ошибка завершения обмена на /callback

- **WHEN** обработка `/callback` завершается ошибкой (Keycloak недоступен, истёкший/неверный код, сетевой сбой)
- **THEN** система ДОЛЖНА отрендерить экран «Не удалось завершить вход» с кнопкой «Повторить вход» и НЕ ДОЛЖНА автоматически перенаправлять на `/`, чтобы не вызывать цикл редиректов

### Requirement: Парсинг JWT claims для определения тенанта и роли

Система ДОЛЖНА (SHALL) декодировать OIDC `id_token` (без проверки подписи) сразу после входа и извлечь claim `groups`. На основе `groups` система ДОЛЖНА сформировать карту `tenantRoles` вида `{ [tenantSlug: string]: 'member' | 'admin' }` и предоставить её через хук `usePermissions()`. Логика членства ДОЛЖНА согласовываться с бэкендом: любая группа `/{tenant}` или `/{tenant}/<сегмент>` означает членство в тенанте (`member`), а группа `/{tenant}/admins` повышает роль до `admin`.

#### Scenario: Пользователь с членством в нескольких тенантах

- **WHEN** claim `groups` в JWT содержит `["/team-alpha", "/team-beta", "/team-beta/admins"]`
- **THEN** `usePermissions()` ДОЛЖЕН вернуть `{ "team-alpha": "member", "team-beta": "admin" }`

#### Scenario: Членство по подгруппе с произвольным сегментом

- **WHEN** claim `groups` содержит `["/team-gamma/oncall"]` без отдельной записи `/team-gamma`
- **THEN** `usePermissions()` ДОЛЖЕН вернуть `{ "team-gamma": "member" }`, согласуясь с серверной проверкой `IsMember` по префиксу `/team-gamma/`

#### Scenario: Пользователь без членства ни в одном тенанте

- **WHEN** claim `groups` в JWT пуст или отсутствует
- **THEN** система ДОЛЖНА перенаправить на страницу ошибки «Нет доступа к командам» с объяснением, что у пользователя нет членства ни в одном тенанте

#### Scenario: Непустой groups, но пустая карта permissions (вероятная неверная настройка маппера)

- **WHEN** claim `groups` в токене непуст, но `usePermissions()` возвращает пустую карту
- **THEN** система ДОЛЖНА показать диагностическое сообщение о вероятной неверной настройке Keycloak-маппера групп, отличая это состояние от обычного «нет членства»

### Requirement: Гард тенантного маршрута

Система ДОЛЖНА (SHALL) отрендерить `<TenantGuard>` вокруг всех маршрутов `/[tenant]/*`. Гард ДОЛЖЕН проверить наличие записи роли текущего пользователя для сегмента `tenant` маршрута перед рендерингом дочернего маршрута.

#### Scenario: Пользователь обращается к тенанту, в котором не состоит

- **WHEN** пользователь переходит на `/unknown-team/incidents`
- **THEN** система ДОЛЖНА отрендерить страницу 403 с сообщением «У вас нет доступа к этой команде»

#### Scenario: Пользователь обращается к своему тенанту

- **WHEN** пользователь переходит на `/team-alpha/incidents` и `team-alpha` есть в его `tenantRoles`
- **THEN** система ДОЛЖНА отрендерить вид инцидентов без перенаправления

---

### Requirement: Гард административного маршрута

Система ДОЛЖНА (SHALL) отрендерить `<AdminGuard>` вокруг `/[tenant]/settings`. Гард ДОЛЖЕН проверить, что роль пользователя для тенанта маршрута равна `admin`.

#### Scenario: Пользователь с ролью member обращается к настройкам

- **WHEN** пользователь с ролью `member` для `team-alpha` переходит на `/team-alpha/settings`
- **THEN** система ДОЛЖНА отрендерить страницу 403 и НЕ ДОЛЖНА рендерить содержимое настроек

---

### Requirement: Лэндинговая страница выбора команды

Система ДОЛЖНА (SHALL) отрендерить страницу `/select-team` со списком всех тенантов из `tenantRoles` пользователя. Выбор команды выполняет навигацию на `/[team]/incidents`.

#### Scenario: Редирект с корневого URL

- **WHEN** аутентифицированный пользователь переходит на `/`
- **THEN** система ДОЛЖНА перенаправить на `/select-team`

#### Scenario: Пользователь с одной командой

- **WHEN** аутентифицированный пользователь состоит ровно в одном тенанте
- **THEN** система ДОЛЖНА перенаправить напрямую с `/` на `/[этот-тенант]/incidents`, минуя `/select-team`

---

### Requirement: Глобальный макет с навигацией

Система ДОЛЖНА (SHALL) рендерить постоянную боковую навигацию со ссылками на «Инциденты», «Расписания», «Эскалации» и «Настройки» в контексте текущего тенанта. Пункт «Настройки» ДОЛЖЕН быть разворачиваемой группой, видимой всем пользователям: при нажатии на него под ним раскрывается подменю. Подпункт «Мой профиль» (ведущий на `/[tenant]/settings/profile`) ДОЛЖЕН быть виден всем пользователям. Административные подпункты — «Webhook-токены», «Конфигурация уведомлений», «Участники команды» — ДОЛЖНЫ отображаться только администраторам текущего тенанта; каждый ведёт на собственный маршрут настроек. Активный подпункт ДОЛЖЕН визуально выделяться. Шапка ДОЛЖНА отображать название текущего тенанта, `preferred_username` вошедшего пользователя (текстом, без ссылки), переключатель тёмной/светлой темы и переключатель звуковых уведомлений.

#### Scenario: Навигация между разделами тенанта

- **WHEN** пользователь нажимает «Расписания» в боковой панели, находясь на `/team-alpha/incidents`
- **THEN** система ДОЛЖНА выполнить навигацию на `/team-alpha/schedules` без полной перезагрузки страницы

#### Scenario: Раскрытие подменю настроек

- **WHEN** администратор нажимает на пункт «Настройки» в боковой навигации
- **THEN** система ДОЛЖНА раскрыть подменю с подпунктами «Webhook-токены», «Конфигурация уведомлений» и «Участники команды»

#### Scenario: Переход в подраздел настроек

- **WHEN** администратор нажимает подпункт «Конфигурация уведомлений» в раскрытом подменю настроек
- **THEN** система ДОЛЖНА выполнить навигацию на `/[tenant]/settings/notifications` без полной перезагрузки страницы и выделить активный подпункт

#### Scenario: Доступ к профилю для обычного участника

- **WHEN** пользователь с ролью `member` (не администратор) раскрывает подменю «Настройки»
- **THEN** система ДОЛЖНА показать подпункт «Мой профиль» (без административных подпунктов) и при переходе по нему отрендерить страницу профиля на `/[tenant]/settings/profile`

---

### Requirement: Тёмная тема по умолчанию

Система ДОЛЖНА (SHALL) применять CSS-класс `dark` к `<html>` при первоначальной загрузке. Предпочтение темы пользователя ДОЛЖНО сохраняться в `localStorage` по ключу `oncall.colorScheme`. При последующих посещениях сохранённое предпочтение ДОЛЖНО применяться до первой отрисовки.

#### Scenario: Первое посещение (предпочтение не сохранено)

- **WHEN** пользователь впервые открывает приложение
- **THEN** ДОЛЖНА быть активна тёмная тема

#### Scenario: Восстановление предпочтения при повторном посещении

- **WHEN** пользователь, ранее выбравший светлую тему, снова открывает приложение
- **THEN** светлая тема ДОЛЖНА применяться до первой отрисовки
