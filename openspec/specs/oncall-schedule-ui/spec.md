# On-Call Schedule UI

## Purpose

Визуализация расписаний дежурств: виджет «Кто дежурит сейчас» с поллингом, помесячная Gantt-диаграмма со сменами и переопределениями, мобильный резервный список, модальное создание переопределений с обработкой конфликтов 409.

## Requirements

### Requirement: Виджет «Кто дежурит сейчас»

Система ДОЛЖНА (SHALL) отображать заметный виджет «Кто дежурит сейчас» в верхней части страницы расписаний. Виджет ДОЛЖЕН запрашивать `GET /api/schedules/v1/{tenant}/oncall?at=now` и отображать имя и логин дежурного инженера, название активного расписания и текущий уровень эскалации. Виджет ДОЛЖЕН обновляться каждые 60 секунд.

#### Scenario: Виджет показывает текущего дежурного инженера

- **WHEN** пользователь открывает страницу расписаний
- **THEN** виджет ДОЛЖЕН отобразить отображаемое имя, логин дежурного инженера и расписание, по которому он дежурит

#### Scenario: Никто не дежурит

- **WHEN** ответ `GET /oncall?at=now` указывает на отсутствие активного дежурства
- **THEN** виджет ДОЛЖЕН отобразить «Дежурный не назначен» в визуально выделенном состоянии предупреждения

---

### Requirement: Помесячная Gantt-диаграмма

Система ДОЛЖНА (SHALL) рендерить горизонтальную Gantt-диаграмму, визуализирующую смены текущего календарного месяца. Каждая ротация ДОЛЖНА отображаться цветной горизонтальной полосой, охватывающей даты смены. Система ДОЛЖНА запрашивать окно расписания через `GET /api/schedules/v1/{tenant}/schedules/{id}/oncall?from=<start>&to=<end>` для каждой ротации. Вся арифметика часовых поясов ДОЛЖНА использовать `date-fns-tz` и учитывать локальный часовой пояс пользователя. На экранах шириной менее 640px Gantt ДОЛЖНА скрываться и заменяться списком предстоящих смен на 7 дней.

#### Scenario: Рендер Gantt на десктопе

- **WHEN** пользователь открывает страницу расписаний на экране шириной ≥640px
- **THEN** система ДОЛЖНА отрендерить горизонтальную календарную сетку текущего месяца с полосами смен для каждой ротации

#### Scenario: Мобильный резервный вид

- **WHEN** пользователь открывает страницу расписаний на экране шириной <640px
- **THEN** Gantt ДОЛЖНА быть скрыта, а система ДОЛЖНА отрендерить список предстоящих смен на ближайшие 7 дней в хронологическом порядке

#### Scenario: Навигация по месяцам

- **WHEN** пользователь нажимает «Следующий месяц» или «Предыдущий месяц»
- **THEN** система ДОЛЖНА запросить и отрендерить данные смен для соседнего месяца

#### Scenario: Переход летнего/зимнего времени внутри смены

- **WHEN** окно смены охватывает переход DST
- **THEN** ширина полосы и подписи начала/конца ДОЛЖНЫ корректно отражать астрономическое время в локальном часовом поясе пользователя

---

### Requirement: Создание переопределения смены

Система ДОЛЖНА (SHALL) предоставлять кнопку «Создать переопределение», открывающую модальное диалоговое окно. Модальное окно ДОЛЖНО позволять выбрать дату/время начала, дату/время окончания и замещающего инженера из списка участников тенанта. При отправке система ДОЛЖНА выполнить POST на `POST /api/schedules/v1/{tenant}/schedules/{id}/overrides`. Ответ `409 Conflict` от бэкенда ДОЛЖЕН отображаться как встроенная ошибка внутри модального окна (не toast) с описанием конфликтующего окна переопределения.

#### Scenario: Успешное создание переопределения

- **WHEN** пользователь заполняет корректные данные переопределения и отправляет форму
- **THEN** система ДОЛЖНА выполнить POST, закрыть модальное окно при успехе и обновить Gantt, отобразив новую полосу переопределения

#### Scenario: Обработка ошибки конфликта

- **WHEN** бэкенд возвращает `409 Conflict` из-за пересечения переопределений
- **THEN** система ДОЛЖНА отобразить встроенную ошибку внутри модального окна «Переопределение конфликтует с существующим: [даты]» и оставить окно открытым для исправления

#### Scenario: Переопределение отображается на Gantt

- **WHEN** переопределение существует в отображаемом месяце
- **THEN** оно ДОЛЖНО рендериться как визуально отличимый элемент (другой паттерн или цвет) на Gantt, перекрывая базовую полосу ротации
