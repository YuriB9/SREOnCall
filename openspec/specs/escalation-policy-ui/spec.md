# Escalation Policy UI

## Purpose

Управление политиками эскалации: список с бейджем «По умолчанию», визуальный конструктор шагов (вертикальный степпер с выбором расписания и таймаутом), назначение политики по умолчанию для тенанта.

## Requirements

### Requirement: Список политик эскалации

Система ДОЛЖНА отображать список всех политик эскалации текущего тенанта, получаемых через `GET /api/escalations/v1/{tenant}/policies`. Для каждой политики ДОЛЖНЫ отображаться: название, количество шагов и признак того, является ли она политикой по умолчанию для тенанта. Система ДОЛЖНА позволять создавать новую политику через кнопку «Новая политика» и удалять не-дефолтную политику через действие удаления.

#### Scenario: Список политик отображается

- **WHEN** пользователь переходит на `/[tenant]/escalations`
- **THEN** система ДОЛЖНА отобразить все политики эскалации с названием, количеством шагов и бейджем «По умолчанию» на активной политике по умолчанию

#### Scenario: Удаление не-дефолтной политики

- **WHEN** пользователь нажимает «Удалить» на не-дефолтной политике
- **THEN** система ДОЛЖНА показать диалог подтверждения и при подтверждении удалить политику через `DELETE /api/escalations/v1/{tenant}/policies/{id}`

---

### Requirement: Визуальный конструктор шагов политики эскалации

Система ДОЛЖНА рендерить редактор политики эскалации в виде вертикального степпера. Каждый шаг ДОЛЖЕН содержать: номер шага, выпадающий список расписаний (заполненный из `GET /api/schedules/v1/{tenant}/schedules`) и поле ввода таймаута (в минутах, целое число, минимум 1). Пользователь ДОЛЖЕН иметь возможность добавлять, удалять и переупорядочивать шаги с помощью кнопок со стрелками вверх/вниз. Сохранение ДОЛЖНО отправлять `PUT /api/escalations/v1/{tenant}/policies/{id}` с полным обновлённым объектом политики.

#### Scenario: Добавление шага

- **WHEN** пользователь нажимает «Добавить шаг» в редакторе политики
- **THEN** система ДОЛЖНА добавить новую строку шага с пустым выпадающим списком расписаний и таймаутом по умолчанию 15 минут

#### Scenario: Удаление шага

- **WHEN** пользователь нажимает кнопку удаления на шаге (минимум 1 шаг должен оставаться)
- **THEN** система ДОЛЖНА удалить этот шаг и перенумеровать последующие шаги

#### Scenario: Переупорядочивание шагов

- **WHEN** пользователь нажимает стрелку «Вверх» на шаге 2
- **THEN** шаги 1 и 2 ДОЛЖНЫ поменяться местами

#### Scenario: Заполнение выпадающего списка расписаний

- **WHEN** редактор политики открывается
- **THEN** выпадающий список расписаний каждого шага ДОЛЖЕН быть заполнен всеми расписаниями, возвращёнными API расписаний тенанта

#### Scenario: Сохранение валидной политики

- **WHEN** во всех шагах выбрано расписание и указан валидный таймаут, и пользователь нажимает «Сохранить»
- **THEN** система ДОЛЖНА выполнить PUT политики и показать toast об успехе

#### Scenario: Сохранение с незаполненным шагом

- **WHEN** выпадающий список расписания одного из шагов пуст и пользователь нажимает «Сохранить»
- **THEN** система ДОЛЖНА отобразить встроенную ошибку валидации на незаполненном шаге и НЕ ДОЛЖНА отправлять запрос

---

### Requirement: Назначение политики по умолчанию

Система ДОЛЖНА позволять администратору назначить любую политику политикой тенанта по умолчанию, нажав «Сделать по умолчанию» на политике. Это ДОЛЖНО вызвать `PATCH /api/escalations/v1/{tenant}/policies/{id}` с `{"is_default": true}`. Предыдущая политика по умолчанию ДОЛЖНА потерять бейдж «По умолчанию» после ответа.

#### Scenario: Установка новой политики по умолчанию

- **WHEN** администратор нажимает «Сделать по умолчанию» на не-дефолтной политике
- **THEN** система ДОЛЖНА обновить политику по умолчанию, обновить список и показать бейдж «По умолчанию» на новой политике, убрав его с предыдущей
