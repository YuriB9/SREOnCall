# Tenant Settings UI

## Purpose

Административная страница тенанта: управление webhook-токенами (генерация с одноразовым показом, отзыв), конфигурация Mattermost/SMTP уведомлений, список участников только для чтения с информационным баннером о Keycloak. Доступна только администраторам тенанта.

## Requirements

### Requirement: Ограничение доступа только для администраторов

Административные разделы настроек тенанта ДОЛЖНЫ (SHALL) быть доступны только пользователям, у которых запись `tenantRoles` для текущего тенанта равна `admin`. Ограничение ДОЛЖНО распространяться на базовый маршрут `/[tenant]/settings` и на административные подмаршруты (`/[tenant]/settings/webhook-tokens`, `/[tenant]/settings/notifications`, `/[tenant]/settings/members`). Пользователи без роли администратора, переходящие на любой из этих маршрутов, ДОЛЖНЫ видеть страницу ошибки 403. В боковой навигации административные подпункты ДОЛЖНЫ быть скрыты для не-администраторов. Подраздел профиля `/[tenant]/settings/profile` НЕ ограничивается ролью администратора и доступен всем участникам тенанта; его подпункт «Мой профиль» в подменю «Настройки» виден всем пользователям.

#### Scenario: Администратор открывает настройки

- **WHEN** пользователь с ролью `admin` для текущего тенанта переходит на любой маршрут `/[tenant]/settings/*`
- **THEN** система ДОЛЖНА отрендерить соответствующий экран настроек

#### Scenario: Участник пытается открыть административный раздел

- **WHEN** пользователь с ролью `member` переходит на административный маршрут (`/[tenant]/settings`, `/[tenant]/settings/webhook-tokens`, `/[tenant]/settings/notifications` или `/[tenant]/settings/members`)
- **THEN** система ДОЛЖНА отрендерить страницу ошибки 403, не раскрывая никакого содержимого настроек

#### Scenario: Участник открывает свой профиль

- **WHEN** пользователь с ролью `member` переходит на `/[tenant]/settings/profile`
- **THEN** система ДОЛЖНА отрендерить страницу профиля без ошибки 403

---

### Requirement: Разделение настроек на отдельные экраны с подменю

Настройки тенанта ДОЛЖНЫ (SHALL) быть разбиты на три отдельных экрана, каждый со своим маршрутом:

- «Webhook-токены» — `/[tenant]/settings/webhook-tokens`
- «Конфигурация уведомлений» — `/[tenant]/settings/notifications`
- «Участники команды» — `/[tenant]/settings/members`

Каждый экран ДОЛЖЕН рендерить только свою секцию (а не все три стопкой на одной странице). Базовый маршрут `/[tenant]/settings` без подраздела ДОЛЖЕН выполнять редирект на `/[tenant]/settings/webhook-tokens`. Переход между подразделами ДОЛЖЕН выполняться через подменю боковой навигации без полной перезагрузки страницы.

#### Scenario: Открытие подраздела webhook-токенов

- **WHEN** администратор переходит на `/[tenant]/settings/webhook-tokens`
- **THEN** система ДОЛЖНА отрендерить только секцию управления webhook-токенами

#### Scenario: Открытие подраздела конфигурации уведомлений

- **WHEN** администратор переходит на `/[tenant]/settings/notifications`
- **THEN** система ДОЛЖНА отрендерить только форму конфигурации уведомлений

#### Scenario: Открытие подраздела участников

- **WHEN** администратор переходит на `/[tenant]/settings/members`
- **THEN** система ДОЛЖНА отрендерить только список участников команды

#### Scenario: Редирект с базового маршрута настроек

- **WHEN** администратор переходит на `/[tenant]/settings` без указания подраздела
- **THEN** система ДОЛЖНА выполнить редирект на `/[tenant]/settings/webhook-tokens`

---

### Requirement: Управление webhook-токенами

Система ДОЛЖНА (SHALL) отображать список существующих webhook-токенов тенанта, получаемых через `GET /api/schedules/v1/tenants/{tenant}/webhook-tokens`. Для каждого токена ДОЛЖНЫ отображаться: источник, дата создания и кнопка «Отозвать». Администратор ДОЛЖЕН иметь возможность сгенерировать новый токен через кнопку «Сгенерировать токен», открывающую диалог создания с выбором источника из списка поддерживаемых значений (`alertmanager`, `grafana`) — произвольный текст источника не допускается.

#### Scenario: Список токенов отображается

- **WHEN** администратор открывает страницу настроек
- **THEN** система ДОЛЖНА отобразить все существующие webhook-токены с источником, датой создания и кнопкой отзыва для каждого

#### Scenario: Генерация нового токена

- **WHEN** администратор нажимает «Сгенерировать токен», выбирает источник из списка и подтверждает
- **THEN** система ДОЛЖНА выполнить POST на `/api/schedules/v1/tenants/{tenant}/webhook-tokens` с выбранным `source`, получить plaintext-токен в ответе и открыть модальное окно одноразового показа

#### Scenario: Модальное окно одноразового показа токена

- **WHEN** бэкенд возвращает только что сгенерированный plaintext-токен
- **THEN** система ДОЛЖНА отобразить токен в копируемом поле с моноширинным шрифтом внутри модального окна; окно ДОЛЖНО содержать обязательный чекбокс «Я скопировал этот токен», который ДОЛЖЕН быть отмечен перед активацией кнопки «Закрыть»; после закрытия plaintext-токен ДОЛЖЕН быть удалён из состояния клиента

#### Scenario: Отзыв токена

- **WHEN** администратор нажимает «Отозвать» на токене и подтверждает диалог подтверждения
- **THEN** система ДОЛЖНА удалить токен через `DELETE /api/schedules/v1/tenants/{tenant}/webhook-tokens/{id}` и убрать его из списка

---

### Requirement: Форма конфигурации уведомлений

Система ДОЛЖНА (SHALL) отображать конфигурацию уведомлений уровня тенанта, получаемую через `GET /api/schedules/v1/tenants/{tenant}/notification-config`, в виде **двух раздельных логических секций** — «Mattermost» и «Email». Каждая секция ДОЛЖНА иметь собственный заголовок, собственный набор полей и собственную кнопку «Сохранить»; сохранение одной секции НЕ ДОЛЖНО требовать заполнения полей другой. Сохранение ДОЛЖНО выполнять `PUT /api/schedules/v1/tenants/{tenant}/notification-config`.

Секция «Mattermost» ДОЛЖНА содержать переключатель `mattermost_enabled` и поля `mattermost_webhook_url` и `mattermost_channel`. Переключатель `mattermost_enabled` симметричен `email_enabled` секции Email и предзаполняется значением из GET-ответа. Если канал включён (`mattermost_enabled = true`), но webhook не задан (ни сохранённый, ни введённый), форма ДОЛЖНА показывать ненавязчивое предупреждение о том, что уведомления не будут отправляться, не блокируя сохранение.

Поле `mattermost_webhook_url` НЕ ДОЛЖНО предзаполняться маскированным значением из GET-ответа; вместо этого рядом с пустым полем отображается текущий маскированный URL как справочная информация. Если пользователь не вводил новое значение URL, форма НЕ ДОЛЖНА включать поле `mattermost_webhook_url` в тело PUT-запроса — сохранённый вебхук не может быть затёрт неявно.

Секция «Email» ДОЛЖНА содержать поля `smtp_from`, `email_reply_to`, `email_subject_prefix` и переключатель `email_enabled`. Поля `email_reply_to` и `email_subject_prefix` НЕ являются секретами и ДОЛЖНЫ предзаполняться значениями из GET-ответа. При выключенном `email_enabled` остальные поля секции Email ДОЛЖНЫ оставаться видимыми и редактируемыми (отключение канала не стирает сохранённые значения).

#### Scenario: Отображение двух секций

- **WHEN** администратор открывает `/[tenant]/settings/notifications`
- **THEN** система ДОЛЖНА отрендерить две раздельные секции — «Mattermost» (с переключателем `mattermost_enabled` и полями `mattermost_webhook_url`, `mattermost_channel`) и «Email» (с переключателем `email_enabled` и полями `smtp_from`, `email_reply_to`, `email_subject_prefix`), каждую со своей кнопкой «Сохранить»

#### Scenario: Сохранение секции Mattermost

- **WHEN** администратор редактирует поле `mattermost_channel` в секции Mattermost и нажимает «Сохранить» этой секции
- **THEN** тело PUT-запроса ДОЛЖНО содержать `mattermost_enabled` и `mattermost_channel`, система ДОЛЖНА выполнить PUT конфига и показать toast об успехе

#### Scenario: Предупреждение о включённом канале без webhook

- **WHEN** в секции Mattermost переключатель `mattermost_enabled` включён, но webhook не задан (ни сохранённый маскированный, ни введённый)
- **THEN** система ДОЛЖНА показать ненавязчивое предупреждение, что уведомления не будут отправляться до указания webhook, и НЕ ДОЛЖНА блокировать сохранение

#### Scenario: Отключение Mattermost-канала сохраняет webhook

- **WHEN** администратор выключает переключатель `mattermost_enabled` и сохраняет секцию Mattermost, не вводя новый webhook
- **THEN** PUT-запрос содержит `mattermost_enabled: false` и НЕ содержит `mattermost_webhook_url`, а сохранённый вебхук остаётся без изменений

#### Scenario: Сохранение без ввода URL не затирает вебхук

- **WHEN** администратор сохраняет секцию Mattermost, оставив поле `mattermost_webhook_url` пустым (новое значение не вводилось)
- **THEN** тело PUT-запроса НЕ содержит поле `mattermost_webhook_url`, и сохранённый вебхук тенанта остаётся без изменений

#### Scenario: Ввод нового URL обновляет вебхук

- **WHEN** администратор вводит непустой валидный URL и сохраняет секцию Mattermost
- **THEN** PUT-запрос содержит новое значение `mattermost_webhook_url`, и оно заменяет сохранённое

#### Scenario: Невалидный URL

- **WHEN** администратор вводит значение, не являющееся валидным URL, в поле `mattermost_webhook_url` и пытается сохранить
- **THEN** система ДОЛЖНА показать встроенную ошибку валидации и НЕ ДОЛЖНА отправлять запрос

#### Scenario: Сохранение секции Email с новыми полями

- **WHEN** администратор редактирует `email_reply_to` и `email_subject_prefix` и нажимает «Сохранить» секции Email
- **THEN** тело PUT-запроса ДОЛЖНО содержать `email_enabled`, `smtp_from`, `email_reply_to` и `email_subject_prefix`, и система ДОЛЖНА показать toast об успехе

#### Scenario: Невалидные email-адреса

- **WHEN** администратор вводит в `smtp_from` или `email_reply_to` значение, не являющееся валидным email
- **THEN** система ДОЛЖНА показать встроенную ошибку валидации и НЕ ДОЛЖНА отправлять запрос

#### Scenario: Отключение email-канала сохраняет значения

- **WHEN** администратор выключает переключатель `email_enabled` и сохраняет секцию Email
- **THEN** PUT-запрос содержит `email_enabled: false`, а сохранённые `smtp_from`, `email_reply_to`, `email_subject_prefix` остаются без изменений и продолжают отображаться в форме

---

### Requirement: Список участников только для чтения с уведомлением о Keycloak

Система ДОЛЖНА (SHALL) отображать список участников тенанта только для чтения, получаемый через `GET /api/schedules/v1/tenants/{tenant}/members`. Для каждого участника ДОЛЖНЫ отображаться отображаемое имя и логин. Список ДОЛЖЕН сопровождаться статическим информационным сообщением: «Состав команды управляется в Keycloak. Для добавления или удаления участников используйте консоль администратора Keycloak.»

#### Scenario: Список участников отображается

- **WHEN** администратор открывает страницу настроек
- **THEN** система ДОЛЖНА отобразить полный список участников с отображаемым именем и логином каждого

#### Scenario: Отсутствие элементов управления добавлением/удалением

- **WHEN** администратор просматривает список участников
- **THEN** в приложении НЕ ДОЛЖНО быть кнопок или элементов управления для добавления или удаления участников напрямую
