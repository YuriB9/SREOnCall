# Incident Dashboard UI

## Purpose

Дашборд инцидентов для SRE-дежурных: таблица с фильтрацией и автообновлением, панель деталей с вкладками «Алерты»/«История»/«Комментарии», кнопки «Подтвердить» и «Закрыть» с оптимистичными обновлениями, горячие клавиши и звуковые уведомления о новых инцидентах.

## Requirements

### Requirement: Список инцидентов с фильтрацией и пагинацией

Система ДОЛЖНА (SHALL) отображать постраничную таблицу инцидентов текущего тенанта, получаемых через `GET /api/incidents/v1/{tenant}/incidents`. Список ДОЛЖЕН автоматически обновляться каждые 12 секунд через фоновый поллинг. Система ДОЛЖНА поддерживать фильтрацию по статусу (мультивыбор `open`, `acknowledged`, `resolved`; значения передаются одним параметром через запятую) и по критичности (одно значение). Состояние фильтров ДОЛЖНО отражаться в query-параметрах URL, чтобы отфильтрованные виды можно было передавать ссылкой.

#### Scenario: Первоначальная загрузка

- **WHEN** пользователь переходит на `/[tenant]/incidents`
- **THEN** система ДОЛЖНА отобразить таблицу инцидентов, отсортированных по `created_at` по убыванию, с колонками: Критичность, Название, Статус, Источник, Создан, Подтверждён кем

#### Scenario: Фильтрация по статусу

- **WHEN** пользователь выбирает «Открытые» в фильтре статуса
- **THEN** система ДОЛЖНА перезапросить данные с `?status=open`, отобразить только открытые инциденты и обновить query-параметр в URL

#### Scenario: Фильтрация по нескольким статусам

- **WHEN** пользователь выбирает «Открытые» и «Подтверждённые» одновременно
- **THEN** система ДОЛЖНА перезапросить данные с `?status=open,acknowledged` и отобразить инциденты обоих статусов

#### Scenario: Автоматическое фоновое обновление

- **WHEN** с момента последнего успешного запроса прошло 12 секунд
- **THEN** система ДОЛЖНА запросить список инцидентов повторно без действий пользователя и обновить отображаемые данные при их изменении

#### Scenario: Обнаружен новый инцидент во время поллинга

- **WHEN** ответ поллинга содержит ID инцидента, отсутствовавший в предыдущем ответе, со статусом `open`
- **THEN** система ДОЛЖНА обновить список и, если звуковые уведомления включены, воспроизвести одиночный сигнал через Web Audio API

---

### Requirement: Панель деталей инцидента с разделённым видом

Система ДОЛЖНА (SHALL) открывать панель деталей инцидента справа от списка при нажатии на строку инцидента. Панель ДОЛЖНА содержать три вкладки: «Алерты», «История», «Комментарии». ID выбранного инцидента ДОЛЖЕН синхронизироваться в search-параметр URL `?incident=<id>` для возможности поделиться ссылкой. При закрытии панели параметр ДОЛЖЕН удаляться.

#### Scenario: Открытие деталей инцидента

- **WHEN** пользователь нажимает на строку инцидента в списке
- **THEN** система ДОЛЖНА открыть панель деталей без навигации со страницы списка и обновить URL, добавив `?incident=<id>`

#### Scenario: Вкладка «Алерты»

- **WHEN** активна вкладка «Алерты» в панели деталей
- **THEN** система ДОЛЖНА отобразить все алерты, сгруппированные в этот инцидент (`GET /incidents/{id}/alerts`), с бейджами источника и fingerprint и статусом (firing/resolved)

#### Scenario: Вкладка «История»

- **WHEN** активна вкладка «История» в панели деталей
- **THEN** система ДОЛЖНА отобразить вертикальный таймлайн журнала инцидента (`GET /incidents/{id}/history`): изменения статуса, изменения лейблов и комментарии в хронологическом порядке; события эскалации в таймлайн не подмешиваются (история эскалации ведётся escalation-сервисом отдельно)

#### Scenario: Вкладка «Комментарии»

- **WHEN** активна вкладка «Комментарии» в панели деталей
- **THEN** система ДОЛЖНА отобразить список комментариев в стиле чата с именем автора, временной меткой и текстом, а также поле ввода для публикации нового комментария

#### Scenario: Публикация комментария

- **WHEN** пользователь отправляет непустой комментарий на вкладке «Комментарии»
- **THEN** система ДОЛЖНА выполнить POST на `POST /api/incidents/v1/{tenant}/incidents/{id}/comments` и добавить новый комментарий в список при успехе

### Requirement: Действия «Подтвердить» и «Закрыть»

Система ДОЛЖНА (SHALL) отображать заметные кнопки «Подтвердить» и «Закрыть» в панели деталей инцидента. «Подтвердить» ДОЛЖНА отправлять `PATCH /api/incidents/v1/{tenant}/incidents/{id}` с `{"status": "acknowledged"}`. «Закрыть» ДОЛЖНА отправлять `PATCH` с `{"status": "resolved"}`. Кнопки ДОЛЖНЫ быть заблокированы и показывать индикатор загрузки во время выполнения запроса. После успешного изменения статус ДОЛЖЕН немедленно обновиться в панели и в строке списка через оптимистичное обновление.

#### Scenario: Подтверждение открытого инцидента

- **WHEN** пользователь нажимает «Подтвердить» на открытом инциденте
- **THEN** система ДОЛЖНА немедленно показать статус «Подтверждён», затем подтвердить через ответ PATCH; при ошибке API оптимистичное обновление ДОЛЖНО быть откачено и показан toast с ошибкой

#### Scenario: Состояние кнопок для уже подтверждённого инцидента

- **WHEN** в панели деталей выбран инцидент со статусом `acknowledged`
- **THEN** кнопка «Подтвердить» ДОЛЖНА быть заблокирована, а кнопка «Закрыть» — активна

#### Scenario: Состояние кнопок для закрытого инцидента

- **WHEN** в панели деталей выбран инцидент со статусом `resolved`
- **THEN** обе кнопки — «Подтвердить» и «Закрыть» — ДОЛЖНЫ быть заблокированы

---

### Requirement: Горячие клавиши для работы с инцидентами

Система ДОЛЖНА (SHALL) обрабатывать горячие клавиши, когда ни одно поле ввода не в фокусе. Клавиша `A` ДОЛЖНА запускать «Подтвердить» для текущего выбранного/открытого инцидента. Клавиша `R` ДОЛЖНА запускать «Закрыть». Клавиша `/` ДОЛЖНА переводить фокус в поле поиска/фильтра. Клавиши `J` и `K` ДОЛЖНЫ перемещать выделение вниз и вверх по списку. `Escape` ДОЛЖНА закрывать панель деталей.

#### Scenario: Подтверждение через клавиатуру

- **WHEN** инцидент выбран и пользователь нажимает `A` без фокуса в поле ввода
- **THEN** система ДОЛЖНА выполнить то же действие, что и нажатие «Подтвердить»

#### Scenario: Горячая клавиша игнорируется при фокусе в поле ввода

- **WHEN** пользователь держит фокус в текстовом поле или textarea и нажимает `A`
- **THEN** система НЕ ДОЛЖНА запускать действие «Подтвердить»

---

### Requirement: Звуковые уведомления о новых инцидентах

Система ДОЛЖНА (SHALL) предоставлять переключатель звуковых уведомлений в шапке приложения, по умолчанию выключенный. Когда переключатель включён и вкладка браузера активна, система ДОЛЖНА синтезировать короткий сигнал через Web Audio API при каждом появлении нового инцидента в статусе `open`. Состояние переключателя ДОЛЖНО сохраняться в `localStorage` по ключу `oncall.audioEnabled`.

#### Scenario: Переключатель звука сохраняется при перезагрузке

- **WHEN** пользователь включает звуковые уведомления и перезагружает страницу
- **THEN** звуковые уведомления ДОЛЖНЫ оставаться включёнными после перезагрузки

#### Scenario: Нет звука при скрытой вкладке

- **WHEN** вкладка браузера свёрнута на фоне и появляется новый инцидент
- **THEN** система НЕ ДОЛЖНА воспроизводить звук (чтобы не пугать инженера при возвращении)
