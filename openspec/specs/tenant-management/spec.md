# Tenant Management

## Purpose

Управление тенантами (командами), участниками (только чтение через Keycloak), вебхук-токенами для ingestion и per-tenant конфигурацией уведомлений.

## Requirements

### Requirement: Создание и управление тенантами

Сервис scheduling ДОЛЖЕН (SHALL) предоставлять CRUD-эндпоинты для тенантов (команд) по адресу `/api/schedules/v1/tenants`.

Тенант ДОЛЖЕН включать: `slug` (уникальный URL-safe идентификатор), `name` (отображаемое название).

Создание тенанта (`POST /api/schedules/v1/tenants`) доступно любому аутентифицированному пользователю (или сервисному ключу); тенантной роли для создания не требуется. Тенант становится работоспособным только после создания соответствующей Keycloak-группы `/{slug}` (ручной шаг администратора Keycloak) — до этого ни один пользователь не имеет к нему доступа.

Удаление тенанта удаляет только запись тенанта в scheduling. Связанные данные (расписания, переопределения, вебхук-токены, включая их Redis-индекс, конфигурация уведомлений) и данные других сервисов (инциденты, состояния эскалации, журнал уведомлений) НЕ удаляются — это документированное ограничение; каскадная очистка (внутрисервисная и межсервисная) потребует отдельного изменения. Администраторам рекомендуется отозвать вебхук-токены тенанта перед его удалением.

#### Scenario: Создание тенанта

- **WHEN** аутентифицированный пользователь выполняет POST на `/api/schedules/v1/tenants` с валидными `slug` и `name`
- **THEN** тенант сохраняется и возвращается с HTTP 201 и сгенерированным `id`

#### Scenario: Slug должен быть уникальным

- **WHEN** выполняется POST с `slug`, уже занятым другим тенантом
- **THEN** сервис возвращает HTTP 409

#### Scenario: Обновление тенанта

- **WHEN** выполняется PATCH на `/api/schedules/v1/tenants/{slug}` администратором тенанта
- **THEN** поля `name` обновляются; `slug` изменить нельзя после создания

#### Scenario: Удаление тенанта

- **WHEN** выполняется DELETE на `/api/schedules/v1/tenants/{slug}` администратором тенанта
- **THEN** запись тенанта удаляется и тенант исчезает из списка; связанные данные scheduling и других сервисов сохраняются (документированное ограничение)

---

### Requirement: Просмотр участников тенанта

Источник истины о членстве — Keycloak. Платформа ДОЛЖНА (SHALL) читать участников через Keycloak Admin API; добавление и удаление производится только через Keycloak Admin UI.

Роли определяются по структуре Keycloak-групп: участник `/team-slug` → `member`; участник `/team-slug/admins` → `admin`.

#### Scenario: Список участников

- **WHEN** участник тенанта выполняет GET `/api/schedules/v1/tenants/{slug}/members`
- **THEN** сервис запрашивает Keycloak Admin API для группы `team-slug` и возвращает список `(user_id, preferred_username, role)`

#### Scenario: Пользователь состоит в нескольких тенантах

- **WHEN** пользователь является членом нескольких Keycloak-групп
- **THEN** GET `/api/schedules/v1/tenants/{slug}/members` возвращает только его для соответствующей группы; принадлежность к другим тенантам не раскрывается

### Requirement: Управление вебхук-токенами для ingestion

Сервис scheduling ДОЛЖЕН (SHALL) позволять администраторам создавать и отзывать вебхук-токены для каждого поддерживаемого источника алертов (`alertmanager`, `grafana`) в рамках тенанта. Значение `source` вне этого множества ДОЛЖНО отклоняться с HTTP 422.

Токен генерируется сервером, отображается однократно при создании, хранится в виде SHA-256 хэша. При создании/отзыве токена сервис обновляет индекс `oncall:tokens:{hash}` в Redis (HSET/HDEL).

#### Scenario: Создание токена

- **WHEN** администратор выполняет POST на `/api/schedules/v1/tenants/{slug}/webhook-tokens` с полем `source` равным `alertmanager` или `grafana`
- **THEN** генерируется токен, возвращается в открытом виде однократно; в БД и Redis сохраняется только хэш

#### Scenario: Неподдерживаемый источник отклоняется

- **WHEN** администратор выполняет POST с `source: zabbix` или иным значением вне множества `alertmanager | grafana`
- **THEN** сервис возвращает HTTP 422 с перечнем поддерживаемых источников

#### Scenario: Отзыв токена

- **WHEN** администратор выполняет DELETE на `/api/schedules/v1/tenants/{slug}/webhook-tokens/{id}`
- **THEN** токен удаляется из БД и Redis; дальнейшие вебхуки с этим токеном возвращают HTTP 401

#### Scenario: Один тенант — несколько токенов на источник

- **WHEN** создаётся второй токен с тем же `source` для одного тенанта
- **THEN** оба токена активны одновременно (позволяет ротацию без простоя)

### Requirement: Конфигурация уведомлений тенанта

Сервис scheduling ДОЛЖЕН (SHALL) хранить per-tenant конфигурацию каналов уведомлений, используемую сервисом notification при отправке.

Конфигурация ДОЛЖНА включать: `mattermost_webhook_url`, `mattermost_channel`, `smtp_from` (опционально, переопределяет глобальный).

Конфигурация ДОЛЖНА быть доступна по HTTP **исключительно** через `GET/PUT /api/schedules/v1/tenants/{slug}/notification-config`. Альтернативные маршруты к этому ресурсу (в частности, `GET/PUT /api/schedules/v1/{tenant}/notification-config`) НЕ ДОЛЖНЫ существовать: любой обходной путь, возвращающий немаскированный `mattermost_webhook_url` по пользовательскому JWT или перезаписывающий сохранённый URL пустым значением, является нарушением данного требования.

Маскирование `mattermost_webhook_url` (отображение только схемы и хоста) ДОЛЖНО применяться только к запросам, аутентифицированным пользовательским JWT. Запросы, аутентифицированные сервисным ключом (`X-Admin-Key`), ДОЛЖНЫ получать полный URL. При неопределённом способе аутентификации URL ДОЛЖЕН маскироваться (безопасное умолчание).

PUT конфигурации с пустым или отсутствующим `mattermost_webhook_url` ДОЛЖЕН сохранять текущее сохранённое значение URL без изменений; остальные переданные поля обновляются. Перезапись URL происходит только при передаче непустого значения.

#### Scenario: Установка конфигурации

- **WHEN** администратор выполняет PUT на `/api/schedules/v1/tenants/{slug}/notification-config` с валидными полями
- **THEN** конфигурация сохраняется и становится активной для всех уведомлений тенанта

#### Scenario: Получение конфигурации

- **WHEN** выполняется GET на `/api/schedules/v1/tenants/{slug}/notification-config` пользователем (JWT)
- **THEN** возвращается текущая конфигурация; `mattermost_webhook_url` маскируется (отображается только домен)

#### Scenario: Получение конфигурации сервисом

- **WHEN** выполняется GET на `/api/schedules/v1/tenants/{slug}/notification-config` с валидным заголовком `X-Admin-Key`
- **THEN** возвращается конфигурация с полным немаскированным `mattermost_webhook_url`

#### Scenario: Обходные маршруты отсутствуют

- **WHEN** выполняется GET или PUT на `/api/schedules/v1/{tenant}/notification-config` (tenant-scoped путь вне канонического `/tenants/{slug}/...`)
- **THEN** сервис возвращает HTTP 404/405; немаскированный `mattermost_webhook_url` недоступен ни по одному маршруту без сервисной аутентификации

#### Scenario: PUT с пустым URL не затирает сохранённый вебхук

- **WHEN** администратор выполняет PUT с `mattermost_channel: "#new"` и пустым/отсутствующим `mattermost_webhook_url`, при том что для тенанта уже сохранён непустой URL
- **THEN** `mattermost_channel` обновляется, а сохранённый `mattermost_webhook_url` остаётся прежним

#### Scenario: PUT с непустым URL обновляет вебхук

- **WHEN** администратор выполняет PUT с непустым валидным `mattermost_webhook_url`
- **THEN** сохранённый URL заменяется переданным значением

### Requirement: Изоляция данных между тенантами

Все доменные сущности (инциденты, расписания, политики эскалации, уведомления) ДОЛЖНЫ (SHALL) быть изолированы по `tenant_id`. Запросы в контексте одного тенанта НЕ ДОЛЖНЫ возвращать или изменять данные другого тенанта.

#### Scenario: Изоляция инцидентов

- **WHEN** пользователь, состоящий в двух тенантах, запрашивает инциденты тенанта A
- **THEN** возвращаются только инциденты тенанта A; инциденты тенанта B не видны

#### Scenario: Несанкционированный доступ к тенанту

- **WHEN** пользователь, не являющийся участником тенанта, обращается к его ресурсам
- **THEN** сервис возвращает HTTP 403
