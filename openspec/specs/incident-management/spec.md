# Incident Management

## Purpose

Создание инцидентов из алертов, управление их жизненным циклом, группировка алертов, ведение лейблов, комментариев и журнала истории.

## Requirements

### Requirement: Создание инцидента из алерта

Сервис incident ДОЛЖЕН (SHALL) потреблять сообщения из очереди `alerts.incident` в RabbitMQ и создавать новый инцидент для каждого firing-алерта, не связанного с существующим открытым инцидентом.

Severity и заголовок инцидента наследуются от **первого** алерта, создавшего инцидент, и НЕ пересчитываются при привязке последующих алертов. Resolved-алерт, для которого не найден соответствующий алерт открытого инцидента (инцидент уже закрыт, либо firing был ранее дедуплицирован), ДОЛЖЕН игнорироваться без ошибки.

При создании инцидента из алерта поле `tenant_slug` ДОЛЖНО заполняться значением `tenant_id` алерта (в событийном конвейере `tenant_id` является slug'ом тенанта: Redis-индекс вебхук-токенов хранит slug). События `incident.created` и `incident.updated`, публикуемые consumer'ом, ДОЛЖНЫ содержать непустой `tenant_slug`; публикация события с пустым `tenant_slug` НЕ ДОПУСКАЕТСЯ, поскольку нижестоящие сервисы (escalation, notification) используют slug для резолва дежурного, получения per-tenant конфигурации и построения ссылки на инцидент.

#### Scenario: Новый firing-алерт создаёт инцидент

- **WHEN** из очереди `alerts.incident` потреблено сообщение о firing-алерте
- **THEN** создаётся новый инцидент со статусом `open`, severity и заголовком из алерта; алерт привязывается к нему, и в RabbitMQ публикуется событие `incident.created` с полями `incident_id`, `tenant_id`, `tenant_slug`, `status`, `title`, `severity`

#### Scenario: tenant_slug заполняется при создании из алерта

- **WHEN** consumer создаёт инцидент из алерта с `tenant_id` равным slug'у тенанта
- **THEN** инцидент сохраняется с `tenant_slug`, равным `tenant_id` алерта, и событие `incident.created` содержит этот непустой `tenant_slug`

#### Scenario: tenant_slug в событии авто-резолва

- **WHEN** consumer закрывает инцидент по resolved-алерту и публикует `incident.updated`
- **THEN** событие содержит непустой `tenant_slug` инцидента

#### Scenario: Severity не пересчитывается при группировке

- **WHEN** к открытому инциденту с severity `warning` привязывается новый алерт с severity `critical`
- **THEN** severity инцидента остаётся `warning` (значение первого алерта)

#### Scenario: Resolved-алерт при одном алерте в инциденте

- **WHEN** потреблено сообщение о resolved-алерте и fingerprint совпадает с единственным алертом открытого инцидента
- **THEN** статус инцидента переходит в `resolved` и публикуется событие `incident.updated`

#### Scenario: Частичный resolve при группированном инциденте

- **WHEN** потреблено сообщение о resolved-алерте и инцидент содержит другие firing-алерты
- **THEN** алерт помечается как `resolved` в `incident_alerts`, но статус инцидента остаётся `open` до тех пор, пока все привязанные алерты не перейдут в `resolved`

#### Scenario: Resolved-алерт без открытого инцидента

- **WHEN** потреблено сообщение о resolved-алерте, fingerprint которого не совпадает ни с одним алертом открытого инцидента тенанта
- **THEN** сообщение подтверждается (ack) и игнорируется без создания инцидента и без ошибки

### Requirement: Настройка группирующих лейблов per-source

Сервис incident ДОЛЖЕН (SHALL) позволять администраторам тенанта задавать список лейблов для группировки алертов отдельно для каждого поддерживаемого источника (`alertmanager`, `grafana`). Конфигурация управляется через REST API; если правило не задано — используются умолчания.

Умолчания: `alertmanager` → `["alertname", "job"]`; `grafana` → `["alertname"]`.

Значение `{source}` в `PUT` и `DELETE /api/incidents/v1/{tenant}/grouping-rules/{source}` вне множества `alertmanager | grafana` ДОЛЖНО отклоняться с HTTP 422 (симметрично валидации источников вебхук-токенов).

При обработке алерта со значением `source: prometheus` (сообщения, опубликованные ingestion до канонизации имени источника) сервис ДОЛЖЕН трактовать его как `alertmanager`: поиск правила группировки и умолчания выполняются по имени `alertmanager`, чтобы заданные администратором правила применялись к таким алертам наравне с новыми.

#### Scenario: Установка группирующих лейблов для источника

- **WHEN** администратор выполняет PUT на `/api/incidents/v1/{tenant}/grouping-rules/{source}` с массивом `grouping_labels` для поддерживаемого источника
- **THEN** правило сохраняется; новые алерты из этого источника будут группироваться по указанным лейблам

#### Scenario: Неподдерживаемый источник отклоняется

- **WHEN** администратор выполняет PUT или DELETE на `/api/incidents/v1/{tenant}/grouping-rules/zabbix` (или с иным значением вне множества `alertmanager | grafana`)
- **THEN** сервис возвращает HTTP 422 с перечнем поддерживаемых источников; правило не сохраняется и не удаляется

#### Scenario: Алерт со старым source prometheus использует правила alertmanager

- **WHEN** из очереди потреблён алерт с `source: prometheus`, а для тенанта задано правило группировки для `alertmanager`
- **THEN** алерт группируется по правилу `alertmanager` (заданному или умолчанию), а не по встроенному фолбэку

#### Scenario: Сброс к умолчанию

- **WHEN** администратор выполняет DELETE на `/api/incidents/v1/{tenant}/grouping-rules/{source}`
- **THEN** правило удаляется и используется встроенное умолчание для источника

#### Scenario: Получение текущих правил

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/grouping-rules`
- **THEN** возвращаются правила для обоих поддерживаемых источников (явно заданные или умолчания с флагом `is_default: true`)

### Requirement: Привязка нескольких алертов к инциденту

Сервис incident ДОЛЖЕН (SHALL) позволять группировать несколько алертов в один инцидент — автоматически (по совпадению лейблов) или вручную через API.

#### Scenario: Алерты с совпадающими группирующими лейблами объединяются

- **WHEN** новый алерт из источника `S` содержит значения всех группирующих лейблов, совпадающие с открытым инцидентом того же тенанта (правила для `S` из `incident_grouping_rules`)
- **THEN** алерт привязывается к существующему инциденту вместо создания нового

#### Scenario: Ручная привязка алерта через API

- **WHEN** выполняется POST на `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts` с валидным ID алерта
- **THEN** алерт привязывается к указанному инциденту

---

### Requirement: Управление жизненным циклом инцидента

Сервис incident ДОЛЖЕН (SHALL) отслеживать переходы статусов инцидента: `open` → `acknowledged` → `resolved` (и повторное открытие из `resolved` в `open`).

#### Scenario: Подтверждение инцидента

- **WHEN** PATCH-запрос устанавливает статус `acknowledged` для открытого инцидента
- **THEN** статус инцидента обновляется, записываются `acknowledged_at` и `acknowledged_by`, и публикуется событие `incident.updated`

#### Scenario: Закрытие инцидента

- **WHEN** PATCH-запрос устанавливает статус `resolved`
- **THEN** записывается `resolved_at` и инцидент закрывается

#### Scenario: Повторное открытие закрытого инцидента

- **WHEN** PATCH-запрос устанавливает статус `open` для resolved-инцидента
- **THEN** инцидент переоткрывается и в истории фиксируется событие повторного открытия

---

### Requirement: Лейблы инцидента

Сервис incident ДОЛЖЕН (SHALL) позволять устанавливать и обновлять произвольные лейблы (ключ-значение) на инциденте.

Лейблы исходного алерта копируются в инцидент только при его создании; лейблы алертов, привязанных к инциденту позднее (группировкой или вручную), НЕ мёржатся в лейблы инцидента автоматически.

#### Scenario: Установка лейблов при создании

- **WHEN** инцидент создаётся из алерта
- **THEN** лейблы этого алерта копируются в инцидент в качестве начальных лейблов

#### Scenario: Лейблы догруппированного алерта не мёржатся

- **WHEN** к существующему инциденту привязывается новый алерт с дополнительными лейблами
- **THEN** набор лейблов инцидента не изменяется; лейблы алерта доступны на вкладке алертов инцидента

#### Scenario: Добавление или обновление лейбла

- **WHEN** выполняется PUT на `/api/incidents/v1/{tenant}/incidents/{incidentId}/labels` с картой ключ-значение
- **THEN** переданные лейблы объединяются с текущим набором лейблов инцидента

---

### Requirement: Комментарии к инциденту

Сервис incident ДОЛЖЕН (SHALL) позволять авторизованным пользователям добавлять, просматривать и удалять текстовые комментарии к инциденту.

#### Scenario: Добавление комментария

- **WHEN** выполняется POST на `/api/incidents/v1/{tenant}/incidents/{incidentId}/comments` с непустым телом
- **THEN** комментарий сохраняется с автором (`sub` из JWT) и временной меткой, возвращается с HTTP 201

#### Scenario: Список комментариев

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}/comments`
- **THEN** все комментарии к инциденту возвращаются в порядке возрастания временной метки

---

### Requirement: История изменений инцидента

Сервис incident ДОЛЖЕН (SHALL) записывать каждое изменение статуса, изменение лейбла и комментарий как неизменяемую запись журнала аудита.

#### Scenario: Изменение статуса фиксируется в истории

- **WHEN** статус инцидента изменяется
- **THEN** в историю добавляется запись с временной меткой, автором, старым и новым значением

#### Scenario: История доступна только на добавление

- **WHEN** запрашивается любая запись истории
- **THEN** прошлые записи не могут быть изменены или удалены через API

---

### Requirement: Список и фильтрация инцидентов

Сервис incident ДОЛЖЕН (SHALL) предоставлять GET `/api/incidents/v1/{tenant}/incidents` с фильтрацией по `status`, `severity`, `from_time` и `to_time` и курсорной пагинацией (`cursor`, `limit`).

Параметр `status` ДОЛЖЕН принимать одно или несколько значений, разделённых запятой (например, `?status=open,acknowledged`); инцидент попадает в выдачу, если его статус равен любому из перечисленных. Каждое значение ДОЛЖНО валидироваться по множеству `open | acknowledged | resolved`; при недопустимом значении сервис возвращает HTTP 400.

#### Scenario: Фильтр по статусу

- **WHEN** GET-запрос содержит `?status=open`
- **THEN** возвращаются только открытые инциденты

#### Scenario: Фильтр по нескольким статусам

- **WHEN** GET-запрос содержит `?status=open,acknowledged`
- **THEN** возвращаются инциденты со статусом `open` или `acknowledged`; инциденты `resolved` не возвращаются

#### Scenario: Недопустимое значение статуса

- **WHEN** GET-запрос содержит `?status=open,bogus`
- **THEN** сервис возвращает HTTP 400 с указанием недопустимого значения

#### Scenario: Пагинация

- **WHEN** фильтру соответствует больше `limit` инцидентов
- **THEN** ответ содержит страницу результатов и курсор для запроса следующей страницы

---

### Requirement: Детали инцидента и его алерты

Сервис incident ДОЛЖЕН (SHALL) предоставлять GET `/api/incidents/v1/{tenant}/incidents/{incidentId}` (полные данные инцидента, включая лейблы) и GET `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts` (все привязанные алерты с их статусом firing/resolved, источником, fingerprint и временем привязки).

#### Scenario: Получение деталей инцидента

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}` участником тенанта
- **THEN** возвращаются данные инцидента: статус, severity, заголовок, лейблы, временные метки, `acknowledged_by`/`acknowledged_at` (если есть)

#### Scenario: Получение алертов инцидента

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts`
- **THEN** возвращаются все привязанные алерты с fingerprint, источником, статусом и временем привязки (`attached_at`)

#### Scenario: Инцидент другого тенанта недоступен

- **WHEN** выполняется GET деталей инцидента, принадлежащего другому тенанту
- **THEN** сервис возвращает HTTP 404
