# Incident Management

## Purpose

Создание инцидентов из алертов, управление их жизненным циклом, группировка алертов, ведение лейблов, комментариев и журнала истории.

## Requirements

### Requirement: Создание инцидента из алерта

Сервис incident ДОЛЖЕН потреблять сообщения из очереди `alerts.incident` в RabbitMQ и создавать новый инцидент для каждого firing-алерта, не связанного с существующим открытым инцидентом.

#### Scenario: Новый firing-алерт создаёт инцидент

- **WHEN** из очереди `alerts.incident` потреблено сообщение о firing-алерте
- **THEN** создаётся новый инцидент со статусом `open`, алерт привязывается к нему, и в RabbitMQ публикуется событие `incident.created`

#### Scenario: Resolved-алерт при одном алерте в инциденте

- **WHEN** потреблено сообщение о resolved-алерте и fingerprint совпадает с единственным алертом открытого инцидента
- **THEN** статус инцидента переходит в `resolved` и публикуется событие `incident.updated`

#### Scenario: Частичный resolve при группированном инциденте

- **WHEN** потреблено сообщение о resolved-алерте и инцидент содержит другие firing-алерты
- **THEN** алерт помечается как `resolved` в `incident_alerts`, но статус инцидента остаётся `open` до тех пор, пока все привязанные алерты не перейдут в `resolved`

---

### Requirement: Настройка группирующих лейблов per-source

Сервис incident ДОЛЖЕН позволять администраторам тенанта задавать список лейблов для группировки алертов отдельно для каждого источника (`alertmanager`, `grafana`, `zabbix`). Конфигурация управляется через REST API; если правило не задано — используются умолчания.

Умолчания: `alertmanager` → `["alertname", "job"]`; `grafana` → `["alertname"]`; `zabbix` → `["host", "trigger_name"]`.

#### Scenario: Установка группирующих лейблов для источника

- **WHEN** администратор выполняет PUT на `/api/incidents/v1/{tenant}/grouping-rules/{source}` с массивом `grouping_labels`
- **THEN** правило сохраняется; новые алерты из этого источника будут группироваться по указанным лейблам

#### Scenario: Сброс к умолчанию

- **WHEN** администратор выполняет DELETE на `/api/incidents/v1/{tenant}/grouping-rules/{source}`
- **THEN** правило удаляется и используется встроенное умолчание для источника

#### Scenario: Получение текущих правил

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/grouping-rules`
- **THEN** возвращаются правила для всех трёх источников (явно заданные или умолчания с флагом `is_default: true`)

---

### Requirement: Привязка нескольких алертов к инциденту

Сервис incident ДОЛЖЕН позволять группировать несколько алертов в один инцидент — автоматически (по совпадению лейблов) или вручную через API.

#### Scenario: Алерты с совпадающими группирующими лейблами объединяются

- **WHEN** новый алерт из источника `S` содержит значения всех группирующих лейблов, совпадающие с открытым инцидентом того же тенанта (правила для `S` из `incident_grouping_rules`)
- **THEN** алерт привязывается к существующему инциденту вместо создания нового

#### Scenario: Ручная привязка алерта через API

- **WHEN** выполняется POST на `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts` с валидным ID алерта
- **THEN** алерт привязывается к указанному инциденту

---

### Requirement: Управление жизненным циклом инцидента

Сервис incident ДОЛЖЕН отслеживать переходы статусов инцидента: `open` → `acknowledged` → `resolved` (и повторное открытие из `resolved` в `open`).

#### Scenario: Подтверждение инцидента

- **WHEN** PATCH-запрос устанавливает статус `acknowledged` для открытого инцидента
- **THEN** статус инцидента обновляется, записываются `acknowledged_at` и `acknowledged_by`, и публикуется событие `incident.updated`

#### Scenario: Закрытие инцидента

- **WHEN** PATCH-запрос устанавливает статус `resolved`
- **THEN** записывается `resolved_at` и инцидент закрывается

#### Scenario: Повторное открытие закрытого инцидента

- **WHEN** PATCH-запрос устанавливает статус `open` для resolved-инцидента
- **THEN** инцидент переоткрывается и в истории фиксируется событие повторного открытия

---

### Requirement: Лейблы инцидента

Сервис incident ДОЛЖЕН позволять устанавливать и обновлять произвольные лейблы (ключ-значение) на инциденте.

#### Scenario: Установка лейблов при создании

- **WHEN** инцидент создаётся (из алерта или через API)
- **THEN** лейблы из исходного алерта копируются в инцидент в качестве начальных лейблов

#### Scenario: Добавление или обновление лейбла

- **WHEN** выполняется PUT на `/api/incidents/v1/{tenant}/incidents/{incidentId}/labels` с картой ключ-значение
- **THEN** переданные лейблы объединяются с текущим набором лейблов инцидента

---

### Requirement: Комментарии к инциденту

Сервис incident ДОЛЖЕН позволять авторизованным пользователям добавлять, просматривать и удалять текстовые комментарии к инциденту.

#### Scenario: Добавление комментария

- **WHEN** выполняется POST на `/api/incidents/v1/{tenant}/incidents/{incidentId}/comments` с непустым телом
- **THEN** комментарий сохраняется с автором (`sub` из JWT) и временной меткой, возвращается с HTTP 201

#### Scenario: Список комментариев

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}/comments`
- **THEN** все комментарии к инциденту возвращаются в порядке возрастания временной метки

---

### Requirement: История изменений инцидента

Сервис incident ДОЛЖЕН записывать каждое изменение статуса, изменение лейбла и комментарий как неизменяемую запись журнала аудита.

#### Scenario: Изменение статуса фиксируется в истории

- **WHEN** статус инцидента изменяется
- **THEN** в историю добавляется запись с временной меткой, автором, старым и новым значением

#### Scenario: История доступна только на добавление

- **WHEN** запрашивается любая запись истории
- **THEN** прошлые записи не могут быть изменены или удалены через API

---

### Requirement: Список и фильтрация инцидентов

Сервис incident ДОЛЖЕН (SHALL) предоставлять GET `/api/incidents/v1/{tenant}/incidents` с фильтрацией по `status`, `severity`, `label`, `from_time` и `to_time`.

Параметр `status` ДОЛЖЕН принимать одно или несколько значений, разделённых запятой (например, `?status=open,acknowledged`); инцидент попадает в выдачу, если его статус равен любому из перечисленных. Каждое значение ДОЛЖНО валидироваться по множеству `open | acknowledged | resolved`; при недопустимом значении сервис возвращает HTTP 400.

#### Scenario: Фильтр по статусу

- **WHEN** GET-запрос содержит `?status=open`
- **THEN** возвращаются только открытые инциденты

#### Scenario: Фильтр по нескольким статусам

- **WHEN** GET-запрос содержит `?status=open,acknowledged`
- **THEN** возвращаются инциденты со статусом `open` или `acknowledged`; инциденты `resolved` не возвращаются

#### Scenario: Недопустимое значение статуса

- **WHEN** GET-запрос содержит `?status=open,bogus`
- **THEN** сервис возвращает HTTP 400 с указанием недопустимого значения

#### Scenario: Пагинация

- **WHEN** фильтру соответствует больше `page_size` инцидентов
- **THEN** результаты разбиваются на страницы с курсором или токеном смещения
