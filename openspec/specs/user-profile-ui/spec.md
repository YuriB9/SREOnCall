# User Profile UI

## Purpose

Страница личного профиля инженера `/profile`: форма контактных данных (Mattermost username, email), переключатели каналов уведомлений с немедленным сохранением и guard против включения канала без заполненных контактов.

## Requirements

### Requirement: Форма личных контактных данных

Система ДОЛЖНА (SHALL) отображать страницу профиля по адресу `/profile`, доступную любому аутентифицированному пользователю, с возможностью просмотра и редактирования личных контактных данных. Контакты привязаны к паре `(user_id, tenant_id)` и обслуживаются notification-сервисом: загрузка через `GET /api/notifications/v1/{tenant}/contacts/{userId}`, сохранение через `PUT /api/notifications/v1/{tenant}/contacts/{userId}`, где `{userId}` — `sub` текущего пользователя.

Если пользователь состоит в нескольких тенантах, страница ДОЛЖНА отображать селектор тенанта (на основе `tenantRoles` из `usePermissions()`); форма показывает и сохраняет контакты выбранного тенанта. При единственном тенанте селектор скрывается и используется он.

Форма ДОЛЖНА содержать поля для имени пользователя в Mattermost и адреса электронной почты. Ответ 404 от API трактуется как «контакты ещё не настроены» (пустая форма).

#### Scenario: Страница профиля загружается с существующими данными

- **WHEN** аутентифицированный пользователь переходит на `/profile`
- **THEN** система ДОЛЖНА загрузить контакты пользователя для выбранного тенанта и предзаполнить имя в Mattermost и email в полях формы

#### Scenario: Переключение тенанта в селекторе

- **WHEN** пользователь, состоящий в нескольких тенантах, выбирает другой тенант в селекторе
- **THEN** система ДОЛЖНА загрузить и отобразить контакты пользователя для выбранного тенанта

#### Scenario: Сохранение контактных данных

- **WHEN** пользователь обновляет имя в Mattermost и сохраняет
- **THEN** система ДОЛЖНА выполнить PUT на `/api/notifications/v1/{tenant}/contacts/{userId}` с обновлёнными контактами и показать toast об успехе

#### Scenario: Невалидный формат email

- **WHEN** пользователь вводит значение, не являющееся валидным адресом email, и пытается сохранить
- **THEN** система ДОЛЖНА показать встроенную ошибку валидации и НЕ ДОЛЖНА отправлять запрос

---

### Requirement: Переключатели каналов уведомлений

Система ДОЛЖНА (SHALL) отображать переключатели для каждого поддерживаемого канала уведомлений (`email`, `mattermost`) под полями контактных данных. Каждый переключатель ДОЛЖЕН отражать текущий массив `enabled_channels` из ответа API контактов. Переключение канала ДОЛЖНО немедленно выполнять PUT с обновлённым массивом `enabled_channels`.

#### Scenario: Отключение email-уведомлений

- **WHEN** пользователь выключает переключатель «Email»
- **THEN** система ДОЛЖНА выполнить PUT с `{ "enabled_channels": ["mattermost"] }` (или пустым массивом, если Mattermost тоже выключен) и отразить новое состояние в UI

#### Scenario: Включение канала при отсутствии контактных данных

- **WHEN** пользователь пытается включить уведомления в Mattermost, но имя пользователя в Mattermost не указано
- **THEN** система ДОЛЖНА показать встроенное предупреждение «Укажите имя пользователя в Mattermost перед включением этого канала» и оставить переключатель в выключенном состоянии до тех пор, пока имя не будет сохранено
