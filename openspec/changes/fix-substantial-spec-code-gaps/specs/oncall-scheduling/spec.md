## MODIFIED Requirements

### Requirement: Переопределения расписания

Сервис scheduling ДОЛЖЕН (SHALL) поддерживать разовые переопределения, заменяющие запланированного дежурного на конкретный временной интервал.

Ответ `409 Conflict` при пересечении переопределений ДОЛЖЕН содержать, помимо поля `error`, данные конфликтующего переопределения: `existing_start` и `existing_end` (временные метки RFC3339) и `existing_user` (`user_id` конфликтующего переопределения) — этот контракт используется UI для встроенного сообщения о конфликте.

#### Scenario: Создание переопределения

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules/{id}/overrides` с `user_id`, `start_at` и `end_at`
- **THEN** переопределение сохраняется; в течение этого окна переопределённый пользователь возвращается как дежурный

#### Scenario: Переопределение имеет приоритет над ротацией

- **WHEN** текущее время попадает в активное окно переопределения
- **THEN** эндпоинт вычисления дежурного возвращает пользователя переопределения, а не пользователя из ротации

#### Scenario: Пересекающиеся переопределения отклоняются

- **WHEN** новое переопределение подаётся с временным окном, пересекающимся с существующим
- **THEN** сервис возвращает HTTP 409 с телом, содержащим `error`, `existing_start`, `existing_end` (RFC3339) и `existing_user` конфликтующего переопределения
