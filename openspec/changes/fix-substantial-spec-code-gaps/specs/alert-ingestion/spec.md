## MODIFIED Requirements

### Requirement: Нормализация алертов в единую схему

Сервис ingestion ДОЛЖЕН (SHALL) нормализовывать все входящие алерты из любого источника в каноническую схему алерта перед дедупликацией и публикацией.

Каноническая схема ДОЛЖНА включать: `fingerprint`, `source`, `severity`, `title`, `description`, `labels` (карта ключ-значение), `status` (firing|resolved), `fired_at`, `received_at`.

Поле `source` ДОЛЖНО принимать значение `alertmanager` для алертов, принятых через вебхук Alertmanager, и `grafana` — через вебхук Grafana. Эти значения совпадают со словарём источников в API вебхук-токенов и правил группировки; значение `prometheus` НЕ ДОЛЖНО использоваться для новых алертов.

Поле `severity` ДОЛЖНО принимать одно из значений `critical | high | warning | info` (маппинг из лейбла `severity` источника; неизвестные значения → `info`).

#### Scenario: Лейблы сохраняются после нормализации

- **WHEN** алерт из любого источника нормализуется
- **THEN** все специфичные для источника лейблы отображаются в каноническую карту `labels` без потери данных

#### Scenario: Source алерта Alertmanager

- **WHEN** алерт принят через `/api/ingest/v1/webhook/alertmanager`
- **THEN** нормализованный алерт имеет `source: alertmanager`, совпадающий со значением `source` вебхук-токена и правил группировки

#### Scenario: Канонические значения severity

- **WHEN** алерт содержит лейбл `severity` со значением вне множества `critical | high | warning | info` (или не содержит его)
- **THEN** нормализованный алерт получает `severity: info`

### Requirement: Дедупликация алертов через Redis

Сервис ingestion ДОЛЖЕН (SHALL) вычислять SHA-256 fingerprint по отсортированной карте канонических `labels`, полю `source` и `tenant_id` (для изоляции дедупликации между тенантами), и использовать Redis SETNX с настраиваемым TTL (по умолчанию 4 часа, соответствует типичному `repeat_interval` Alertmanager) для дедупликации идентичных алертов в пределах окна.

Длинный TTL не задерживает обработку повторных срабатываний после устранения проблемы: resolved-алерт удаляет ключ дедупликации, и следующий firing того же алерта проходит немедленно.

#### Scenario: Дублирующий алерт в пределах окна дедупликации

- **WHEN** алерт с идентичным fingerprint поступает в пределах TTL дедупликации
- **THEN** алерт отбрасывается и не публикуется в RabbitMQ

#### Scenario: Тот же алерт после истечения окна дедупликации

- **WHEN** алерт с тем же fingerprint поступает после истечения ключа Redis
- **THEN** алерт обрабатывается как новый и публикуется в RabbitMQ

#### Scenario: Resolved-алерт обходит дедупликацию

- **WHEN** поступает resolved-алерт
- **THEN** он всегда публикуется в RabbitMQ вне зависимости от состояния дедупликации, а ключ дедупликации удаляется
