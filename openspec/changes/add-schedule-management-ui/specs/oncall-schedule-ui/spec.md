## ADDED Requirements

### Requirement: Создание и управление расписаниями

Система ДОЛЖНА (SHALL) предоставлять в UI возможность создавать, редактировать и
удалять расписания дежурств, используя существующие эндпоинты сервиса scheduling:
`POST /api/schedules/v1/{tenant}/schedules`, `PATCH .../schedules/{id}`,
`DELETE .../schedules/{id}`.

На странице расписаний ДОЛЖНА присутствовать кнопка «Создать расписание»,
доступная в том числе при отсутствии расписаний. Кнопка ДОЛЖНА открывать модальное
окно с полями: название (`name`), часовой пояс (`timezone`, по умолчанию `UTC`),
продолжительность смены (`shift_duration`, ISO-8601, выбор из пресетов), дата старта
ротации (`start_date`, формат `YYYY-MM-DD`) и упорядоченная ротация инженеров
(`rotation` — массив `user_id`), выбираемых из участников тенанта. Порядок инженеров
в ротации ДОЛЖЕН быть редактируемым и определять очередь дежурств.

При отправке формы создания система ДОЛЖНА выполнить
`POST /api/schedules/v1/{tenant}/schedules` с указанными полями. Ответ `422`
(валидация: `missing_fields` или `error`) ДОЛЖЕН отображаться встроенной ошибкой
внутри модального окна (не toast), оставляя окно открытым. При успехе (`201`) окно
ДОЛЖНО закрываться, а список расписаний (Gantt и виджет «Кто дежурит сейчас») —
обновляться.

Каждое существующее расписание ДОЛЖНО иметь в UI действия «Редактировать» и
«Удалить». Редактирование ДОЛЖНО открывать ту же форму с предзаполненными полями и
отправлять `PATCH`. Удаление ДОЛЖНО запрашивать подтверждение и выполнять `DELETE`;
после успеха расписание ДОЛЖНО исчезать из интерфейса.

#### Scenario: Создание первого расписания на пустом тенанте

- **WHEN** у тенанта нет расписаний и пользователь нажимает «Создать расписание», заполняет название, продолжительность смены, дату старта и ротацию минимум из одного инженера, и отправляет форму
- **THEN** система ДОЛЖНА выполнить `POST .../schedules`, закрыть модальное окно при успехе и отрендерить новое расписание в Gantt-диаграмме и в виджете «Кто дежурит сейчас»

#### Scenario: Ошибка валидации при создании

- **WHEN** пользователь отправляет форму с незаполненными обязательными полями или пустой ротацией и бэкенд возвращает `422`
- **THEN** система ДОЛЖНА отобразить встроенную ошибку валидации внутри модального окна и оставить его открытым для исправления

#### Scenario: Редактирование расписания

- **WHEN** пользователь выбирает «Редактировать» у существующего расписания, изменяет поля (например, ротацию или продолжительность смены) и сохраняет
- **THEN** система ДОЛЖНА выполнить `PATCH .../schedules/{id}` и после успеха отразить изменения в списке и Gantt

#### Scenario: Удаление расписания

- **WHEN** пользователь выбирает «Удалить» у расписания и подтверждает действие
- **THEN** система ДОЛЖНА выполнить `DELETE .../schedules/{id}` и убрать расписание из Gantt и виджета «Кто дежурит сейчас»

## MODIFIED Requirements

### Requirement: Помесячная Gantt-диаграмма

Система ДОЛЖНА (SHALL) рендерить горизонтальную Gantt-диаграмму, визуализирующую смены текущего календарного месяца. Каждое расписание ДОЛЖНО отображаться строкой с цветными полосами смен. Система ДОЛЖНА запрашивать окно смен **одним запросом на расписание** через `GET /api/schedules/v1/{tenant}/schedules/{scheduleId}/shifts?from=<начало месяца>&to=<начало следующего месяца>` (RFC3339); ответ содержит смены с `user_id`, `start_at`, `end_at` и флагом `is_override`. Арифметика дат ДОЛЖНА учитывать локальный часовой пояс пользователя. На экранах шириной менее 640px Gantt ДОЛЖНА скрываться и заменяться списком предстоящих смен на 7 дней. Если у тенанта нет ни одного расписания, Gantt ДОЛЖНА отображать пустое состояние с кнопкой «Создать расписание» (а не предлагать создавать расписание через API).

#### Scenario: Рендер Gantt на десктопе

- **WHEN** пользователь открывает страницу расписаний на экране шириной ≥640px
- **THEN** система ДОЛЖНА отрендерить календарную сетку текущего месяца с полосами смен для каждого расписания; переопределения визуально отличимы (пунктирная рамка)

#### Scenario: Мобильный резервный вид

- **WHEN** пользователь открывает страницу расписаний на экране шириной <640px
- **THEN** Gantt ДОЛЖНА быть скрыта, а система ДОЛЖНА отрендерить список предстоящих смен на ближайшие 7 дней в хронологическом порядке

#### Scenario: Навигация по месяцам

- **WHEN** пользователь нажимает «Следующий месяц» или «Предыдущий месяц»
- **THEN** система ДОЛЖНА запросить `/shifts` с окном соседнего месяца и отрендерить его смены

#### Scenario: Переход летнего/зимнего времени внутри смены

- **WHEN** окно смены охватывает переход DST
- **THEN** ширина полосы и подписи начала/конца ДОЛЖНЫ корректно отражать астрономическое время в локальном часовом поясе пользователя

#### Scenario: Пустое состояние предлагает создать расписание

- **WHEN** у тенанта нет ни одного расписания
- **THEN** Gantt-диаграмма ДОЛЖНА показать пустое состояние с кнопкой «Создать расписание», открывающей форму создания
