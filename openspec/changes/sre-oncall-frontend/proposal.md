# Фронтенд SRE OnCall

## Зачем

Бэкенд SRE OnCall (sre-oncall-platform) предоставляет полноценный REST API, однако у него нет пользовательского интерфейса — инженеры не могут взаимодействовать с инцидентами, расписаниями и политиками эскалации иначе как через сырые API-вызовы. Необходим специально разработанный веб-фронтенд, ставящий скорость и клавиатурное управление выше эстетики, чтобы дежурный инженер мог подтвердить инцидент и оценить его масштаб за считанные секунды с любого устройства.

## Что меняется

- Новое самостоятельное фронтенд-приложение (React + Vite SPA), раздаваемое как статический бандл рядом с существующими Go-микросервисами.
- **Оболочка аутентификации**: Keycloak Authorization Code Flow с PKCE через `oidc-client-ts`; JWT claims разбираются на клиенте для определения ролей пользователя в каждом тенанте (`/tenant/members`, `/tenant/admins`).
- **Мультитенантный роутинг**: структура URL `/[tenant]/...` зеркалирует параметр `{tenant}` бэкенда; middleware-гарды перенаправляют неаутентифицированных пользователей и блокируют административные маршруты для не-администраторов.
- **Дашборд инцидентов**: список с поллингом (10–15 сек через TanStack Query) с фильтрацией и сортировкой; панель деталей с разделённым видом (вкладки «Алерты», «История», «Комментарии»); действия «Подтвердить» / «Закрыть» в один клик; горячие клавиши (`A`, `R`, `/`).
- **UI расписаний дежурств**: помесячная Gantt-диаграмма смен; виджет «Кто дежурит сейчас»; модальное окно создания переопределения с обработкой конфликта 409.
- **Конструктор политик эскалации**: вертикальный степпер для создания и редактирования многоуровневых политик эскалации; выпадающий список расписаний из бэкенда.
- **Настройки тенанта** (только для администраторов): список webhook-токенов с одноразовым модальным показом; форма конфигурации уведомлений (Mattermost webhook, SMTP); список участников только для чтения со ссылкой на Keycloak UI.
- **Профиль пользователя**: поля личных контактов (имя пользователя в Mattermost, email) и переключатели каналов уведомлений.
- **UX-улучшения для SRE**: тёмная тема (Tailwind + Shadcn/ui), опциональный звуковой сигнал при появлении новых инцидентов в статусе `open`, адаптивная вёрстка для подтверждения инцидентов с мобильного.

## Возможности

### Новые возможности

- `frontend-shell`: Точка входа в приложение — scaffolding проекта Vite/React, интеграция с Keycloak OIDC, мультитенантный роутер, ролевые гарды маршрутов, глобальный макет (боковая навигация, шапка, переключатель тёмной темы, переключатель звуковых уведомлений).
- `incident-dashboard-ui`: Список инцидентов с фильтрацией по статусу/критичности/источнику и пагинацией; панель деталей с разделённым видом (вкладки «Алерты», «История» (временная шкала статусов), «Комментарии»); кнопки «Подтвердить» и «Закрыть»; горячие клавиши; поллинг через TanStack Query.
- `oncall-schedule-ui`: Помесячная Timeline/Gantt-визуализация ротаций смен; виджет «Кто дежурит сейчас» через `/oncall?at=now`; модальное окно создания переопределения смены с обработкой ошибки конфликта.
- `escalation-policy-ui`: Вертикальный степпер для создания и редактирования политик эскалации (выпадающий список расписаний, поле таймаута для каждого шага); назначение политики по умолчанию для тенанта.
- `tenant-settings-ui`: Управление webhook-токенами (генерация / отзыв, одноразовый показ); форма конфигурации уведомлений; список участников только для чтения; гард маршрута только для администраторов.
- `user-profile-ui`: Форма личных контактных данных и панель переключателей включённых каналов уведомлений.

## Влияние

- Новая кодовая база в директории `frontend/` (или `web/`); существующие Go-микросервисы не затрагиваются.
- Зависимость во время выполнения от всех REST API бэкенда: инциденты, алерты, комментарии, история, расписания, переопределения, политики эскалации, webhook-токены, конфигурация уведомлений, контакты пользователей.
- Зависимость от аутентификации: конфигурация realm/client в Keycloak должна раскрывать claim `groups` и разрешать PKCE flow с origin фронтенда.
- Сборка: инструментарий Node.js (Vite, TypeScript); результат — статический бандл, который можно раздавать через любое из существующих правил Ingress или выделенный контейнер nginx.
- Новые эндпоинты бэкенда для v1 не требуются; все данные доступны через существующую API-поверхность.
