## Why

Redis в текущей конфигурации работает без персистентности (`--save "" --appendonly no`), а сервис scheduling наполняет индекс вебхук-токенов `oncall:tokens:{hash}` только в момент создания/отзыва токена. После перезапуска или сброса Redis индекс теряется, а механизма восстановления из Postgres при старте нет. В результате токен «осиротевает»: хэш остаётся в `scheduling.tenant_webhook_tokens`, но отсутствует в Redis, и ingestion отклоняет вебхуки с HTTP 401, хотя токен формально действителен.

## What Changes

- При старте сервиса scheduling, после применения миграций и подключения к Redis, выполняется **регидрация индекса токенов**: все активные хэши из `scheduling.tenant_webhook_tokens` загружаются в Redis (`HSET oncall:tokens:{hash} tenant_id {slug}`).
- Регидрация идемпотентна и не зависит от текущего содержимого Redis (повторный запуск безопасен; ранее присутствующие записи перезаписываются тем же значением).
- Отсутствие или недоступность Redis на старте не блокирует запуск сервиса — поведение остаётся прежним (индекс отключён, в лог пишется предупреждение), регидрация просто пропускается.
- Добавляется метод чтения всех хэшей токенов с их slug из стораджа scheduling.

Не является breaking-изменением: публичный HTTP/событийный контракт не меняется, меняется только поведение при инициализации.

### Modified Capabilities
- `tenant-management`: к требованию «Управление вебхук-токенами для ingestion» добавляется правило о восстановлении Redis-индекса токенов из Postgres при старте scheduling, чтобы валидные токены продолжали резолвиться после перезапуска/сброса Redis.

## Impact

- **Сервисы:** `services/scheduling` — последовательность инициализации в `cmd/server/main.go`; `internal/tokenindex` (массовая загрузка); `internal/store` (чтение всех активных хэшей токенов).
- **Инфраструктура:** Redis (тот же ключ `oncall:tokens:{hash}`, без TTL). Postgres — только чтение существующей таблицы `scheduling.tenant_webhook_tokens`, схема не меняется.
- **Косвенно:** `services/ingestion` — устраняется ложный HTTP 401 на вебхуках после рестарта Redis (поведение самого ingestion не меняется).
- **События RabbitMQ:** не затрагиваются.
- **API:** не затрагивается.
