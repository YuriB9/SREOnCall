# Контекст

Новая платформа дежурств для SRE-команд, работающих на Kubernetes. Миграция с легаси не требуется. Локальная разработка — на k3s; production-цель — любой Kubernetes-кластер. Ключевая инфраструктура: PostgreSQL (персистентное хранилище), Redis (дедупликация, кэш, rate-limiting), RabbitMQ (асинхронная шина между сервисами). RabbitMQ используется как внешний сервис в production; в локальной разработке поднимается в k3s через Helm-чарт Bitnami. Платформа мультитенантная: тенант = команда (один деплой, одна БД, изоляция по `tenant_id`). Все сервисы написаны на Go.

## Цели / не-цели

**Цели:**

- Мультитенантность: тенант = команда; каждая команда имеет собственные расписания, политики эскалации, инциденты и конфигурацию уведомлений; пользователь может состоять в нескольких командах.
- Приём и дедупликация алертов из Zabbix, Grafana и Prometheus Alertmanager через HTTP-вебхуки; идентификация тенанта по вебхук-токену.
- Группировка алертов в инциденты с полным журналом изменений (лейблы, комментарии, история статусов).
- Управление расписаниями дежурств и ротациями per-tenant; вычисление дежурного в любой момент времени.
- Применение политик эскалации при отсутствии подтверждения инцидента.
- Рассылка уведомлений по Email (SMTP) и Mattermost (входящие вебхуки per-tenant).
- Каждый домен — отдельный Go-микросервис в собственном Kubernetes Deployment.
- Воспроизводимая локальная разработка через `k3s` + Helm/Helmfile.

**Не-цели:**

- UI / фронтенд (только API в первой версии).
- Правила замолчания и ингибиции алертов (обрабатываются в Alertmanager).
- SaaS-мультитенантность (отдельные организации): тенант = внутренняя команда одной организации.
- Push-уведомления на мобильные устройства.
- Инструменты миграции с ServiceNow / PagerDuty / OpsGenie.

## Решения

### 1. Декомпозиция на микросервисы

Пять сервисов: `ingestion`, `incident`, `scheduling`, `escalation`, `notification`.

**Обоснование**: у каждого домена своя модель данных, профиль нагрузки и внешние зависимости. `ingestion` испытывает пики во время волн алертов; `notification` требует изолированного rate-limiting; `escalation` управляется таймерами. Объединение усложнило бы независимое масштабирование и создало бы риски при деплое несвязанных изменений.

**Рассмотренная альтернатива**: монолит. Проще для старта, но усложняет независимое масштабирование и делает необязательной единую шину событий, что ломает изоляцию сервисов.

### 2. RabbitMQ как внутренняя шина событий

Всё взаимодействие между сервисами — через AMQP. Топология exchanges и очередей:

- exchange `alerts` (topic) → очередь `alerts.incident` → сервис `incident`
- exchange `incidents` (topic) → очередь `incidents.escalation` → сервис `escalation`
- exchange `escalations` (topic) → очередь `escalations.notification` → сервис `notification`

Все exchanges и очереди — durable. Потребители используют manual ACK.

**Обоснование**: RabbitMQ уже эксплуатируется как внешний сервис в production — нет операционных издержек на поддержку нового брокера. Kafka избыточна для объёма on-call алертов: replay-семантика и партиционирование нам не нужны. Fan-out через topic exchange с отдельными очередями обеспечивает ту же изоляцию потребителей.

**Рассмотренная альтернатива**: Kafka. Даёт replay и Consumer Groups, но требует Zookeeper/KRaft, сложнее операционно, избыточна для данной нагрузки.

### 3. Дедупликация алертов через Redis в `ingestion`

Входящие алерты получают fingerprint (хэш отсортированных лейблов + поле source). Redis SETNX с настраиваемым TTL (по умолчанию 5 минут) защищает от повторной публикации одного алерта в рамках окна дедупликации.

**Обоснование**: Redis SET/SETNX атомарен и работает за субмиллисекунды; подходит для огневого характера обработки вебхуков.

**Рассмотренная альтернатива**: уникальный индекс в БД. Работает, но добавляет задержку при записи и нагрузку на базу во время волн алертов.

### 4. Схема PostgreSQL на сервис (общий кластер, отдельные схемы)

Каждый сервис владеет своей PostgreSQL-схемой (`ingestion`, `incident`, `scheduling`, `notification`). Все схемы живут в одном кластере для простоты локальной разработки.

**Обоснование**: изоляция на уровне схем закрепляет границы владения без операционных издержек нескольких кластеров в разработке.

**Рассмотренная альтернатива**: отдельные базы данных на сервис. Лучшая изоляция, но требует больше инфраструктуры при локальной разработке; можно мигрировать позже при необходимости.

### 5. Эскалация через периодический опрос + RabbitMQ

Сервис `escalation` периодически опрашивает собственную таблицу `incident_escalation_states` на наличие записей с истёкшим `escalate_at`. При срабатывании обновляет `current_tier` и `escalate_at` в `incident_escalation_states` (только своя схема), запрашивает scheduling service за текущим дежурным (`notify_schedule_id` уровня известен заранее) и публикует обогащённое событие `escalation.triggered` на exchange `escalations`, включая `oncall_user_id` и `oncall_username` прямо в payload.

Сервис `notification` потребляет только события с exchange `escalations` (`escalation.triggered`, `escalation.exhausted`). Так как on-call пользователь уже содержится в payload, notification не делает синхронных вызовов в scheduling за on-call данными. Для tenant notification config (Mattermost webhook URL) notification кэширует ответ scheduling in-process (LRU, TTL 5 минут) — промах кэша допустим, так как конфиг меняется редко. Событие `incident.created` notification не потребляет: первый `escalation.triggered` (tier 1) покрывает первичное оповещение.

**Обоснование**: интервал опроса (например, 30 сек) прост, аудируем и не требует распределённой системы таймеров/крон-планировщика. Приемлемая задержка для сценариев эскалации.

**Рассмотренная альтернатива**: keyspace-уведомления Redis по TTL-истечению. Нестабильно (события могут теряться при перезапуске Redis) и связывает логику эскалации с Redis.

### 6. Rate-limiting уведомлений через Redis token bucket

Сервис `notification` применяет per-contact rate-limiting через Redis-токен-бакет перед отправкой Email или Mattermost-сообщений.

**Обоснование**: предотвращает заваливание почтового ящика инженера уведомлениями во время крупных инцидентов.

### 7. REST API c паттерном шлюза

Каждый сервис предоставляет собственный REST API. Ingress (nginx) маршрутизирует `/api/ingestion/*`, `/api/incidents/*`, `/api/schedules/*`, `/api/escalations/*`, `/api/notifications/*` к соответствующему сервису.

**Обоснование**: позволяет независимо деплоить и тестировать сервисы. Позволяет избежать отдельного сервиса API-шлюза в первой версии.

### 8. Мультитенантность: row-level isolation, tenant_id в URL

Тенант = команда. Изоляция реализована на уровне строк БД: каждая доменная таблица содержит `tenant_id NOT NULL`. Все запросы фильтруются по `tenant_id`.

Модель данных тенантности (в схеме `scheduling`):

```sql
tenants              (id, slug, name, created_at)
users                (id, sub, preferred_username, name, email, last_seen_at)  -- кэш Keycloak
tenant_webhook_tokens (id, tenant_id, token_hash, source, created_at)
tenant_notification_config (tenant_id, mattermost_webhook_url, mattermost_channel, smtp_from)
```

Идентификация тенанта:

- **Ingestion webhooks**: `X-Webhook-Token: <secret>` → lookup в Redis (`HGET oncall:tokens:{SHA256(token)}` → `tenant_id`). Scheduling service поддерживает этот индекс: `HSET` при создании токена, `HDEL` при отзыве. Ingestion не читает `scheduling` схему напрямую и не делает HTTP-вызовов к scheduling.
- **Все остальные API**: tenant slug в URL: `/api/incidents/v1/{tenant}/incidents`

Пользователь может состоять в нескольких тенантах с разными ролями. Роль `admin` даёт право управлять участниками, токенами и конфигурацией уведомлений тенанта.

**Рассмотренная альтернатива**: schema-per-tenant. Более строгая изоляция, но усложняет миграции и избыточна для внутреннего инструмента с доверенными пользователями.

### 9. Аутентификация через Keycloak (OIDC) + локальный суперадмин

Все сервисы проверяют Bearer JWT, выданный Keycloak. Токен валидируется через JWKS-эндпоинт (env `KEYCLOAK_JWKS_URL`). Общий пакет `pkg/auth` реализует middleware и парсинг claims для всех пяти сервисов.

Маппинг claims:

- `sub` → `user_id` во всех доменных операциях
- `groups` (стандартный Keycloak claim) → список тенантов и ролей:
  - `/team-slug` → пользователь является участником тенанта `team-slug`
  - `/team-slug/admins` → пользователь является администратором тенанта `team-slug`

Локальный суперадмин: статичный API-ключ в Kubernetes Secret (env `ADMIN_API_KEY`), передаётся заголовком `X-Admin-Key`. Полностью обходит проверки Keycloak и tenant-контекста.

Middleware (`pkg/auth`) — stateless: только валидирует JWT через JWKS-эндпоинт, извлекает `sub`, `preferred_username`, `name`, `email` из claims и кладёт в Go context. Никакого IO из middleware.

Кэш пользователей: upsert `(sub, preferred_username, name, email, last_seen_at)` в таблицу `users` схемы `scheduling` выполняется только внутри сервиса `scheduling` при обращении пользователя к его эндпоинтам. Остальные сервисы хранят только `user_id` (значение `sub`). Display name для on-call эндпоинтов и уведомлений возвращается из scheduling service — сервисы `notification` и `escalation` и так вызывают его для определения дежурного.

Для GET `/tenants/{slug}/members` сервис обращается к Keycloak Admin API (client credentials: env `KEYCLOAK_ADMIN_CLIENT_ID` / `KEYCLOAK_ADMIN_CLIENT_SECRET`) и возвращает участников группы `team-slug`. Мутации участников — исключительно через Keycloak Admin UI.

**Обоснование**: Keycloak уже предполагается как инфраструктурный компонент; выносить управление пользователями и SSO в платформу — дублирование. Локальный суперадмин нужен для bootstrap (создание первого тенанта до того, как в системе появятся пользователи).

**Рассмотренная альтернатива**: локальная `tenant_memberships` таблица. Требует синхронизации с Keycloak и дополнительного API управления — лишняя сложность при наличии OIDC.

## Риски / компромиссы

- **RabbitMQ в локальной разработке** → Митигация: Bitnami RabbitMQ Helm chart для k3s; в prod — внешний сервис, credentials через Kubernetes Secret.
- **Коллизии fingerprint алертов** → Митигация: fingerprint — SHA-256 отсортированных пар ключ-значение лейблов + source; алгоритм задокументирован, чтобы интеграции могли предсказывать поведение.
- **Задержка опроса эскалаций (до 30 сек)** → Приемлемо для сценариев дежурств; задокументировать в SLA. Настраивается через env-переменную.
- **Keycloak как внешняя зависимость** → Митигация: Bitnami Keycloak Helm chart в k3s для локальной разработки; в prod — внешний экземпляр. При недоступности Keycloak JWT-валидация упадёт → сервисы вернут 401. Допустимо: Keycloak высокодоступен отдельно от платформы.
- **Рассинхронизация схем между сервисами** → Митигация: каждый сервис владеет своими файлами миграций (golang-migrate); CI запускает миграции в тестах.

## План развёртывания

1. Установить инфраструктуру (PostgreSQL, Redis, RabbitMQ, Keycloak) через Helm-чарты на k3s (для prod — RabbitMQ и Keycloak внешние, подключаются через Secret).
2. Задеплоить сервисы в порядке зависимостей: `ingestion` → `incident` → `scheduling` → `escalation` → `notification`.
3. Настроить маршруты Ingress.
4. Зарегистрировать вебхуки в Zabbix, Grafana и Alertmanager, указывающие на эндпоинты `ingestion`.
5. Откат: удалить Kubernetes Deployments и Helm-релизы; данные сохраняются в PVC.

## Открытые вопросы

- Стоит ли сервису `scheduling` предоставлять iCal-фид для импорта в календарь?
- Уведомления в Mattermost: только в канал тенанта (`mattermost_channel`); дежурный упоминается через @mention.
- Авто-закрытие алертов: да. Ingestion нормализует resolved-алерт → публикует в RabbitMQ (минуя дедупликацию, ключ удаляется) → incident service закрывает инцидент когда все привязанные алерты перешли в resolved.
