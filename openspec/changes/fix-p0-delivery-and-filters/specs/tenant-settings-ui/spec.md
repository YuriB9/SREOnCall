## MODIFIED Requirements

### Requirement: Форма конфигурации уведомлений

Система ДОЛЖНА (SHALL) отображать форму настройки уведомлений уровня тенанта, получаемых через `GET /api/schedules/v1/tenants/{tenant}/notification-config`. Форма ДОЛЖНА содержать поля для `mattermost_webhook_url`, `mattermost_channel` и `smtp_from`. Сохранение ДОЛЖНО выполнять `PUT /api/schedules/v1/tenants/{tenant}/notification-config`.

Поле `mattermost_webhook_url` НЕ ДОЛЖНО предзаполняться маскированным значением из GET-ответа; вместо этого рядом с пустым полем отображается текущий маскированный URL как справочная информация. Если пользователь не вводил новое значение URL, форма НЕ ДОЛЖНА включать поле `mattermost_webhook_url` в тело PUT-запроса — сохранённый вебхук не может быть затёрт неявно.

#### Scenario: Сохранение конфигурации уведомлений

- **WHEN** администратор редактирует поле `mattermost_channel` и сохраняет
- **THEN** система ДОЛЖНА выполнить PUT конфига и показать toast об успехе

#### Scenario: Сохранение без ввода URL не затирает вебхук

- **WHEN** администратор сохраняет форму, оставив поле `mattermost_webhook_url` пустым (новое значение не вводилось)
- **THEN** тело PUT-запроса НЕ содержит поле `mattermost_webhook_url`, и сохранённый вебхук тенанта остаётся без изменений

#### Scenario: Ввод нового URL обновляет вебхук

- **WHEN** администратор вводит непустой валидный URL и сохраняет
- **THEN** PUT-запрос содержит новое значение `mattermost_webhook_url`, и оно заменяет сохранённое

#### Scenario: Невалидный URL

- **WHEN** администратор вводит значение, не являющееся валидным URL, в поле `mattermost_webhook_url` и пытается сохранить
- **THEN** система ДОЛЖНА показать встроенную ошибку валидации и НЕ ДОЛЖНА отправлять запрос
