# Дизайн: Устранение P0-дефектов доставки уведомлений и фильтрации

## Context

Сверка `docs/openspec-code-verification.md` (раздел 2) подтвердила четыре дефекта:

1. **Межсервисные вызовы без авторизации.** Все маршруты scheduling закрыты JWT-middleware (`pkg/auth.Middleware`, 401 без Bearer); единственный обход — заголовок `X-Admin-Key` (`pkg/auth/auth.go:70-74`). Клиенты `escalation/internal/schedclient` и `notification/internal/schedclient` не отправляют ни того, ни другого. В k8s (`deploy/k8s/scheduling/configmap.yaml`) JWKS включён → escalation получает 401 при резолве дежурного, `triggerTier` публикует `escalation.triggered` с пустым `oncall_user_id`, notification пропускает уведомления пользователю. В docker-compose JWKS не задан, поэтому локально дефект не виден.
2. **Маскированный webhook URL у notification.** `GET /tenants/{slug}/notification-config` маскирует `mattermost_webhook_url` до `scheme://host` для защиты UI (`scheduling/internal/handler/handler.go:666-673`). Notification использует тот же эндпоинт и кэширует маскированное значение — Mattermost-отправки уходят на несуществующий URL.
3. **Мульти-статусный фильтр.** UI шлёт `?status=open,acknowledged` (`IncidentListPage.tsx:70`), бэкенд делает `i.status = $1` (`incident/internal/store/store.go:74-76`) → пустой результат при выборе двух и более статусов.
4. **Затирание webhook URL.** Форма настроек всегда шлёт `mattermost_webhook_url: webhookUrl` (`TenantSettingsPage.tsx:327`); при незаполненном поле PUT-upsert перезаписывает сохранённый URL пустой строкой.

## Goals / Non-Goals

**Goals:**

- Восстановить сквозную доставку уведомлений (escalation → scheduling → notification → email/Mattermost) в конфигурации с включённым JWKS.
- Дать notification доступ к полному (немаскированному) `mattermost_webhook_url`, не ослабляя маскирование для пользователей.
- Сделать фильтр `status` корректным для нескольких значений.
- Исключить неявное затирание webhook URL при сохранении формы настроек.

**Non-Goals:**

- Полноценный OAuth client-credentials для межсервисных вызовов (Keycloak service accounts) — follow-up; для P0 достаточно существующего механизма `X-Admin-Key`.
- Обогащение payload `escalation.triggered` (title/severity/status) — отдельное P1-изменение.
- Фильтры `source`/`label` для списка инцидентов — P1/P2.
- Изменение модели хранения конфигурации тенанта (перенос в БД notification).

## Decisions

### 1. Сервисная аутентификация через существующий `X-Admin-Key`

**Решение**: schedclient'ы escalation и notification получают параметр `adminKey` из конфигурации (`SCHEDULING_ADMIN_KEY` или переиспользуемый `ADMIN_KEY`) и добавляют заголовок `X-Admin-Key` в каждый запрос. В k8s ключ выдаётся через общий Secret сервисам scheduling (проверка), escalation и notification (использование).

**Обоснование**: механизм уже реализован в `pkg/auth.Middleware` и не требует изменений Keycloak; минимальный диф для P0.

**Альтернативы**:
- Keycloak client-credentials (правильнее, ротация, аудит) — отклонено для P0: требует настройки realm, клиентов и токен-кэша; зафиксировано как follow-up.
- Отдельная внутренняя сеть/порт без auth — отклонено: усложняет деплой и ослабляет защиту в общем неймспейсе.

### 2. Условное маскирование вместо отдельного internal-эндпоинта

**Решение**: `pkg/auth` помечает в контексте способ аутентификации (`service` при `X-Admin-Key`, `user` при JWT). `GetTenantNotificationConfig` маскирует `mattermost_webhook_url` только для `user`-запросов; `service`-запросы получают полный URL. Notification продолжает использовать тот же путь `/tenants/{slug}/notification-config`.

**Обоснование**: один эндпоинт — один контракт; не плодим дубликаты маршрутов; маскирование остаётся свойством представления, а не хранения.

**Альтернатива**: отдельный `GET /api/schedules/v1/internal/tenants/{slug}/notification-config` без маскирования. Отклонено: два маршрута с разными политиками легко рассинхронизировать; «internal»-путь всё равно требует той же проверки ключа.

### 3. Мульти-статус: разбор CSV в `IN`-фильтр

**Решение**: handler инцидентов разбирает `status` по запятой, валидирует каждое значение по enum (`open|acknowledged|resolved`, иначе 400), store строит `i.status = ANY($n)` (pgx позволяет передать срез). Формат фронтенда `join(',')` становится каноническим и фиксируется в спеке.

**Альтернатива**: повторяемый параметр `?status=a&status=b`. Отклонено: фронт уже шлёт CSV; CSV короче и совместим с текущим клиентом без изменений.

### 4. Семантика «пусто = не менять» для webhook URL — на сервере и на клиенте

**Решение** (двухслойная защита):
- **Сервер**: `PutTenantNotificationConfig` при пустом или отсутствующем `mattermost_webhook_url` сохраняет текущее значение из БД. Явная очистка — значением-сентинелом `"-"` либо отдельным DELETE (выбран сентинел не будет; см. Open Questions).
- **Клиент**: `TenantSettingsPage` не включает поле в тело PUT, если пользователь его не заполнял.

**Обоснование**: правка только клиента оставляет API-ловушку для любых других потребителей (curl, автоматизация); правка только сервера лишает возможности очистить URL. Дешевле закрыть оба слоя.

**Альтернатива**: PATCH-семантика для всего конфига. Отклонено для P0: меняет метод и контракт целиком; текущая проблема локальна для одного маскируемого поля.

## Risks / Trade-offs

- **Общий статический ключ — слабее OAuth**: утечка ключа открывает все scheduling-эндпоинты. → Митигация: ключ только в k8s Secret (не в configmap), follow-up на client-credentials; ключ не логируется.
- **Условное маскирование зависит от корректности признака аутентификации** — ошибка в `pkg/auth` раскроет URL пользователю. → Митигация: юнит-тест на оба режима; маскирование по умолчанию (unmasked только при явном `service`).
- **`X-Admin-Key` обходит и tenant-проверки** (`RequireTenantMember/Admin` пропускаются вместе с JWT) — сервисный ключ даёт cross-tenant доступ. → Принято осознанно: сервисы и так работают со всеми тенантами; фиксируется в спеке как свойство сервисного режима.
- **Кэш notification (`tenantcache`) может держать маскированное значение после деплоя** до истечения TTL. → Митигация: TTL мал; рестарт пода при деплое сбрасывает in-process кэш.
- **`status = ANY($n)`** меняет план запроса при множестве значений. → Незначимо: enum из трёх значений, индекс по `tenant_id, status` сохраняет селективность.

## Migration Plan

1. Бэкенд-изменения (pkg/auth, scheduling, incident) — обратносовместимы: старые клиенты без ключа продолжают получать маскированный конфиг и одиночный статус.
2. Деплой scheduling → escalation/notification с ключом в Secret (порядок не критичен: до выдачи ключа поведение не хуже текущего).
3. Фронтенд — обычный релиз статики; формат `status` CSV уже отправляется.
4. Откат: убрать ключ из окружения сервисов — система возвращается к текущему (дефектному, но известному) поведению; откат образов стандартный.

## Open Questions

- Нужна ли явная очистка `mattermost_webhook_url` администратором? Если да — добавить `DELETE /tenants/{slug}/notification-config/webhook-url` или допускать `null` (в отличие от пустой строки) как «очистить». Для P0 оставляем «пусто = не менять», очистку — через прямое обновление с непустым новым URL.
- Один общий `ADMIN_KEY` на все сервисы или отдельный ключ per-потребитель (escalation/notification)? Для P0 — один; per-потребитель имеет смысл вместе с переходом на client-credentials.
