# Дизайн: Синхронизация спецификаций с реализацией

## Context

После сверки (`docs/openspec-code-verification.md`, разделы 3-5) известно, какая сторона каждого расхождения реализована в коде. Это изменение — целиком документационное: 9 дельта-спек приводят требования к фактическому поведению. Кодовых правок нет, поэтому design фиксирует не «как реализовать», а **какая сторона каждого конфликта объявляется канонической и почему**.

## Goals / Non-Goals

**Goals:**

- Главные спеки в `openspec/specs/` после архивации соответствуют коду: каждый упомянутый эндпоинт существует с тем же путём, методом и полями.
- Недоопределённое поведение (severity, осиротевший resolved, каскад удаления, права на создание тенанта) зафиксировано как осознанное решение, а не случайность реализации.

**Non-Goals:**

- Любые изменения поведения кода (фильтр `source`, события эскалации в таймлайне, каскад удаления тенанта — если они понадобятся, это отдельные изменения с реализацией).
- Правка дельт активных изменений `fix-p0-delivery-and-filters` и `enrich-notifications-drop-zabbix` — их тексты принимаются как база.
- Введение общего реестра API-контрактов (отдельная процессная рекомендация).

## Decisions

### 1. Во всех конфликтах «UI-спека ↔ код» канон — код

Код фронтенда уже использует бэкенд-контракты (см. вердикты в `docs/openspec-code-verification.md`, раздел 4) и работает; переписывание кода под устаревшие UI-спеки дало бы регресс без пользы. Единственные места, где спека «выигрывала» у кода, уже вынесены в P0/P1.

### 2. Спорные фичи, отсутствующие с обеих сторон, удаляются из спек

Фильтр `source` (нет ни в API, ни в UI), фильтр `label` (нет в API), события эскалации в таймлайне инцидента (история эскалации живёт в escalation-сервисе и в UI не подмешивается), «текущий уровень эскалации» в виджете дежурного (нет источника данных) — **удаляются из требований**, а не остаются как «обещания». Обоснование: спека описывает контракт системы, а не backlog; для возврата фичи нужен ADDED-delta с реализацией.

### 3. Решения по пробелам — «как реализовано», с явной формулировкой

- **Severity инцидента** — от первого алерта, без пересчёта при группировке; лейблы копируются только при создании. Принято: предсказуемо и дёшево; пересчёт max-severity — возможное будущее изменение.
- **Resolved-алерт без открытого инцидента** — игнорируется без ошибки (идемпотентность повторных resolved).
- **Удаление тенанта** — каскад только в данных scheduling (расписания, переопределения, токены, конфигурация); инциденты, эскалации и журнал уведомлений остаются — фиксируется как документированное ограничение.
- **`POST /tenants`** — любой аутентифицированный пользователь; тенант становится рабочим только после создания соответствующей Keycloak-группы. Фиксируется вместе с предупреждением (создание группы — ручной шаг).

### 4. Редактирование политики эскалации — документируем замену

`PUT /policies/{id}` не существует; фронт делает POST новой → перенос default → DELETE старой. Спека фиксирует это поведение, включая известное ограничение: операция не атомарна, при сбое между шагами могут остаться две политики. Альтернатива — потребовать PUT в бэкенде — отклонена: это изменение поведения, выходящее за рамки документационной синхронизации (кандидат в отдельное изменение).

### 5. Порядок применения: после P0 и P1

Требования «Список и фильтрация инцидентов» (P0: мульти-статус) и «Срабатывание эскалации по таймауту» (P0: сервисная аутентификация; P1: обогащённый payload) уже модифицированы активными изменениями. Дельты этого изменения написаны поверх их текстов; архивация в порядке P0 → P1 → P2 обязательна, иначе при sync потеряются правки предыдущих изменений.

## Risks / Trade-offs

- **Три активных изменения трогают два общих требования** → риск конфликтов при архивации. → Митигация: явная зависимость порядка (Decision 5); перед архивацией P2 — diff дельты против актуального текста главной спеки.
- **«Канон — код» может узаконить случайные решения** (например, 503 vs 500). → Принято осознанно: каждое решение перечислено в дельте и прошло ревью этим изменением; всё спорное уже вынесено в P0/P1.
- **Удалённые из спек фичи (source-фильтр, таймлайн эскалаций) могут потеряться как идеи.** → Митигация: перечислены в Open Questions как кандидаты в backlog.

## Migration Plan

Кодовых изменений нет. Порядок: реализация и архивация `fix-p0-delivery-and-filters` → `enrich-notifications-drop-zabbix` → выполнение проверочных задач этого изменения → архивация (sync дельт в главные спеки). Откат — не архивировать изменение.

## Open Questions

- Кандидаты в backlog после удаления из спек: фильтр инцидентов по источнику, объединённый таймлайн (история инцидента + события эскалации), агрегированный виджет «кто дежурит по всем расписаниям», `PUT /policies/{id}` для атомарного редактирования, пересчёт severity по максимуму алертов группы.
- Стоит ли одновременно с архивацией обновить Purpose-секции спек (упоминания Zabbix, «эндпоинт API» в ссылках) — да, как часть sync-задач.
