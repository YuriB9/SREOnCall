## MODIFIED Requirements

### Requirement: Создание и управление тенантами

Сервис scheduling ДОЛЖЕН (SHALL) предоставлять CRUD-эндпоинты для тенантов (команд) по адресу `/api/schedules/v1/tenants`.

Тенант ДОЛЖЕН включать: `slug` (уникальный URL-safe идентификатор), `name` (отображаемое название).

Создание тенанта (`POST /api/schedules/v1/tenants`) доступно любому аутентифицированному пользователю (или сервисному ключу); тенантной роли для создания не требуется. Тенант становится работоспособным только после создания соответствующей Keycloak-группы `/{slug}` (ручной шаг администратора Keycloak) — до этого ни один пользователь не имеет к нему доступа.

Удаление тенанта каскадно удаляет только данные scheduling-сервиса: расписания, переопределения, вебхук-токены и конфигурацию уведомлений. Данные других сервисов (инциденты, состояния эскалации, журнал уведомлений) НЕ удаляются — это документированное ограничение; межсервисный каскад потребует отдельного изменения.

#### Scenario: Создание тенанта

- **WHEN** аутентифицированный пользователь выполняет POST на `/api/schedules/v1/tenants` с валидными `slug` и `name`
- **THEN** тенант сохраняется и возвращается с HTTP 201 и сгенерированным `id`

#### Scenario: Slug должен быть уникальным

- **WHEN** выполняется POST с `slug`, уже занятым другим тенантом
- **THEN** сервис возвращает HTTP 409

#### Scenario: Обновление тенанта

- **WHEN** выполняется PATCH на `/api/schedules/v1/tenants/{slug}` администратором тенанта
- **THEN** поля `name` обновляются; `slug` изменить нельзя после создания

#### Scenario: Удаление тенанта

- **WHEN** выполняется DELETE на `/api/schedules/v1/tenants/{slug}` администратором тенанта
- **THEN** тенант и связанные данные scheduling-сервиса (расписания, переопределения, токены, конфигурация уведомлений) удаляются; инциденты и журналы других сервисов сохраняются
