## Context

Вебхук-токены ingestion резолвятся через Redis-индекс `oncall:tokens:{hash} → tenant_id(slug)`. Индекс наполняет только scheduling в обработчиках создания/отзыва токена (`tokenindex.Set`/`Del`). Источник истины — Postgres-таблица `scheduling.tenant_webhook_tokens` (хранит `id, tenant_id, token_hash, source, created_at`).

Redis в docker-compose запущен без персистентности (`--save "" --appendonly no`), а на старте scheduling нет шага восстановления индекса из БД. Поэтому после перезапуска/сброса Redis действующие токены пропадают из индекса: ingestion отвечает HTTP 401, хотя строка в Postgres сохранилась. Единственный текущий обходной путь — пересоздать токен.

В `cmd/server/main.go` уже есть точка, где доступны и `store` (`st`), и опциональный Redis-клиент (`rdb`), и создаётся `tidx = tokenindex.New(rdb)` — естественное место для регидрации.

## Goals / Non-Goals

**Goals:**
- На старте scheduling восстанавливать индекс токенов в Redis из Postgres.
- Идемпотентность и отказоустойчивость: повторный старт безопасен; недоступность Redis или сбой отдельных записей не прерывают запуск.
- Минимальные изменения, без правок схемы БД и без новых внешних зависимостей.

**Non-Goals:**
- Включение персистентности Redis (отдельное инфраструктурное изменение, не входит сюда).
- Удаление осиротевших ключей Redis, отсутствующих в Postgres (reconcile-уборка).
- Периодическая фоновая синхронизация — только однократная регидрация при старте.
- Изменение HTTP/событийных контрактов.

## Decisions

**1. Регидрация при старте, синхронно, до приёма трафика.**
Выполняется один раз в `main.go` сразу после `tidx = tokenindex.New(rdb)` и до `ListenAndServe`. Так к моменту приёма запросов индекс уже консистентен. Альтернатива — ленивый fallback в ingestion (читать Postgres при промахе Redis) — отвергнута: добавляет связанность ingestion↔БД scheduling и усложняет горячий путь приёма алертов.

**2. Источник данных — новый метод стораджа `ListWebhookTokenHashes`.**
Возвращает пары `(token_hash, tenant_id)` по всем строкам `scheduling.tenant_webhook_tokens`. Отдельный узкий метод (а не переиспользование `ListWebhookTokens`, отдающего доменные объекты с маскированием) — чтобы не тащить лишние поля и не светить чувствительные данные.

**3. Массовая загрузка в `tokenindex` — новый метод `SetMany`.**
Загружает срез пар одним проходом (предпочтительно через Redis pipeline для снижения RTT). Семантика на каждый ключ та же, что у `Set` (`HSET ... tenant_id ...`), поэтому существующее значение перезаписывается тем же — отсюда идемпотентность.

**4. Деградация без падения.**
Если `rdb == nil` (Redis не поднялся) — регидрация пропускается, как и сейчас отключён сам индекс. Ошибка чтения из Postgres или записи в Redis логируется как warning и не вызывает `os.Exit` — сервис стартует, а отсутствующие токены можно пересоздать (текущее поведение как нижняя граница).

## Risks / Trade-offs

- **Старые/отозванные токены в БД** → отзыв уже удаляет строку из Postgres (`DELETE`), поэтому в таблице остаются только действующие токены; регидрация не воскрешает отозванные.
- **Большое число токенов замедляет старт** → объём мал (токены на тенант×источник), загрузка через pipeline; при необходимости можно вынести в фоновую горутину, но для текущего масштаба синхронно проще и безопаснее.
- **Гонка: токен создан между чтением БД и стартом приёма запросов** → обработчик создания токена всё равно делает `Set` в Redis в реальном времени, поэтому такой токен попадёт в индекс независимо от регидрации.
- **Осиротевшие ключи Redis (есть в Redis, нет в БД)** → вне scope; отзыв чистит их штатно через `HDEL`.

## Migration Plan

- Изменения только в коде scheduling (`main.go`, `internal/store`, `internal/tokenindex`). Миграции БД не требуются.
- Деплой обычный, обратной совместимости: при старте новый код просто дополнительно наполняет существующий ключевой формат Redis.
- Откат — вернуть предыдущий бинарь; данные в Postgres/Redis не изменяются деструктивно.
- Проверка: остановить/сбросить Redis, поднять scheduling, убедиться, что вебхук с ранее выданным токеном резолвится (HTTP 200, а не 401).

## Open Questions

- Нужна ли регидрация синхронной или допустимо в фоне на старте? Для текущего масштаба выбран синхронный вариант; пересмотреть при росте числа токенов.
