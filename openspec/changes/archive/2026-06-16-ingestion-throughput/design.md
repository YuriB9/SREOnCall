## Context

Путь приёма вебхука в ingestion обрабатывает алерты строго последовательно:
`HandleAlertmanager`/`HandleGrafana` → `processAlerts` → цикл `processOne`, где каждый
алерт делает `Redis SETNX → Postgres INSERT raw_alerts → AMQP publish` поочерёдно
(`services/ingestion/internal/handler/handler.go:44-86`). Тело вебхука Alertmanager
штатно несёт десятки алертов, поэтому стоимость = `N × (RTT Redis + RTT PG + publish)`.

Сопутствующие находки того же класса: построчные INSERT'ы лейблов в incident
(`mergeLabels`, N round-trip'ов) и двойной `json.Marshal` структуры `Alert` (для
JSONB-колонки и для конверта).

Опорные точки уже готовы предыдущими чейнджами: переиспользуемый канал публикации
(CH14, `pkg/amqp.Publisher` держит долгоживущий канал), вынесенный store-слой ingestion
(CH09), доменные метрики приёма (CH11). Воркер-пул консьюмеров (P2(б)/C8) сделан в CH07.

## Goals / Non-Goals

**Goals:**
- Свести число последовательных round-trip'ов на тело вебхука к ~3 групповым операциям
  (1 пайплайн Redis + 1 батч PG + публикации) вместо `3×N` последовательных.
- Один round-trip на upsert лейблов инцидента (P3).
- Одна сериализация `Alert` на алерт (P5).
- Сохранить наблюдаемое поведение: коды ответов, формат событий, семантику дедупа и
  порядок, откат дедуп-ключа при ошибке публикации.
- Подтвердить выигрыш benchstat'ом (требование аудита).

**Non-Goals:**
- Асинхронный приём (ответ `202` + очередь приёма) — меняет гарантию доставки.
- Воркер-пул консьюмеров (P2(б)) — уже в CH07.
- Publisher confirms, пул каналов — вне P-находок (бэклог CH14).
- Изменение wire-формата событий или схемы БД.

## Decisions

### Решение 1 — групповой `processAlerts` вместо поалертного цикла
Тело вебхука обрабатывается тремя фазами на весь батч:
1. **Сериализация (P5):** один `json.Marshal(alert)` на алерт → `json.RawMessage`,
   используется и для `raw_alerts.payload`, и для конверта публикации.
2. **Дедуп-пайплайн (P2):** `Deduplicator.Classify(ctx, alerts)` собирает все `SETNX`
   (firing) и `DEL` (resolved) в один Redis-пайплайн (один round-trip), возвращает
   `[]bool` «подавить/нет» параллельно срезу алертов. Метрики `hits/misses` сохранены.
3. **Батч-INSERT (P2):** `Store.SaveRawAlerts(ctx, []RawAlert)` через `pgx.Batch` —
   один round-trip на тело.
4. **Публикация (P2):** неподавленные алерты публикуются на переиспользуемом канале
   (CH14). При ошибке публикации дедуп-ключ конкретного алерта откатывается
   (`Clear`), возвращается ошибка → `503`, как и сейчас.

_Альтернатива (отклонена):_ `COPY` вместо `pgx.Batch` для `raw_alerts`. `COPY` быстрее
на больших объёмах, но не поддерживает `RETURNING`/`ON CONFLICT` и усложняет код;
`pgx.Batch` достаточно для десятков строк типового вебхука и проще. При профиле с
тысячами строк — пересмотреть.

### Решение 2 — расширение `dedup.Cache`, а не замена
К `Cache` добавляется `Apply(ctx, setKeys, val, ttl, delKeys) ([]bool, error)`
(пайплайн SETNX+DEL в один round-trip). Старые `SetNX`/`Del` и
`Deduplicator.IsDuplicate`/`Clear` **оставлены** — на них опираются регресс-тесты и
откат при ошибке публикации. Изменение аддитивное → ниже риск.

_Альтернатива (отклонена):_ заменить `SetNX`/`Del` на единственный батч-примитив —
потребовало бы переписать все тесты и убрать точечный `Clear` (нужен для rollback).

### Решение 3 — multi-row upsert лейблов через `unnest` (P3)
`mergeLabels` строит два среза (`keys`, `values`) из map и выполняет
`INSERT ... SELECT FROM unnest($2::text[], $3::text[]) ON CONFLICT ... DO UPDATE`.
Один общий приватный helper покрывает и `MergeLabels`, и `CreateIncidentTx`
(оба идут через `mergeLabels(ctx, q dbConn, ...)`), поэтому правка локальна и
сохраняет работу внутри транзакции CH08.

### Решение 4 — P5 без правок `pkg/amqp`
`Wrap(eventType, tenantID, payload any)` маршалит `payload`. Если передать
`json.RawMessage`, `json.Marshal` возвращает байты как есть (без рефлексии структуры).
Поэтому достаточно добавить `publisher.PublishAlertPayload(ctx, tenantID, json.RawMessage)`,
который зовёт `Wrap` с готовым payload — `pkg/amqp` не трогаем.

## Risks / Trade-offs

- **[Риск] Сломать семантику дедупа при батчинге** (два алерта с одним fingerprint в
  одном теле) → в Redis-пайплайне `SETNX` исполняются по порядку: первый `set=true`,
  второй `set=false`. Поведение идентично поалертному. Покрыть тестом батч-дедупа.
- **[Риск] Частичная ошибка публикации в середине батча** → откатываем дедуп-ключ только
  упавшего алерта и возвращаем `503`. Успевшие опубликоваться сохраняют дедуп-ключ →
  при ретрае отправителя они подавляются (не дублируются), переобрабатывается только
  упавший. Семантика не хуже текущей (which также прерывался на первой ошибке).
- **[Риск] `raw_alerts` — append-only аудит; ретрай вебхука вставит строки повторно** →
  это уже так в текущей реализации (аудит-лог, не уникальный ключ). Не регрессия.
- **[Trade-off] Батч PG/Redis повышает пиковую память на тело** (срезы payload'ов).
  Тело ограничено `MaxBytesReader` (4 МБ), N ограничен → приемлемо.
- **[Риск] Метрики приёма (`ingestion_alerts_received_total`, dedup hits/misses) должны
  остаться поалертными** → инкрементируются в цикле подготовки/классификации, не
  сворачиваются. Проверить в тестах.

## Migration Plan

Деплой без миграций БД и без смены контракта событий — катится как обычный rolling
update ingestion и incident. Откат — простой откат образа: формат `raw_alerts`,
`incident_labels` и конверта `alert.received` не менялся, старые и новые поды
взаимозаменяемы. Benchstat до/после прикладывается в коммит (`benchstat.txt`).
