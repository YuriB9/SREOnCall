# Дизайн: Фронтенд SRE OnCall

## Контекст

Платформа sre-oncall-platform предоставляет полноценный REST API через пять Go-микросервисов за Kubernetes Ingress. Все пути API привязаны к тенанту (`/api/{service}/v1/{tenant}/...`). Аутентификация — Keycloak OIDC: каждый запрос должен содержать Bearer JWT; роль и членство в тенанте закодированы в claim `groups`. UI отсутствует.

Фронтенд должен интегрироваться с этой API-поверхностью без добавления новых эндпоинтов бэкенда для v1. Это SPA, раздаваемое как статический бандл — без server-side rendering, без BFF-прокси. Пользователи — SRE-инженеры, взаимодействующие с платформой преимущественно во время инцидентов, часто ночью и в стрессовых условиях.

## Цели / не-цели

**Цели:**

- Создать веб-интерфейс с клавиатурным управлением и тёмной темой по умолчанию для управления инцидентами, расписаниями дежурств, конфигурацией политик эскалации и администрирования тенанта.
- Реализовать Keycloak Authorization Code Flow + PKCE на стороне клиента через `oidc-client-ts`.
- Зеркалировать параметр `{tenant}` бэкенда как верхний сегмент маршрута, чтобы глубокие ссылки и история браузера оставались осмысленными.
- Принудительно проверять ролевой доступ на стороне клиента (гард маршрута читает claim `groups` из JWT); реальная граница безопасности — на сервере.
- Поддерживать быстрое подтверждение инцидентов из мобильных браузеров (адаптивная вёрстка, крупные сенсорные цели).

**Не-цели:**

- Server-side rendering или Backend-for-Frontend прокси.
- Нативные мобильные приложения или push-уведомления на мобильные устройства.
- Обновления в реальном времени через WebSocket/SSE (в бэкенде v1 не предусмотрены; поллинг достаточен).
- Собственная библиотека компонентов — Shadcn/ui + Radix Primitives используются как есть.
- Создание правил алертинга (это делается в Prometheus/Grafana/Zabbix, а не в данной платформе).

## Решения

### 1. React + Vite (не Next.js)

**Решение**: SPA на базе Vite с React 18 и TypeScript.

**Обоснование**: Приложение — внутренний инструмент, SEO не требуется. Next.js App Router добавляет сложность SSR/RSC и требует Node-сервер, который нужно контейнеризировать и поддерживать. Vite производит статический бандл, который может раздавать любой существующий Ingress или nginx-сайдкар; отдельный Kubernetes Deployment для фронтенда не нужен. Файловый роутинг обеспечивает `react-router-dom` v6 с вручную описанным деревом маршрутов, точно зеркалирующим структуру URL бэкенда.

**Рассмотренная альтернатива**: Next.js с App Router. Отклонён, поскольку ценность BFF/middleware низкая (в v1 не нужно скрывать эндпоинты Keycloak или агрегировать ответы микросервисов), а операционная стоимость запуска Node-сервера существенна.

### 2. TanStack Query для загрузки данных и поллинга

**Решение**: Все вызовы API бэкенда проходят через хуки TanStack Query (React Query v5). Список и детали инцидентов используют `refetchInterval: 12_000` (12 секунд) как разумный баланс между актуальностью и нагрузкой на API.

**Обоснование**: TanStack Query обеспечивает кэширование, фоновое обновление, оптимистичные обновления для PATCH/Acknowledge/Resolve и автоматические повторные попытки при сбое сети — всё это критично для инструмента дежурства. Интервал поллинга 12 секунд означает, что новый инцидент появится в UI в течение ~12 секунд после создания на бэкенде без WebSockets.

**Рассмотренная альтернатива**: SWR. Схожий набор функций; TanStack Query выбран за более эргономичный API мутаций и лучший вывод типов TypeScript из коробки.

### 3. Keycloak OIDC через oidc-client-ts (PKCE flow)

**Решение**: `oidc-client-ts` в режиме silent-renew. Токен хранится в `sessionStorage` (не `localStorage`), чтобы не переживал перезапуск браузера. Обновление через скрытый iframe (`check-session` iframe Keycloak).

**Обоснование**: Authorization Code Flow + PKCE — правильный flow для публичных клиентов (SPA). `oidc-client-ts` — поддерживаемый преемник `oidc-client`, полностью управляющий жизненным циклом OIDC, включая тихое обновление токена без бэкенда.

**Парсинг токена**: После входа `id_token` декодируется (без проверки подписи — бэкенд уже верифицирует `access_token`). Массив `groups` извлекается для построения локальной карты `{ tenantSlug → role }`, используемой гардами маршрутов и хуком `usePermissions()`.

**Рассмотренная альтернатива**: NextAuth.js. Требует бэкенд; отклонён (см. Решение 1).

### 4. Тенантный роутинг как верхний сегмент

**Решение**: Дерево маршрутов:

```text
/                         → редирект на /select-team или первый тенант
/select-team              → список тенантов из JWT groups
/[tenant]/incidents       → список инцидентов + панель деталей
/[tenant]/incidents/:id   → прямая глубокая ссылка на инцидент (тот же вид, панель открыта)
/[tenant]/schedules       → расписание дежурств
/[tenant]/escalations     → политики эскалации
/[tenant]/settings        → настройки тенанта (только для администраторов)
/profile                  → личные настройки пользователя
```

Middleware гардов маршрутов: `<TenantGuard>` оборачивает все маршруты `/[tenant]/*`. Читает claim `groups` и перенаправляет на страницу 403, если пользователь не является участником запрошенного тенанта. `<AdminGuard>` оборачивает `/[tenant]/settings` и требует группу `/team-slug/admins`.

**Обоснование**: Зеркалирует структуру путей бэкенда; глубокие ссылки работают корректно; кнопки «назад/вперёд» браузера навигируют интуитивно. Keycloak перенаправляет обратно на исходную глубокую ссылку после входа через параметр `state`.

### 5. Детали инцидента в разделённом виде (не в модальном окне)

**Решение**: Клик по инциденту открывает правую панель/drawer без смены маршрута. Полный вид деталей по адресу `/[tenant]/incidents/:id` предназначен только для прямого доступа по глубокой ссылке и рендерит тот же компонент панели на всю ширину.

**Обоснование**: Разделённый вид сохраняет контекст списка инцидентов, снижая когнитивную нагрузку во время аварий. URL при открытии панели не меняется, чтобы кнопка «назад» не сбрасывала состояние фильтров, но ID выбранного инцидента синхронизируется в search-параметр URL (`?incident=<id>`) для возможности поделиться ссылкой.

### 6. Система горячих клавиш

**Решение**: Глобальный обработчик `keydown` (через хук `useKeyMap`) активен, когда нет фокуса в input/textarea. Биндинги по умолчанию:

- `A` → Подтвердить (Acknowledge) выбранный/открытый инцидент
- `R` → Закрыть (Resolve) выбранный/открытый инцидент
- `/` → Фокус на поле поиска/фильтра
- `Esc` → Закрыть панель / отменить модальное окно
- `J/K` → Навигация по списку инцидентов вниз/вверх

**Обоснование**: Дежурные инженеры работают в условиях жёстких временных ограничений; снижение зависимости от мыши при подтверждении инцидента — основная UX-цель.

### 7. Одноразовый показ webhook-токена

**Решение**: Когда администратор генерирует новый webhook-токен, бэкенд возвращает plaintext-токен ровно один раз. Фронтенд открывает модальное окно с токеном в поле с возможностью копирования. Окно требует обязательного чекбокса подтверждения («Я сохранил этот токен») перед закрытием. После закрытия plaintext-токен удаляется из состояния клиента.

**Обоснование**: Бэкенд хранит только SHA-256 хэш; механизма восстановления нет. Явное подтверждение намерения «сохрани сейчас» предотвращает случайную потерю токена. Этот паттерн (показ один раз, подтверждение перед закрытием) используется GitHub PAT и аналогичными системами.

### 8. Звуковые уведомления

**Решение**: Опциональный звуковой сигнал браузера при появлении новых инцидентов в статусе `open`. Реализован через Web Audio API (синтезированный звуковой сигнал — внешний аудиофайл не требуется). Переключатель сохраняется в `localStorage`. Сигнал воспроизводится, когда ответ поллинга содержит ID инцидентов, отсутствовавших в предыдущем ответе, со статусом `open`.

**Обоснование**: Не требует сетевого запроса для аудиофайла; Web Audio API поддерживается повсеместно; сохранение в `localStorage` означает, что предпочтение переживает перезагрузку страницы.

### 9. Стилизация: Tailwind CSS + Shadcn/ui

**Решение**: Tailwind CSS для утилитарных классов; Shadcn/ui (Radix Primitives) для доступных компонентов (Dialog, DropdownMenu, Table, Tabs и т.д.). Тёмная тема по умолчанию через Tailwind `darkMode: 'class'` с принудительным классом `dark` на `<html>`; пользователь может переключиться на светлую.

**Обоснование**: Компоненты Shadcn/ui копируются в репозиторий, а не импортируются из пакета — их можно свободно кастомизировать. Radix Primitives предоставляют клавиатурную доступность из коробки (перехват фокуса, ARIA-роли), что важно для инструмента с клавиатурным управлением.

## Риски / компромиссы

- **Поллинг против реального времени**: Интервал 12 секунд означает задержку до 12 секунд до появления нового инцидента. Допустимо для v1, так как WebSocket на бэкенде не предусмотрен. → Митигация: интервал — настраиваемая константа; легко уменьшить или заменить на SSE/WebSocket, когда бэкенд это поддержит.
- **Проверка ролей только на клиенте**: Гарды маршрутов — UX-удобство, а не граница безопасности. Реальный контроль доступа обеспечивает бэкенд. Пользователь, обходящий клиентский гард, получит 403/401 от API. → Митигация не требуется, кроме явной документации этого факта.
- **Расхождение часов JWT (clock skew)**: Тихое обновление использует iframe `check-session` Keycloak; если Keycloak недоступен, `oidc-client-ts` не сможет обновить токен и перенаправит на страницу входа. Во время активного инцидента это деструктивно. → Митигация: настроить щедрое значение `accessTokenExpiringNotificationTimeInSeconds` (120 с) и показывать постоянный баннер «сессия истекает» до выхода.
- **Сложность Gantt-расписания**: Помесячный вид требует вычисления окон смен на клиенте из объектов расписания бэкенда. Эта логика должна обрабатывать переходы летнего/зимнего времени (DST) и многочасовые команды. → Митигация: использовать `date-fns-tz` для всей арифметики часовых поясов; писать юнит-тесты для вычисления окон смен.
- **Мобильная вёрстка**: Полный вид Gantt неудобен на экране 375px. → Митигация: на мобильных (<640px breakpoint) Gantt скрыт и заменён виджетом «Кто дежурит сейчас» + списком предстоящих смен на 7 дней. Действия «Подтвердить» и «Закрыть» остаются доступны.

## Открытые вопросы

- Нужна ли поддержка нескольких одновременных контекстов тенанта (например, пользователь дежурит сразу в двух командах)? Для v1 предполагается единый активный контекст тенанта, выбираемый через `/select-team`.
- Экспорт iCal-фида из расписаний? (Открытый вопрос в дизайне бэкенда.) Если бэкенд добавит это, фронтенду нужна только ссылка для скачивания — дополнительной работы по дизайну не требуется.
- Стратегия обработки ошибок при сбое поллинга: тихая повторная попытка или toast-уведомление? Склоняемся к небольшому индикатору статуса в шапке («Обновлено: 43 с назад — повторная попытка...»), а не к навязчивым тостам.
