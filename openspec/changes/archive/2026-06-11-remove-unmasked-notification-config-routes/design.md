## Context

Конфигурация уведомлений тенанта обслуживается двумя наборами маршрутов scheduling-сервиса:

| Маршрут | Защита GET | Поведение PUT с пустым URL | Кто использует |
|---|---|---|---|
| `/api/schedules/v1/tenants/{slug}/notification-config` (канонический, по спеке) | маскирование для JWT, полный URL только для `X-Admin-Key` | сохраняет текущее значение | frontend (`tenantSettings.ts`), notification (`schedclient`) |
| `/api/schedules/v1/{tenant}/notification-config` (параллельный, вне спеки) | **без маскирования**, доступен любому участнику (`RequireTenantMember`) | **затирает** сохранённый URL | только интеграционный тест scheduling |

Параллельная пара (`GetNotificationConfig`/`PutNotificationConfig` в `handler.go`, маршруты в `main.go:146-147`) появилась исторически и не участвует в решении проблемы «фронтенд затирает вебхук маской» — эта проблема закрыта на каноническом эндпоинте (серверное «пустое значение = оставить» + фронтенд не предзаполняет поле и не шлёт его без ввода).

## Goals / Non-Goals

**Goals:**
- Немаскированный `mattermost_webhook_url` недоступен ни по одному маршруту без сервисного ключа.
- Единственный HTTP-интерфейс конфигурации уведомлений — канонический; одна точка правды для маскирования и семантики PUT.
- Интеграционное покрытие переезжает на канонические маршруты и проверяет маскирование/сохранение вебхука.

**Non-Goals:**
- Изменение поведения канонических эндпоинтов (маскирование, сохранение при пустом URL) — остаётся как есть.
- Механизм явной очистки сохранённого вебхука (`null`/DELETE) — осознанно вынесен в отдельное изменение, если понадобится продуктово.
- Изменения фронтенда, notification, escalation — не требуются.

## Decisions

**1. Полное удаление, а не выравнивание поведения (вариант A, а не B).**
Внешних потребителей параллельных маршрутов нет (проверено grep'ом по `services/`, `frontend/src/`, `tests/`, `scripts/`); внутри репозитория — один интеграционный тест. Держать два URL одного ресурса означало бы дублировать маскирование и семантику PUT в двух обработчиках и тестировать обоих. Удаление дешевле и устраняет класс ошибок целиком.

*Альтернатива (отклонена):* делегировать параллельные маршруты каноническим обработчикам — оправдано только при наличии внешних клиентов старого URL, которых нет.

**2. Удалять и обработчики, не только маршруты.**
Неподключённые обработчики `GetNotificationConfig`/`PutNotificationConfig` — мёртвый код с опасной семантикой (немаскированный вывод, затирание); оставлять их «на всякий случай» — приглашение к повторному подключению.

**3. Тест переводится, а не удаляется, и расширяется.**
Существующий тест проверял запись/чтение конфигурации. На канонических маршрутах он дополнительно проверяет три сценария спеки: маска для JWT-запроса, полный URL для `X-Admin-Key`, PUT с пустым URL не затирает сохранённый вебхук. Это закрывает регресс на будущее.

## Risks / Trade-offs

- [Гипотетический внешний скрипт использует старый URL] → маршрут начнёт отвечать 404/405; миграция тривиальна — заменить путь на `/tenants/{slug}/notification-config` (для чтения полного URL потребуется `X-Admin-Key`, что и является целью). В репозитории и docs/env таких потребителей нет.
- [Канонический PUT не позволяет очистить вебхук] → существующее ограничение, не вводится этим изменением; при необходимости решается отдельным изменением (явный `null` или DELETE).

## Migration Plan

1. Удалить маршруты и обработчики, перевести тест.
2. Обычный деплой scheduling; откат — обычный откат образа. Данные и схема БД не затрагиваются.
