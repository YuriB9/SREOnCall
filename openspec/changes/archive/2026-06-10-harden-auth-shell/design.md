# Дизайн: Устранение замечаний оболочки аутентификации

## Context

Оболочка аутентификации фронтенда (`frontend/src/auth/*`, `frontend/src/pages/CallbackPage.tsx`) реализована на `oidc-client-ts` с Authorization Code Flow + PKCE против Keycloak. Бэкенд (`pkg/auth/auth.go`) валидирует `access_token` по JWKS и извлекает claims `sub`, `preferred_username`, `name`, `email`, `groups`.

Текущее состояние и выявленные расхождения (аудит секции 2 изменения `sre-oncall-frontend`):

- **Источник `groups`**: фронт декодирует `groups` из `id_token` (`user.profile` в `usePermissions.ts`), бэкенд — из `access_token` (`auth.go`). Оба токена должны содержать groups-маппер; запрошенный scope — `'openid profile email'`, отдельного groups-scope нет.
- **Трактовка подгрупп**: бэкендовый `IsMember` считает участником любого с префиксом `/{tenant}/...`; фронтовый `parseGroups` регистрирует тенант только по `/{tenant}` или `/{tenant}/admins`.
- **Недоступность Keycloak**: `RequireAuth → signIn() → signinRedirect` и `CallbackPage` (на ошибке делает `navigate('/')`) не имеют graceful-обработки — возможен цикл редиректов и непонятная пользователю ошибка.
- **Баннер сессии**: `SessionBanner` реагирует на любой `silentRenewError`, включая транзиентный, и через 30 сек принудительно уводит на вход.

Граница безопасности — на бэкенде; изменения здесь касаются корректности клиентской авторизации (UX/видимость) и устойчивости к сбоям Keycloak. Бэкенд не меняется.

## Goals / Non-Goals

**Goals:**

- Согласовать клиентскую трактовку членства в тенанте с бэкендом (префиксные подгруппы).
- Сделать поведение при недоступности Keycloak предсказуемым: понятный экран вместо ошибки браузера, отсутствие цикла редиректов.
- Сделать баннер истечения сессии устойчивым к транзиентным сбоям тихого обновления.
- Дать пользователю/оператору диагностический сигнал при вероятной неверной настройке Keycloak-маппера (пустые permissions при непустом `groups`).
- Зафиксировать требование к конфигурации Keycloak (groups-маппер в обоих токенах) в документации.

**Non-Goals:**

- Изменение бэкенда (`pkg/auth/auth.go`) или контракта токенов.
- Введение проверки подписи `id_token` на клиенте (граница безопасности остаётся на бэкенде).
- Переход на другой OIDC-клиент или storage-стратегию.
- Создание тестовой инфраструктуры (vitest/CI) — она в backlog (бывшая секция 10); тесты добавляются, когда инфраструктура появится.

## Decisions

### 1. `parseGroups`: префиксная трактовка членства

**Решение**: переписать `parseGroups` так, чтобы любая группа `/{tenant}` или `/{tenant}/<любой-сегмент>` означала членство (`member`), а `/{tenant}/admins` повышала роль до `admin`. Тенант извлекается как первый сегмент пути; admin определяется наличием сегмента `admins` на втором уровне.

**Обоснование**: зеркалирует серверный `IsMember` (`g == "/"+tenant || strings.HasPrefix(g, "/"+tenant+"/")`) и `IsAdmin` (`g == "/"+tenant+"/admins"`). Устраняет ложное скрытие тенанта на фронте.

**Альтернатива**: оставить как есть и задокументировать ограничение «используйте только `/team` и `/team/admins`». Отклонено: хрупко, расходится с сервером, тихо ломает доступ при нестандартной иерархии групп.

### 2. Graceful-экран при ошибке `CallbackPage`

**Решение**: при reject `signinRedirectCallback()` вместо `navigate('/')` рендерить компонент-заглушку с заголовком «Не удалось завершить вход», кратким объяснением и кнопкой «Повторить вход» (вызывает `signIn()`). Никакого автоматического редиректа.

**Обоснование**: автоматический `navigate('/')` повторно триггерит `RequireAuth → signIn`, создавая цикл при недоступном/неверно настроенном Keycloak. Явная кнопка разрывает цикл и даёт пользователю контроль.

**Альтернатива**: ретрай с backoff. Отклонено для v1: ручная кнопка проще и достаточна; авто-ретрай может маскировать конфигурационные ошибки.

### 3. Заставка «Сервис авторизации недоступен» при сбое входа

**Решение**: обернуть точку запуска входа так, чтобы сетевой сбой `signinRedirect` (или таймаут перехода) приводил к экрану «Сервис авторизации недоступен. Повторите попытку позже» с кнопкой повтора. Реализуется как состояние ошибки в `RequireAuth`/обёртке, а не глобальный React error-boundary (ошибка — это отклонённый промис, а не исключение рендера), с возможным дополнительным error-boundary для непредвиденных исключений.

**Обоснование**: `signinRedirect` обычно уводит со страницы, но при недоступности Keycloak промис отклоняется и пользователь остаётся на белом экране. Явный экран информативнее.

**Альтернатива**: только toast. Отклонено: toast легко пропустить, и под ним остаётся пустой экран.

### 4. Троттлинг `SessionBanner`

**Решение**: не показывать баннер и не запускать 30-секундный отсчёт по первому `silentRenewError`. Вместо этого:

- запускать отсчёт-до-выхода только когда до `exp` токена осталось менее порога (использовать `user.expires_at`), ИЛИ
- после N (например, 3) подряд неудачных попыток тихого обновления.
Успешный `userLoaded` сбрасывает счётчик и скрывает баннер (как сейчас).

**Обоснование**: единичный транзиентный сбой сети не должен пугать дежурного «выход через 30 сек», пока токен ещё валиден. Привязка к реальному `exp` сохраняет спек-гарантию (баннер до фактического истечения) без ложных срабатываний.

**Альтернатива**: фиксированная задержка-дебаунс на событие. Отклонено: `exp`-ориентированный триггер семантически точнее соответствует требованию «за 120 сек до истечения».

### 5. Диагностика пустых permissions

**Решение**: в `usePermissions()`/потребителе, если карта пуста, но в декодированном токене `groups` непуст, показывать диагностическое сообщение (например, на странице 403/«Нет доступа к командам») с подсказкой о вероятной неверной настройке Keycloak-маппера. Без изменения нормального потока «нет членства».

**Обоснование**: различает два разных состояния — «пользователь реально без команд» и «маппер отдал группы, но фронт их не распознал/не получил в нужном токене».

**Альтернатива**: молча редиректить на «Нет доступа». Отклонено: скрывает конфигурационную ошибку, усложняет диагностику в проде.

### 6. Документация требования Keycloak

**Решение**: зафиксировать в README фронтенда (и/или деплой-доке) обязательность groups-маппера в `id_token` и `access_token`, со ссылкой на причину (фронт vs бэкенд). Изменение кода не требуется.

**Обоснование**: связка неочевидна и не выводится из кода одной стороны; явная документация снижает риск тихого сбоя при настройке окружения.

## Risks / Trade-offs

- **Изменение `parseGroups` затрагивает все гарды** (`TenantGuard`, `AdminGuard`, `SelectTeamPage`). → Митигация: покрыть юнит-тестами (несколько тенантов, префиксные подгруппы, admin, пустой/отсутствующий `groups`) когда появится тест-инфраструктура; до тех пор — ручная проверка против набора реальных claim из Keycloak.
- **`exp`-ориентированный баннер зависит от точности `user.expires_at`** и расхождения часов клиента. → Митигация: использовать тот же порог (120 сек), что и `accessTokenExpiringNotificationTimeInSeconds`; отсчёт — мягкий, реальная граница на бэкенде (401).
- **Заставка при сбое входа может замаскировать реальную ошибку конфигурации** (например, неверный `redirect_uri`). → Митигация: логировать оригинальную ошибку в консоль; сообщение содержит кнопку повтора, а не бесконечный спиннер.
- **Диагностика пустых permissions требует доступа к декодированным claims токена** на стороне потребителя. → Митигация: переиспользовать уже декодированный `user.profile`; не вводить проверку подписи.

## Migration Plan

Изменение чисто фронтовое и обратносовместимое: новый билд статики раздаётся как обычно. Откат — возврат предыдущего образа фронтенда. Координации с бэкендом или миграции данных не требуется. Документационная правка (требование Keycloak-маппера) деплоя не требует.
