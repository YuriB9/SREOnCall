## MODIFIED Requirements

### Requirement: Настройка группирующих лейблов per-source

Сервис incident ДОЛЖЕН (SHALL) позволять администраторам тенанта задавать список лейблов для группировки алертов отдельно для каждого поддерживаемого источника (`alertmanager`, `grafana`). Конфигурация управляется через REST API; если правило не задано — используются умолчания.

Умолчания: `alertmanager` → `["alertname", "job"]`; `grafana` → `["alertname"]`.

При обработке алерта со значением `source: prometheus` (сообщения, опубликованные ingestion до канонизации имени источника) сервис ДОЛЖЕН трактовать его как `alertmanager`: поиск правила группировки и умолчания выполняются по имени `alertmanager`, чтобы заданные администратором правила применялись к таким алертам наравне с новыми.

#### Scenario: Установка группирующих лейблов для источника

- **WHEN** администратор выполняет PUT на `/api/incidents/v1/{tenant}/grouping-rules/{source}` с массивом `grouping_labels` для поддерживаемого источника
- **THEN** правило сохраняется; новые алерты из этого источника будут группироваться по указанным лейблам

#### Scenario: Алерт со старым source prometheus использует правила alertmanager

- **WHEN** из очереди потреблён алерт с `source: prometheus`, а для тенанта задано правило группировки для `alertmanager`
- **THEN** алерт группируется по правилу `alertmanager` (заданному или умолчанию), а не по встроенному фолбэку

#### Scenario: Сброс к умолчанию

- **WHEN** администратор выполняет DELETE на `/api/incidents/v1/{tenant}/grouping-rules/{source}`
- **THEN** правило удаляется и используется встроенное умолчание для источника

#### Scenario: Получение текущих правил

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/grouping-rules`
- **THEN** возвращаются правила для обоих поддерживаемых источников (явно заданные или умолчания с флагом `is_default: true`)
