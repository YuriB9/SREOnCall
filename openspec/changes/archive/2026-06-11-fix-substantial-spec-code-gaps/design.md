## Context

Пять расхождений из раздела «Существенные расхождения» аудита (`docs/spec-vs-code-audit.md`, пункты 3–7). Текущее состояние:

- **(3)** `DEDUP_TTL_SECONDS` по умолчанию 4 часа (`services/ingestion/internal/config/config.go`), спека `alert-ingestion` декларирует 5 минут.
- **(4)** Ingestion нормализует алерты Alertmanager с `Source: domain.SourcePrometheus = "prometheus"`; правила группировки, токены и API оперируют `alertmanager`. Правило, заданное администратором для `alertmanager`, не находится consumer'ом (он ищет по `prometheus`); дефолты совпадают только из-за дублирующей ветки в `DefaultGroupingLabels`.
- **(5)** Фронтенд объявляет `IncidentSeverity = 'critical' | 'high' | 'medium' | 'low'`; бэкенд производит `critical | high | warning | info` (`mapSeverity` в ingestion). Фильтры `medium`/`low` пусты, бейджи `warning`/`info` без подписи и стиля.
- **(6)** `useAudioEnabled` трактует отсутствие ключа `localStorage` как «включено» (`!== 'false'`); спека требует «по умолчанию выключен».
- **(7)** `CreateOverride` при пересечении возвращает `409 {"error": "..."}`; фронтенд (`OverrideConflictError`) ожидает `existing_start`/`existing_end`/`existing_user` и рендерит «Invalid Date — Invalid Date (undefined)».

## Goals / Non-Goals

**Goals:**
- Спека и код согласованы по всем пяти пунктам; направление каждого исправления зафиксировано (3 — спека к коду, 4/5/6/7 — код к спеке/канону).
- Кастомные правила группировки для Alertmanager начинают применяться к реальным알ертам.
- Фильтр и бейджи критичности в UI работают для всех значений, которые реально производит бэкенд.
- Модальное окно конфликта переопределений показывает осмысленное описание конфликта.

**Non-Goals:**
- «Мелкие расхождения» аудита (пункты 8–14), включая 422-валидацию source в grouping-rules (п. 9) — отдельные изменения; здесь не трогаем, чтобы не раздувать объём.
- Введение новых значений severity или маппинга `medium/low` на бэкенде.
- Изменение семантики дедупликации (только документация значения по умолчанию).

## Decisions

**(3) Спека → 4 часа, код не меняется** *(решение пользователя)*.
4 часа соответствуют типичному `repeat_interval` Alertmanager: повторные firing-уведомления того же алерта не создают шум в шине и в `raw_alerts`. Resolved-алерт удаляет dedup-ключ, поэтому реальное повторное срабатывание после резолва проходит немедленно — длинный TTL не задерживает новые инциденты. Альтернатива (код → 5 минут) увеличила бы трафик событий без функциональной выгоды: инцидент всё равно не дублируется благодаря группировке.

**(4) Канонизация `alertmanager` в ingestion + backfill + алиас на переходный период** *(решение пользователя)*.
- `pkg/domain`: `SourcePrometheus` заменяется на `SourceAlertmanager AlertSource = "alertmanager"` (единый словарь с токенами и правилами группировки).
- Backfill-миграции: `UPDATE ingestion.raw_alerts SET source='alertmanager' WHERE source='prometheus'` и `UPDATE incident.incident_alerts SET source='alertmanager' WHERE source='prometheus'` — идемпотентны.
- Алиас в incident-сервисе: при поиске правила группировки source `prometheus` трактуется как `alertmanager` (нормализация в consumer перед `GetGroupingRule`), `DefaultGroupingLabels` сохраняет обе ветки. Это покрывает сообщения старого формата, оставшиеся в очереди `alerts.incident` на момент деплоя.
- **Известное ограничение (принято осознанно):** fingerprint = SHA-256(labels + source + tenant). Алерт, чей firing был принят со `source=prometheus`, после деплоя пришлёт resolved с другим fingerprint — `ResolveAlert` его не найдёт, сообщение будет проигнорировано (по спеке resolved без матча игнорируется без ошибки), инцидент останется открытым до ручного закрытия. Окно риска — алерты, активные в момент деплоя. Пересчитать сохранённые fingerprint'ы нельзя (в `incident_alerts` нет labels). Альтернатива «алиас навсегда, source не менять» отклонена пользователем: два имени одного источника навсегда хуже одноразового переходного окна.

**(5) Фронтенд → словарь бэкенда, спека UI фиксирует перечень.**
`IncidentSeverity = 'critical' | 'high' | 'warning' | 'info'`; подписи «Критический / Высокий / Предупреждение / Инфо»; стили: `warning` наследует жёлтую схему текущего `medium`, `info` — нейтральную текущего `low`. Источник истины — `pkg/domain` (`mapSeverity`), UI лишь отображает. Альтернатива (маппить на бэкенде в `medium/low`) меняла бы каноническую схему алерта и все события ради косметики UI.

**(6) Дефолт звука — выключен.**
`useAudioEnabled`: `localStorage.getItem('oncall.audioEnabled') === 'true'`. Логика сохранения (`String(value)`) не меняется; для пользователей, уже включавших звук явно, поведение сохраняется (`'true'` в storage). Спека уже корректна.

**(7) Структурированное тело 409.**
`store.CreateOverride` при обнаружении пересечения возвращает конфликтующее переопределение (расширение `ErrConflict` до типизированной ошибки с данными либо отдельное возвращаемое значение); handler сериализует `{"error": ..., "existing_start": ..., "existing_end": ..., "existing_user": ...}` (RFC3339). `existing_user` — `user_id` конфликтующего переопределения (username резолвится на фронте через members; передавать `user_id` достаточно — UI уже строит `userMap`). Контракт совпадает с уже написанным `ConflictDetail` фронтенда — правок UI не требуется, кроме возможного резолва имени.

## Risks / Trade-offs

- [(4) Незакрытые инциденты для алертов, активных в момент деплоя] → деплоить в окно с минимумом активных алертов; задокументировать в release notes; resolved без матча по спеке игнорируется без ошибки, система не падает.
- [(4) Старые сообщения в очереди с `source=prometheus`] → алиас при поиске правил группировки; backfill выполняется при старте после потребления очереди — порядок не критичен, т.к. алиас покрывает оба имени.
- [(5) Закладки/ссылки с `?severity=medium|low`] → фильтр просто вернёт пустой список, как и сейчас; значения в селекторе обновляются — новые ссылки корректны.
- [(6) Пользователи, привыкшие к звуку без явного включения] → разовое неудобство: включить тумблер заново; соответствует спеке.
- [(7) Изменение формата ошибки 409] → поле `error` сохраняется, добавляются новые поля — обратносовместимо для любого клиента, читавшего только `error`.

## Migration Plan

1. Миграции backfill (ingestion, incident) — идемпотентны, накатываются при старте сервисов.
2. Деплой ingestion и incident (порядок не важен благодаря алиасу), затем scheduling и frontend в любом порядке.
3. Откат: код откатывается свободно; backfill source не требует отката (значение становится каноническим; старый код ingestion писал `prometheus` только для новых алертов, алиас в incident не удаляем в этом изменении).
