## MODIFIED Requirements

### Requirement: Список инцидентов с фильтрацией и пагинацией

Система ДОЛЖНА (SHALL) отображать постраничную таблицу инцидентов текущего тенанта, получаемых через `GET /api/incidents/v1/{tenant}/incidents`. Список ДОЛЖЕН автоматически обновляться каждые 12 секунд через фоновый поллинг. Система ДОЛЖНА поддерживать фильтрацию по статусу (мультивыбор `open`, `acknowledged`, `resolved`; значения передаются одним параметром через запятую) и по критичности (одно значение). Состояние фильтров ДОЛЖНО отражаться в query-параметрах URL, чтобы отфильтрованные виды можно было передавать ссылкой.

Словарь критичности ДОЛЖЕН совпадать с канонической схемой алерта бэкенда: `critical | high | warning | info` (подписи: «Критический», «Высокий», «Предупреждение», «Инфо»). Фильтр критичности ДОЛЖЕН предлагать только эти значения; бейджи критичности ДОЛЖНЫ иметь подпись и стиль для каждого из четырёх значений. Значения `medium` и `low` НЕ ДОЛЖНЫ использоваться — бэкенд их не производит.

Таблица ДОЛЖНА содержать колонку «Эскалация», отражающую состояние эскалации инцидента. Состояния запрашиваются массово через `GET /api/escalations/v1/{tenant}/incidents/state?incident_ids=...` по идентификаторам отображаемых инцидентов (один запрос на страницу, без поштучных вызовов) и обновляются вместе со списком. Для инцидента без состояния эскалации колонка ДОЛЖНА показывать «—» (эскалация не запускалась). Для инцидента с состоянием колонка ДОЛЖНА показывать бейдж с текущим уровнем (`current_tier`) и статусом (`active | acknowledged | resolved | exhausted`). Недоступность escalation-сервиса НЕ ДОЛЖНА ломать отображение списка: при ошибке запроса состояний колонка показывает «—».

#### Scenario: Первоначальная загрузка

- **WHEN** пользователь переходит на `/[tenant]/incidents`
- **THEN** система ДОЛЖНА отобразить таблицу инцидентов, отсортированных по `created_at` по убыванию, с колонками: Критичность, Название, Статус, Alertname (значение лейбла `alertname` инцидента или «—»), Создан, Подтверждён кем, Эскалация; источник каждого алерта доступен на вкладке «Алерты» панели деталей

#### Scenario: Колонка эскалации для инцидента без эскалации

- **WHEN** в массовом ответе состояний нет элемента для отображаемого инцидента
- **THEN** в колонке «Эскалация» этой строки отображается «—»

#### Scenario: Колонка эскалации для инцидента с активной эскалацией

- **WHEN** массовый ответ содержит состояние инцидента с `status: active` и `current_tier: N`
- **THEN** в колонке «Эскалация» отображается бейдж с уровнем `N` и признаком активного статуса

#### Scenario: Escalation-сервис недоступен

- **WHEN** запрос массовых состояний эскалации завершился ошибкой
- **THEN** список инцидентов отображается штатно, а колонка «Эскалация» во всех строках показывает «—»

#### Scenario: Фильтрация по статусу

- **WHEN** пользователь выбирает «Открытые» в фильтре статуса
- **THEN** система ДОЛЖНА перезапросить данные с `?status=open`, отобразить только открытые инциденты и обновить query-параметр в URL

#### Scenario: Фильтрация по нескольким статусам

- **WHEN** пользователь выбирает «Открытые» и «Подтверждённые» одновременно
- **THEN** система ДОЛЖНА перезапросить данные с `?status=open,acknowledged` и отобразить инциденты обоих статусов

#### Scenario: Фильтрация по критичности warning

- **WHEN** пользователь выбирает «Предупреждение» в фильтре критичности
- **THEN** система ДОЛЖНА перезапросить данные с `?severity=warning` и отобразить только инциденты с критичностью `warning`

#### Scenario: Бейдж для инцидента с критичностью info

- **WHEN** в списке отображается инцидент с `severity: info`
- **THEN** строка содержит бейдж с подписью «Инфо» и определённым для `info` стилем (не пустой бейдж)

#### Scenario: Автоматическое фоновое обновление

- **WHEN** с момента последнего успешного запроса прошло 12 секунд
- **THEN** система ДОЛЖНА запросить список инцидентов повторно без действий пользователя и обновить отображаемые данные при их изменении

#### Scenario: Обнаружен новый инцидент во время поллинга

- **WHEN** ответ поллинга содержит ID инцидента, отсутствовавший в предыдущем ответе, со статусом `open`
- **THEN** система ДОЛЖНА обновить список и, если звуковые уведомления включены, воспроизвести одиночный сигнал через Web Audio API
