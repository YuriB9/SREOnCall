## Why

В списке инцидентов дежурный не видит, запущена ли по инциденту эскалация и на каком уровне она сейчас. Эта информация есть в escalation-сервисе (`current_tier`, `status`), но доступна только поштучно через `GET .../incidents/{incidentId}/state` (404, если эскалация не назначена). Чтобы вывести её колонкой в таблице, нужен способ получить состояния сразу для группы инцидентов — иначе на странице со списком получится N+1 запросов.

## What Changes

- **Backend (escalation):** добавляется массовый эндпоинт `GET /api/escalations/v1/{tenant}/incidents/state?incident_ids=<id1,id2,...>`, возвращающий массив состояний эскалации для запрошенных инцидентов. Инциденты без состояния просто отсутствуют в ответе (трактуются UI как «эскалация не запускалась»). Существующий поштучный `/state` не меняется.
- **Frontend (дашборд):** в таблицу инцидентов добавляется колонка «Эскалация». Для каждого инцидента отображается:
  - «—», если состояние эскалации отсутствует (не запускалась);
  - бейдж с текущим уровнем (`current_tier`) и статусом (`active | acknowledged | resolved | exhausted`), если эскалация запущена.
  - Состояния запрашиваются одним массовым вызовом по ID видимых инцидентов и обновляются вместе со списком (поллинг 12 c).

Изменения аддитивны: новый эндпоинт и новая колонка, существующие контракты не меняются.

### Modified Capabilities
- `escalation-policies`: добавляется требование о массовом получении состояний эскалации по списку инцидентов (новый эндпоинт; существующее требование «Видимость состояния эскалации» сохраняется без изменений).
- `incident-dashboard-ui`: в требование «Список инцидентов с фильтрацией и пагинацией» добавляется колонка «Эскалация» с признаком запуска и текущим уровнем.

## Impact

- **Сервисы:** `services/escalation` — `internal/handler` (новый bulk-обработчик), `internal/store` (выборка состояний по множеству `incident_id`), маршрут в `cmd/server/main.go`.
- **API:** новый `GET /api/escalations/v1/{tenant}/incidents/state?incident_ids=...` (аддитивно).
- **Frontend:** `frontend/src/api/escalations.ts` (хук массовой выборки состояний), `frontend/src/pages/IncidentListPage.tsx` (колонка «Эскалация» и рендер бейджа), при необходимости типы в `frontend/src/api/types.ts`.
- **События RabbitMQ:** не затрагиваются.
- **БД:** схема не меняется (чтение существующей таблицы состояний эскалации).
