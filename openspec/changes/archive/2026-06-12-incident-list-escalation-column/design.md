## Context

Состояние эскалации хранится в escalation-сервисе (`EscalationState`: `incident_id, current_tier, status, escalate_at, ...`) и читается поштучно через `GET .../incidents/{incidentId}/state` (404, если состояния нет). Список инцидентов рендерит `IncidentListPage` из данных incident-сервиса; колонки: Критичность, Название, Статус, Alertname, Создан, Подтверждён. Vite-прокси уже маршрутизирует `/api/escalations` → escalation-сервис (8083).

Чтобы показать эскалацию колонкой, UI нужны состояния для всех строк страницы. Поштучные запросы дали бы N+1, поэтому требуется массовая выборка.

## Goals / Non-Goals

**Goals:**
- Один массовый эндпоинт состояний по списку `incident_ids`.
- Колонка «Эскалация» в списке: «не запускалась» / уровень + статус.
- Устойчивость UI к недоступности escalation-сервиса (список не ломается).

**Non-Goals:**
- Изменение логики эскалации, истории или поштучного `/state`.
- Серверное соединение инцидентов и состояний (join на бэке incident-сервиса) — UI соединяет по `incident_id` на клиенте.
- Пагинация/фильтрация самого эндпоинта состояний (выборка строго по переданным id).

## Decisions

**1. Массовый эндпоинт на стороне escalation, выборка по `incident_ids`.**
`GET /api/escalations/v1/{tenant}/incidents/state?incident_ids=A,B,C` → массив состояний только для существующих. UI передаёт id видимой страницы. Альтернатива «вернуть все активные состояния тенанта» отвергнута: несогласовано с пагинацией списка и может расти неограниченно; выборка по явным id точна и ограничена.

**2. Соединение на клиенте.**
UI строит `Map<incident_id, EscalationState>` и рендерит колонку по строкам. Это держит incident- и escalation-сервисы развязанными (никаких синхронных вызовов между ними) и согласуется с архитектурой (межсервисные join'ы не делаются на бэке ради UI).

**3. Отдельный TanStack Query-хук, привязанный к id страницы.**
`useEscalationStates(tenant, incidentIds)` с ключом, включающим отсортированные id; `enabled` при непустом списке; тот же ритм обновления, что у списка (можно завязать на те же 12 c). При ошибке — пустая карта, колонка показывает «—» (graceful degradation).

**4. Ограничение размера запроса.**
Поскольку список постраничный, число id ограничено размером страницы — длина URL безопасна. Если в будущем размер страницы вырастет, перейти на POST с телом; сейчас GET с query достаточно.

**5. Рендер бейджа.**
`current_tier` → «Уровень N» (с учётом 0/1-индексации — уточнить по данным); `status` задаёт цвет (active — акцент/опасность, acknowledged — нейтральный, resolved — зелёный, exhausted — серый/предупреждение). Точные подписи/стили — деталь фронтенда.

## Risks / Trade-offs

- **Рассинхрон списка и состояний (разные моменты выборки)** → оба обновляются на одном ритме поллинга; кратковременная неконсистентность некритична для индикатора.
- **Длина URL при больших страницах** → ограничена размером страницы; запасной путь — POST с телом.
- **Недоступность escalation-сервиса** → колонка деградирует до «—», список работает (требование зафиксировано в спеке).
- **Индексация уровня (`current_tier`)** → согласовать отображение с фактической нумерацией tier в данных, чтобы «Уровень N» совпадал с политикой.

## Migration Plan

- Только код (escalation-хендлер/стор/маршрут + фронт). Миграций БД нет.
- Деплой обычный, аддитивный: новый эндпоинт и новая колонка; старые клиенты не затронуты.
- Проверка: инцидент с запущенной эскалацией показывает уровень/статус в колонке; инцидент без эскалации — «—»; при остановленном escalation-сервисе список грузится, колонка «—».
- Откат — предыдущие бинарь/сборка фронта; данные не меняются.

## Open Questions

- Нумерация уровней в подписи: `current_tier` 0-индексирован? Отразить «Уровень N» консистентно с UI политик.
- Привязывать обновление состояний к тому же 12-секундному поллингу, что и список, или к собственному интервалу? Базово — единый ритм.
