# Design — guard-webhook-ssrf

## Контекст

S2 (см. `docs/audit/05-security.md`): тенант-задаваемый `mattermost_webhook_url` используется как цель
исходящего HTTP-запроса notification-сервиса без ограничения адреса → SSRF во внутреннюю сеть и к
cloud-metadata. Нужен барьер на двух границах: на записи (быстрый отказ пользователю) и на отправке
(defense in depth для уже сохранённых значений, редиректов и DNS-rebinding).

## Решение

### `pkg/ssrf`

Новый общий пакет (нужен и scheduling, и notification — выносим, чтобы не дублировать логику диапазонов):

- `ValidateURL(raw string) error` — синтаксический и адресный фильтр:
  1. парсит URL; пустой/битый → `ErrInvalidURL`;
  2. схема обязана быть `https` → иначе `ErrNotHTTPS`;
  3. резолвит хост (`net.DefaultResolver.LookupIPAddr`); **все** полученные IP проверяются — если
     хотя бы один в заблокированном диапазоне → `ErrBlockedAddress`. Литеральный IP в хосте проверяется
     напрямую без DNS.
- `GuardedDialContext(base *net.Dialer) func(ctx, network, addr) (net.Conn, error)` — обёртка дозвона:
  резолвит фактический адрес и блокирует приватный IP **в момент соединения**. Это ловит TOCTOU между
  валидацией и отправкой, DNS-rebinding и редиректы (HTTP-клиент при редиректе дозванивается заново
  через тот же `DialContext`).
- `isBlockedIP(ip net.IP) bool` — единый предикат: loopback, private (RFC 1918), link-local (incl.
  `169.254.169.254`), unspecified, multicast, ULA (`fc00::/7`), а также IPv4-mapped варианты.

### Блок приватных диапазонов vs allowlist доменов

Выбран **блок приватных диапазонов** (ADR-0013). Allowlist доменов строже, но требует ведения
конфигурации и ломает self-hosted Mattermost на произвольных публичных доменах — а это легитимный
сценарий продукта. Блок-лист закрывает вектор SSRF (внутренняя сеть + metadata) без эксплуатационной
нагрузки на тенантов.

### Точки применения

- **scheduling** — `PutTenantNotificationConfig`: при непустом `mattermost_webhook_url` →
  `ssrf.ValidateURL`; ошибка → `writeError(w, 422, ...)` (по конвенции 422 для ошибок валидации).
  Маскированное значение (`***`) сюда не попадает: UI присылает пустое поле, когда URL не меняется.
- **notification** — `dispatcher.NewMattermost`: `http.Client.Transport` получает
  `GuardedDialContext`. Синтаксический `webhookURLUsable` остаётся как дешёвый pre-check; реальное
  адресное ограничение на отправке обеспечивает dialer. Ошибка дозвона → существующий путь `failed`.

## Совместимость и миграция

- Изменений событий шины и схемы БД нет.
- Уже сохранённые в БД небезопасные URL **не мигрируются**: на отправке они начнут блокироваться
  guarded-dialer (доставка `failed` + лог) — это и есть желаемое поведение S2. Тенант-админ, увидев
  отказ, должен задать корректный публичный https-URL (новый PUT провалидируется на записи).
- Успешный путь для публичных https-URL Mattermost не меняется — деплой безопасен, без флагов.

## Отклонённые альтернативы

- **Allowlist доменов Mattermost** — отклонён: ломает self-hosted на произвольных доменах, требует
  конфигурации (см. ADR-0013).
- **Только валидация на записи (без dial-guard)** — отклонён: не закрывает уже сохранённые URL,
  DNS-rebinding и редиректы на приватные адреса.
- **Только dial-guard (без 422 на записи)** — отклонён: пользователь не получает понятной ошибки и
  узнаёт о проблеме лишь по молчаливому `failed` в журнале доставки.
- **Готовый сторонний SSRF-пакет** — отклонён на данном этапе: добавляет зависимость ради небольшого
  объёма логики; диапазоны и dial-guard реализуются стандартной библиотекой.
