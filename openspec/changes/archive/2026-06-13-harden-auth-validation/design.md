## Context

Аутентификация платформы держится на `pkg/auth/auth.go`: `Middleware(jwksURL, adminKey)` валидирует Bearer JWT по JWKS (Keycloak) либо пропускает запрос по статическому `X-Admin-Key` (межсервисный god-доступ). Каждый из 4 сервисов (`incident`, `escalation`, `notification`, `scheduling`) подключает middleware **условно**: при пустом `KEYCLOAK_JWKS_URL` ставится проходной обработчик. `ingestion` JWT-middleware не использует (webhook-token авторизация) — вне объёма.

Аудит (`docs/audit/05-security.md`, находки S1/S3/S4/S5) показал, что эта схема fail-open и валидация JWT неполная. CH03 укрепляет именно `pkg/auth` + wiring 4 сервисов. Консолидация wiring в общий `pkg/httpserver` (F10) и rate-limit (S6) — отдельный CH10; SSRF (S2) — CH04.

## Goals / Non-Goals

**Goals:**
- Fail-closed: без сконфигурированного auth сервис не стартует, если явно не разрешено (S1).
- Constant-time сравнение admin-key (S3).
- Строгая валидация JWT: alg-allowlist + обязательный `exp` всегда; `iss`/`aud` — когда заданы (S4).
- Форс https для JWKS с escape-hatch для локалки (S5).
- Эргономичная сигнатура `Middleware` (структура опций вместо растущего списка позиционных параметров).

**Non-Goals:**
- Вынос wiring в `pkg/httpserver` (F10 → CH10).
- Жёсткое требование `iss`/`aud` для всех деплоев (ужесточение после раскатки env).
- Сужение полномочий admin-key / mTLS / ротация (организационная мера из S3, не код CH03).
- Rate-limit (S6 → CH10), SSRF webhook (S2 → CH04).
- Изменение webhook-token авторизации `ingestion`.

## Decisions

**1. Сигнатура `Middleware(Options)` вместо позиционных параметров.**
Нужно передать 5 параметров (JWKSURL, AdminKey, Issuer, Audience, AllowInsecureJWKS). Структура опций (`auth.Options`) читаема на вызове и расширяема без новой ломки сигнатуры. Альтернатива — функциональные опции (`WithIssuer(...)`) — избыточна для одного конструктора с фиксированным набором полей; отклонена ради простоты.

**2. Fail-closed остаётся в `main.go` каждого сервиса (пока).**
Логика «пустой JWKS → exit, кроме `AUTH_DISABLED`» дублируется в 4 `main.go` (4 почти идентичных блока). Это сознательно: консолидация в общий хелпер — это F10/CH10, расширять объём CH03 нельзя. Блоки делаются единообразными, чтобы CH10 их легко вынес. Альтернатива (хелпер `auth.BuildMiddleware(cfg, logger)` уже сейчас) отклонена как заступ на чужой чейндж.

**3. `iss`/`aud` — enforce-if-configured, не обязательны.**
`WithValidMethods(["RS256"])` и `WithExpirationRequired()` включаются всегда (нулевой риск для валидных Keycloak-токенов). `WithIssuer`/`WithAudience` добавляются только если заданы соответствующие env; при пустых — `Warn` на старте. Причина: жёсткое требование могло бы уронить деплои, ещё не прописавшие эти env. Ужесточение — отдельный шаг после раскатки конфигурации. Альтернатива (требовать всегда) отклонена из-за риска массового отказа старта при выкатке.

**4. Проверка https JWKS — в `Middleware`, возврат `error`.**
`Middleware` уже возвращает `error`, сервис на ошибке делает `os.Exit(1)`. Проверку схемы (`https`, либо `http`+`AllowInsecureJWKS`) кладём туда же — единая точка, нельзя «забыть». Альтернатива (валидация в `config.Load`) рассеивает security-инвариант по 4 конфигам; отклонена.

## Risks / Trade-offs

- **Fail-closed уронит мисконфигнутый деплой** → это цель S1. Митигация: явный `AUTH_DISABLED=true` для локалки + понятное сообщение в `Error`-логе с подсказкой про флаг.
- **`WithExpirationRequired()` отклонит легаси-токены без `exp`** → у Keycloak access-token всегда есть `exp`, риск низкий. Митигация: задокументировано в ADR-0012.
- **Дублирование fail-closed-блока в 4 main.go** → временный долг до CH10. Митигация: единообразные блоки, явная ссылка в design на F10.
- **Смена сигнатуры `Middleware`** → ломает только внутренние вызовы, все правятся в этом же чейндже; `go build ./...` по всему воркспейсу это поймает.

## Migration Plan

1. Выкатка кода безопасна для деплоев с непустым `KEYCLOAK_JWKS_URL`: стартуют как раньше; при отсутствии `KEYCLOAK_ISSUER`/`KEYCLOAK_AUDIENCE` — предупреждение в логе, поведение валидации не строже прежнего по iss/aud.
2. **До выката** убедиться, что у всех prod-сервисов `KEYCLOAK_JWKS_URL` задан (иначе под не поднимется). Для локалки/CI без Keycloak — выставить `AUTH_DISABLED=true`.
3. Прописать `KEYCLOAK_ISSUER`/`KEYCLOAK_AUDIENCE` в манифестах → включается полная валидация (отдельный шаг, можно после CH03).
4. Если JWKS тянется по http внутри кластера — временно `AUTH_INSECURE=true`, план перевода на https.
5. **Rollback:** откат образа — поведение возвращается к fail-open (нежелательно, но безопасно для доступности).
