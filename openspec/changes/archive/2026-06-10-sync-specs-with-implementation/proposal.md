# Why

Сверка `docs/openspec-code-verification.md` показала: код фронтенда и бэкенда уже следует контрактам, отличающимся от написанного в спецификациях — UI-спеки ссылаются на несуществующие эндпоинты (тенант-уровневый `/oncall`, `PATCH is_default`, контакты через scheduling), бэкенд-спеки описывают неверные пути ingestion, поле `timeout_minutes` вместо фактического `timeout_seconds`, lookup токена «в таблице» вместо Redis и умалчивают о реализованных эндпоинтах. Пока спеки лгут, они бесполезны как источник истины для ревью и для генерации следующих изменений. Это изменение — документационная синхронизация: спецификации приводятся к фактической реализации, а решения по обнаруженным пробелам (severity инцидента, осиротевший resolved-алерт, границы каскада удаления тенанта, права на создание тенанта) фиксируются явно.

## What Changes

- **UI-спеки → фактические контракты**: виджет «кто дежурит» — карточки per-schedule через `GET /schedules/{id}/oncall` (без уровня эскалации); Gantt — один запрос `GET /schedules/{id}/shifts?from&to` на расписание; профиль — контакты через notification-сервис с селектором тенанта; назначение default-политики — `PUT /default-policy`; редактирование политики — замена (POST новой → перенос default → DELETE старой); пути настроек тенанта — `/api/schedules/v1/tenants/{slug}/...`; фильтры дашборда — мультивыбор статуса и severity (без фильтра источника); вкладка «История» — без событий эскалации.
- **Бэкенд-спеки → фактическая реализация**: пути ingestion `/api/ingest/v1/webhook/{source}`; синхронная публикация с HTTP 503 и откатом dedup-ключа при сбое; идентификация тенанта через Redis-индекс `oncall:tokens:{hash}`; модель политики эскалации без `schedule_id` (tier: `tier_number`, `timeout_seconds`, `notify_schedule_id`); немедленная публикация tier-1 `escalation.triggered` при назначении политики; остановка эскалации при `resolved`; документирование реализованных эндпоинтов (`GET /default-policy`, история эскалации, `GET /incidents/{id}`, алерты инцидента, удаление переопределений).
- **Фиксация решений по пробелам**: severity инцидента наследуется от первого алерта и не пересчитывается; лейблы копируются только при создании; resolved-алерт без открытого инцидента игнорируется; удаление тенанта каскадно только в данных scheduling; `POST /tenants` доступен любому аутентифицированному пользователю.
- Изменений поведения кода нет — только спецификации (плюс проверочные задачи «спека = код»).

### Modified Capabilities

- `alert-ingestion`: пути вебхуков, Redis-lookup токена, синхронная модель публикации с 503.
- `escalation-policies`: модель политики и tier'ов (`timeout_seconds`, без `schedule_id`), немедленный tier-1 триггер, остановка при `resolved`, `GET /default-policy`, история эскалации, редактирование политики заменой.
- `incident-management`: severity и лейблы от первого алерта, осиротевший resolved, фактический набор фильтров списка (без `label`), эндпоинты деталей инцидента и его алертов.
- `tenant-management`: права на `POST /tenants`, границы каскада удаления тенанта.
- `oncall-schedule-ui`: per-schedule виджеты, Gantt через `/shifts`.
- `user-profile-ui`: контакты через notification-сервис, селектор тенанта.
- `escalation-policy-ui`: default через `PUT /default-policy`, редактирование заменой, `timeout_seconds`.
- `tenant-settings-ui`: пути `/tenants/{slug}/...` для токенов и участников, выбор источника из enum.
- `incident-dashboard-ui`: фильтры без источника, история без событий эскалации.

## Impact

- **Код**: изменений нет; задачи — только верификация «дельта = код» и при необходимости микроправки текстов ошибок.
- **Спеки**: 9 дельта-файлов; после архивации главные спеки становятся достоверным источником истины.
- **Зависимости**: применяется и архивируется **после** `fix-p0-delivery-and-filters` и `enrich-notifications-drop-zabbix` — дельты требований «Список и фильтрация инцидентов» и «Срабатывание эскалации по таймауту» строятся поверх текстов этих изменений.
- **Процесс**: закрывает пункты 9-11 плана `docs/openspec-code-verification.md`; вместе с реестром контрактов (отдельная рекомендация) снижает риск повторного дрейфа.
