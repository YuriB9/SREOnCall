## Контекст

Бэкенд-контракт расписания (`internal/handler/handler.go`):

- `POST /api/schedules/v1/{tenant}/schedules` — тело `{ name, timezone?, rotation[], shift_duration, start_date }`.
  - Обязательны: `name`, `rotation` (непустой), `shift_duration`, `start_date`.
  - `timezone` опционален, по умолчанию `UTC`.
  - `shift_duration` — строка ISO-8601 duration (валидируется `rotation.ParseISO8601Duration`, например `P1D`, `P7D`, `P14D`, `PT8H`, `PT12H`).
  - `start_date` — `YYYY-MM-DD`.
  - `rotation` — упорядоченный массив `user_id` (sub из Keycloak); порядок определяет очередь ротации.
  - Ошибки: `422` с `{ "missing_fields": [...] }` либо `{ "error": "..." }`.
  - Успех: `201` с объектом `Schedule`.
- `PATCH .../schedules/{id}` — частичное обновление, все поля опциональны (`200`).
- `DELETE .../schedules/{id}` — `204`, либо `404`.

Тип `Schedule` уже есть во фронтенде (`frontend/src/api/types.ts`): `{ id, tenant_id, name, timezone, rotation, shift_duration, start_date, created_at, updated_at }`.

Существующий паттерн модалки — `CreateOverrideModal.tsx` (fixed-overlay, `<form>`,
inline-ошибки, мутация через TanStack Query). Новую форму строим в том же стиле,
переиспользуя `Button`, `useMembers`, инвалидцию через `scheduleKeys`.

## Цели / Не-цели

**Цели**
- Дать возможность создать, отредактировать и удалить расписание полностью из UI.
- Переиспользовать существующий контракт бэкенда без изменений.
- Удержать UX согласованным с `CreateOverrideModal` (стиль, обработка ошибок, инвалидция кэша).

**Не-цели**
- Изменение бэкенда, схемы БД, событийного конвейера.
- Расширенный редактор ротации (drag-and-drop сложный). Достаточно
  добавления/удаления и перемещения вверх/вниз для задания порядка.
- Предпросмотр будущих смен внутри формы (Gantt и так перерисуется после сохранения).

## Решения

### Маппинг полей формы → API

| Поле формы | Контрол | Поле API |
|---|---|---|
| Название | text input | `name` |
| Часовой пояс | text/select (по умолчанию `UTC`) | `timezone` |
| Продолжительность смены | select из пресетов (`P1D`/`P7D`/`P14D`/`PT8H`/`PT12H`) | `shift_duration` |
| Дата старта | `type="date"` | `start_date` (`YYYY-MM-DD`) |
| Ротация | список выбранных участников с порядком | `rotation` (массив `user_id`) |

`shift_duration` ограничиваем набором пресетов — это покрывает реальные сценарии
(`scripts/seed-schedules.sh` использует `P1D`/`P7D`/`P14D`) и избавляет от
свободного ввода ISO-8601 пользователем. Бэкенд всё равно валидирует значение.

### Редактирование переиспользует ту же модалку

`ScheduleFormModal` принимает опциональный `schedule?: Schedule`. Если он передан —
режим редактирования (заголовок «Редактировать расписание», предзаполненные поля,
`PATCH`); иначе — создание (`POST`). Это устраняет дублирование формы.

### Ротация — выбор из участников тенанта

Берём список участников из `useMembers(tenant)` (уже используется на странице).
В форме: выпадающий список «добавить инженера» + упорядоченный список выбранных с
кнопками удалить и переместить вверх/вниз. Отправляем `rotation` как массив
`user_id`. Подписи показываем по `preferred_username`.

### Обработка ошибок

`422` от бэкенда (`missing_fields` или `error`) показываем встроенно в модалке, как
в `CreateOverrideModal` (не toast). Клиентская валидация (`canSubmit`) блокирует
отправку при незаполненных обязательных полях и пустой ротации.

### Удаление

Кнопка «Удалить» на расписании → `window.confirm` (или простое inline-подтверждение)
→ `useDeleteSchedule`. После успеха инвалидируем `scheduleKeys(tenant).list()` —
строка исчезает из Gantt и из виджета «Кто дежурит сейчас».

## Риски / Компромиссы

- **Свободный `timezone` vs валидный IANA tz.** Невалидная зона не отлавливается
  явной валидацией на бэкенде при создании (используется как есть). Компромисс:
  даём select из распространённых зон + дефолт `UTC`, чтобы снизить риск опечаток.
- **Порядок ротации важен** (определяет очередь). UI должен явно показывать индекс/порядок.
- **Пресеты `shift_duration`** не покрывают экзотические длительности; при
  необходимости позже можно добавить «другое» со свободным вводом — вне рамок этого change.
