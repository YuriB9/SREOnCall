# Design: db-atomicity-and-state-transitions

## Контекст

Переходы состояний в проекте — read-modify-write без защиты от конкуренции:

- **escalation:** монитор читает просроченные state'ы (`ListExpiredStates`) и для каждого
  вызывает `AdvanceOrExhaust`, который пишет новое состояние безусловно `WHERE id=$`.
  `FOR UPDATE SKIP LOCKED` стоит вне транзакции — блокировка живёт лишь на время SELECT.
- **incident:** `PatchStatus` читает инцидент, валидирует переход стейт-машиной, пишет статус
  безусловно `WHERE id=$ AND tenant_id=$`.

Обе схемы ломаются при конкуренции (две реплики escalation, параллельные PATCH'и,
авто-резолв из конвейера на фоне UI-действия). Транзакция используется ровно в одном месте
(`CreatePolicy`).

## Решение

Единый паттерн **optimistic concurrency через guarded-CAS**, выполняемый внутри транзакции
вместе с записью аудит-истории.

### 1. Guarded-CAS вместо безусловного UPDATE (D1, D3)

UPDATE дополняется условием на **ожидаемое предыдущее состояние** и проверкой `RowsAffected()`:

```sql
-- escalation: захват просроченного state'а
UPDATE escalation.incident_escalation_states
SET current_tier=$1, status=$2, escalate_at=$3, updated_at=now()
WHERE id=$4 AND current_tier=$5 AND status=$6   -- $5/$6 = ожидаемые tier/status
```
```sql
-- incident: смена статуса
UPDATE incident.incidents SET status=$new, ...
WHERE id=$id AND tenant_id=$t AND status=$expectedOld
```

`RowsAffected()==0` означает «состояние уже изменил другой обработчик»:
- в escalation → `errs.ErrConflict`, `AdvanceOrExhaust` **тихо пропускает** строку (debug-лог),
  **не публикуя** `escalation.triggered`/`exhausted`. Ровно один обработчик из конкурирующих
  выигрывает CAS и публикует.
- в incident → `errs.ErrConflict`, `PatchStatus` отдаёт **HTTP 409** (код конфликта по
  `config.yaml`).

Порядок критичен: CAS-UPDATE «застолбляет» работу **до** любой публикации в RabbitMQ. Публикация
и сетевые вызовы (`triggerTier`→scheduling) остаются вне транзакции и идут только после commit.

### 2. Транзакционные составные записи (D2)

В каждый store добавлен приватный хелпер `withTx(ctx, fn func(pgx.Tx) error) error`
(`Begin`/`defer Rollback`/`Commit`, как в существующем `CreatePolicy`). Внутренние варианты
вставок принимают узкий интерфейс `dbConn` (методы `Exec`/`Query`/`QueryRow`), которому
удовлетворяют и `*pgxpool.Pool`, и `pgx.Tx` — это позволяет переиспользовать SQL и из пула, и
из транзакции без дублирования. Публичные методы (`CreateIncident`, `MergeLabels`,
`AppendHistory`, `AttachAlert`, …) сохраняются как тонкие обёртки над `s.db` для одиночных
вызовов из других путей.

Новые транзакционные методы:
- `incident.Store.CreateIncidentTx(ctx, inc, labels, hist, ia)` — создание инцидента из алерта
  целиком (consumer).
- `incident.Store.TransitionStatus(ctx, tenantID, id, status, expectedStatus, authorID, hist)` —
  guarded-CAS статуса + история (handler).
- `escalation.Store.AdvanceEscalationState(ctx, st, expectedTier, expectedStatus, hist)` —
  guarded-CAS перехода tier'а + история (escalator). `hist == nil` допускается (ветка advance
  пишет «triggered»-историю отдельно в `triggerTier`).

Публикация событий выполняется вызывающим кодом **после** успешного commit (best-effort,
с warn-логом при сбое — как сейчас).

### 3. Keyset-пагинация (D5)

Курсор инцидентов становится непрозрачным токеном, несущим обе компоненты сортировки:

```
cursor = base64( RFC3339Nano(created_at) + "|" + id )
```

Запрос — keyset по строковому сравнению кортежей:
```sql
WHERE (i.created_at, i.id) < ($cursorTime, $cursorID)
ORDER BY i.created_at DESC, i.id DESC
```

Это устраняет и пропуски/дубли на равных `created_at`, и зависимость от существования строки
(подзапрос `SELECT created_at WHERE id=` убран). `next_cursor` кодируется из последней строки
страницы. Невалидный/неразбираемый курсор трактуется как отсутствие курсора (первая страница) —
без 500.

## Совместимость при деплое и миграция

- **Схема БД не меняется** — миграций нет, откат тривиален (revert кода).
- **Wire-формат событий не меняется** — старые сообщения в очередях читаются как прежде;
  консьюмеры разных версий совместимы. Меняется только число публикаций при гонке (было
  ≥1 дубликат, стало ровно 1) — строго улучшение.
- **Формат курсора пагинации меняется** (был «голый» id, стал base64-токен). Курсоры
  непрозрачны для клиента, в БД/сессиях не персистятся. Единственный эффект — курсор,
  выданный старой версией и переданный новой во время раскатки, не распарсится и будет
  трактован как первая страница. Допустимо (UI просто перезапросит список). **Не BREAKING**
  для схемы API.
- **Поведение CAS-конфликта:**
  - escalation: при единственной реплике CAS всегда успешен (поведение идентично прежнему);
    выгода проявляется только при ≥2 репликах/перекрытии тиков.
  - incident: 409 возможен лишь при реальной конкуренции (UI + авто-резолв). Обычный
    последовательный PATCH ведёт себя как прежде.

## Отклонённые альтернативы

- **Транзакция вокруг `SELECT … FOR UPDATE SKIP LOCKED` на весь батч монитора.** Корректно,
  но держит транзакцию и блокировки на всё время обработки батча (включая сетевые вызовы в
  scheduling), плохо масштабируется и удлиняет блокировки. CAS дешевле и не зависит от
  длительности обработки — выбран как рекомендованный аудитом «самый дешёвый фикс под текущую
  архитектуру монитора».
- **Версионный столбец `version int` + `WHERE version=$`.** Более общий optimistic-lock, но
  требует миграции схемы во всех таблицах состояний. Для текущих переходов достаточно
  естественных дискриминаторов (`current_tier`+`status`, `status`) — миграция не нужна.
- **Расширение CAS на `Stop`/`MaybeResolve`/`PutLabels`/`AddComment`.** Вне объёма CH08
  (матрица покрытия: D1/D3 — advance и PatchStatus). `Stop` идемпотентен и не создаёт двойной
  эскалации (CAS монитора отбросит устаревший advance, если `Stop` опередил). Эти пути
  получают только фикс E4 (логирование ошибки истории), CAS на них — отдельная будущая работа.
- **Распарсивать «старый» курсор (голый id) для обратной совместимости.** Усложняет код ради
  переходного окна в секунды; курсоры эфемерны. Отклонено в пользу простого fallback на первую
  страницу.
