# Design — add-ci-and-quality-gates

## Context

`go.work`-монорепо из 7 модулей (`pkg` + 5 сервисов + `tests/e2e`) без единого
автоматического гейта. Тесты есть, но запускаются вручную; уязвимости auth (DC1)
пролежали незамеченными именно из-за отсутствия `govulncheck` в CI. Чейндж
вводит сетку (CI + линтер + пины + автообновление) и снимает ошибочный тег
`integration` со стаб-тестов. Поведение сервисов не меняется — это чистый
infra/tooling.

Ограничения:
- Go 1.26.4, `go.work` с per-module `replace` — модули собираются и в воркспейсе,
  и автономно.
- Накопленный lint-долг большой (минорные находки областей 4/9 аудита) — нельзя
  блокировать CI на нём в этом чейндже.
- DC1 (две достижимые уязвимости auth) фиксится в CH02, не здесь.

## Goals / Non-Goals

**Goals:** автоматический гейт (`build`/`test -race`/`mod tidy`/`lint`/
`govulncheck`/`integration`); `.golangci.yml`; пин инструментов; Dependabot;
дефолтный `go test ./...` снова покрывает поведенческие тесты (снятие тега);
e2e-`go.sum`.

**Non-Goals:** фикс уязвимостей (CH02), Dockerfile/Trivy/SBOM (CH19),
goleak/новые юниты/table-driven (CH17), исправление накопленного lint-долга
(профильные чейнджи).

## Decisions

1. **Матрица по модулям `go.work`, `fail-fast: false`.** Каждый модуль — своя
   ячейка: `go build ./...`, `go test -race -shuffle=on ./...`,
   `go mod tidy && git diff --exit-code`. Альтернатива (один прогон из корня
   воркспейса) отклонена: хуже изолирует падения и не проверяет автономную
   сборку каждого модуля.

2. **`govulncheck` — информационный (`continue-on-error: true`).** Он сразу
   подсветит DC1, но фикс — CH02. Блокирующий гейт сейчас сериализовал бы все
   последующие PR за CH02 и держал бы main красным. CH02 снимает
   `continue-on-error` после бампа версий. Альтернатива (блокирующий сразу)
   отклонена по этой причине; альтернатива (не добавлять джоб) — противоречит DC2.

3. **`golangci-lint` с `only-new-issues: true`.** Гейтит новый/изменённый код, не
   топя в накопленном долге. Профильные чейнджи (CH12 — ключи логов, CH18 —
   доки/стиль) закрывают долг, редактируя свои файлы, и линтер их поймает.
   Альтернатива (полный прогон, блокирующий) отклонена: сотни существующих
   замечаний сделали бы CI красным с первого дня. Альтернатива (`continue-on-error`)
   отклонена: тогда линтер не гейтит вообще.

4. **Снятие тега `integration` только со стаб-файлов (7 из 9).** Файлы,
   декларирующие «in-memory stubs — no external services required», становятся
   обычными юнит-тестами. `scheduling/store` и `scheduling/tokenindex` тег
   сохраняют (нужны реальные Postgres/Redis). Деление «быстрые/медленные», если
   понадобится, — через `testing.Short()`, а не build-тег (рекомендация аудита T2).

5. **Integration-джоб с `services:`.** Поднимает Postgres/Redis/RabbitMQ как
   сервис-контейнеры, прогоняет `go test -tags integration ./...`. e2e —
   отдельным lightweight-джобом (schedule + `workflow_dispatch`), чтобы не
   утяжелять каждый push.

6. **Пин инструментов через `tool`-директивы (Go 1.24+) + версии в CI.**
   golangci-lint и govulncheck пинятся для репродуцируемости. Dependabot (нативный,
   нулевая настройка) выбран вместо Renovate (требует установки GitHub App).

## Risks / Trade-offs

- **Снятие тега обнажит падающие тесты в дефолтном прогоне** → перед коммитом
  прогнать `go test -race -shuffle=on ./...`; если что-то падает не из-за
  отсутствия инфры — зафиксировать как находку профильного чейнджа, тег не
  возвращать без причины.
- **`only-new-issues` сравнивает с базой (`origin/main`)** → на первом прогоне
  базы ещё нет; полагаемся на сравнение с merge-base PR; для push в main —
  допустимо, что находки информативны.
- **govulncheck информационный → окно без защиты от vuln между CH01 и CH02** →
  принято осознанно; CH02 срочный и следующий, промоутит джоб в блокирующий.

## Migration Plan

Аддитивный чейндж: добавляются файлы CI/конфигов, у 7 тестов убирается build-тег.
Откат — реверт коммита (CI-файлы и снятие тега независимы от рантайма сервисов).
Схемы БД и контракты событий не затрагиваются — миграции не нужны.

**Хэндофф CH02:** снять `continue-on-error` с джоба `govulncheck` после фикса DC1.
