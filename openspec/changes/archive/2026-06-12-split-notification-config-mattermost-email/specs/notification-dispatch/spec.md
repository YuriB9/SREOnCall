## MODIFIED Requirements

### Requirement: Отправка Email-уведомлений через SMTP

Сервис notification ДОЛЖЕН (SHALL) отправлять уведомления об инцидентах и эскалациях по электронной почте через настроенный SMTP-сервер, учитывая per-tenant конфигурацию email из `tenant_notification_config` (`email_enabled`, `smtp_from`, `email_reply_to`, `email_subject_prefix`).

Если для тенанта `email_enabled = false`, email-доставка для события ДОЛЖНА быть пропущена (Mattermost-доставка не затрагивается); пропуск ДОЛЖЕН быть зафиксирован в логе без записи `failed` в журнал доставки. Значение по умолчанию для тенантов без явной настройки — email включён.

Адрес отправителя берётся из `smtp_from` тенанта (при пустом — из глобального `SMTP_FROM` сервиса). Если задан `email_reply_to`, письмо ДОЛЖНО содержать заголовок `Reply-To` с этим адресом. Если задан `email_subject_prefix`, он ДОЛЖЕН добавляться в начало темы письма (например, `[ACME PROD] [SEV1] <incident_title>`).

Email ДОЛЖЕН содержать: ID инцидента, заголовок инцидента (`incident_title` из payload события), severity, текущий статус, уровень эскалации (tier), ссылку на инцидент в веб-интерфейсе и временную метку. Ссылка строится из настраиваемого базового URL фронтенда (`FRONTEND_BASE_URL`) в формате `{base}/{tenant_slug}/incidents?incident={incident_id}`; если базовый URL не задан, письмо отправляется без ссылки с warn-записью в лог.

#### Scenario: Успешная отправка email

- **WHEN** для пользователя сработало уведомление, для него настроен адрес электронной почты, и у тенанта `email_enabled = true`
- **THEN** письмо отправляется через SMTP; тема содержит (при наличии) `email_subject_prefix`, severity и заголовок инцидента, тело — ID, заголовок, severity, статус, tier, ссылку на инцидент и временную метку

#### Scenario: Email-канал отключён для тенанта

- **WHEN** для тенанта инцидента `email_enabled = false`
- **THEN** email-доставка для события пропускается; в лог пишется информационная запись; Mattermost-доставка выполняется как обычно

#### Scenario: Задан адрес для ответа

- **WHEN** у тенанта задан непустой `email_reply_to`
- **THEN** отправленное письмо содержит заголовок `Reply-To` с этим адресом

#### Scenario: Задан префикс темы

- **WHEN** у тенанта задан непустой `email_subject_prefix`
- **THEN** тема отправленного письма начинается с этого префикса

#### Scenario: Базовый URL фронтенда не настроен

- **WHEN** `FRONTEND_BASE_URL` не задан в конфигурации сервиса
- **THEN** письмо отправляется без ссылки на инцидент, в лог пишется предупреждение

#### Scenario: SMTP-сервер недоступен

- **WHEN** SMTP-сервер недостижим
- **THEN** сервис выполняет до 3 попыток отправки (первая и до 2 повторных) с экспоненциальной задержкой, затем помечает доставку как `failed` и логирует структурированную ошибку

### Requirement: Авторизованное получение per-tenant конфигурации из scheduling

Сервис notification ДОЛЖЕН (SHALL) запрашивать конфигурацию уведомлений тенанта (`GET /api/schedules/v1/tenants/{slug}/notification-config`) с сервисной аутентификацией — заголовком `X-Admin-Key`, значение которого поступает из конфигурации сервиса (k8s Secret).

Полученная конфигурация ДОЛЖНА содержать `mattermost_enabled`, полный (немаскированный) `mattermost_webhook_url`, пригодный для отправки сообщений, а также поля email-канала `email_enabled`, `smtp_from`, `email_reply_to`, `email_subject_prefix`. Сервис НЕ ДОЛЖЕН использовать маскированное значение (`scheme://host` без пути) для отправки: если в ответе получен маскированный или пустой URL, Mattermost-доставка пропускается с записью `failed` в журнал доставки и структурированной ошибкой в лог.

Если для тенанта `mattermost_enabled = false`, Mattermost-доставка для события ДОЛЖНА быть пропущена симметрично email-каналу: пропуск ДОЛЖЕН быть зафиксирован в логе информационной записью **без** записи `failed` в журнал доставки (в отличие от случая отсутствующего/маскированного webhook при включённом канале). Значение по умолчанию для тенантов без явной настройки — Mattermost включён.

Сервис ДОЛЖЕН устойчиво обрабатывать ситуацию, когда для тенанта конфигурация ещё не сохранялась: scheduling возвращает по каноническому маршруту 200 с дефолтным конфигом (`email_enabled=true`, пустые поля). При отсутствующем или `nil` конфиге (например, недоступность scheduling) сервис ДОЛЖЕН трактовать email как включённый и отправлять письмо через глобальный `SMTP_FROM`; отсутствие конфига НЕ ДОЛЖНО приводить к пропуску email-доставки. Mattermost при этом пропускается (нет webhook).

#### Scenario: Запрос конфигурации с сервисным ключом

- **WHEN** сервис notification запрашивает конфигурацию тенанта у scheduling
- **THEN** запрос содержит заголовок `X-Admin-Key`, scheduling возвращает HTTP 200 с полным `mattermost_webhook_url` и полями email-канала, и notification использует их для отправки

#### Scenario: Сбой авторизации при получении конфигурации

- **WHEN** scheduling отвечает 401/403 на запрос конфигурации (ключ отсутствует или неверен)
- **THEN** сервис notification логирует структурированную ошибку с уровнем `error` и тенантом; Mattermost-доставка для события пропускается с записью `failed` в журнал, email-доставка продолжается с глобальным `smtp_from`

#### Scenario: Конфигурация тенанта ни разу не сохранялась

- **WHEN** обрабатывается `escalation.triggered` для тенанта без сохранённой конфигурации уведомлений, у дежурного включён канал email
- **THEN** email-уведомление отправляется через глобальный `SMTP_FROM` (канал считается включённым по умолчанию), Mattermost пропускается из-за отсутствия webhook; отсутствие конфигурации НЕ приводит к полному отсутствию уведомления

#### Scenario: Mattermost-канал отключён для тенанта

- **WHEN** обрабатывается событие для тенанта, у которого `mattermost_enabled = false`, даже при заданном валидном `mattermost_webhook_url`
- **THEN** Mattermost-доставка пропускается; в лог пишется информационная запись без записи `failed`; email-доставка выполняется как обычно
