# Зачем

SRE-командам не хватает единой платформы управления дежурствами, которая объединяла бы алерты из разнородных систем мониторинга (Zabbix, Grafana, Prometheus), группировала бы их в инциденты и маршрутизировала уведомления дежурному инженеру — из-за чего команды вынуждены работать с разрозненными инструментами без общей истории инцидентов и логики эскалаций.

## Что меняется

- Новая платформа на Go-микросервисах, развёрнутая в Kubernetes (k3s для локальной разработки); PostgreSQL, Redis и RabbitMQ как ключевая инфраструктура.
- **Сервис приёма алертов (ingestion)**: HTTP-вебхуки для Zabbix, Grafana и Prometheus Alertmanager; дедупликация через Redis; публикация событий в RabbitMQ.
- **Сервис инцидентов (incident)**: группировка алертов в инциденты; поддержка лейблов, комментариев и полной истории изменений; хранение в PostgreSQL.
- **Сервис расписаний (scheduling)**: управление тенантами (команда = тенант), участниками, вебхук-токенами и конфигурацией уведомлений; ротации и смены per-tenant; движок вычисления «кто дежурит прямо сейчас».
- **Движок эскалаций (escalation)**: описание политик (таймауты, уровни, переопределения); автоматическая эскалация, если инцидент не подтверждён в заданный срок.
- **Диспетчер уведомлений (notification)**: отправка уведомлений по Email (SMTP) и Mattermost (вебхук); rate-limiting через Redis; журнал доставки.

## Возможности

### Новые возможности

- `tenant-management`: регистрация команд (тенантов), управление участниками и ролями, вебхук-токены для ingestion, конфигурация каналов уведомлений.
- `alert-ingestion`: приём, нормализация и дедупликация алертов из Zabbix, Grafana и Prometheus Alertmanager; публикация событий в RabbitMQ; идентификация тенанта по вебхук-токену.
- `incident-management`: создание и управление инцидентами из алертов в контексте тенанта; лейблы, комментарии, переходы статусов, история изменений.
- `oncall-scheduling`: описание расписаний и ротаций для тенанта, переопределения; запрос дежурного в любой момент времени.
- `escalation-policies`: многоуровневые политики эскалации с таймаутами; автоматическая эскалация неподтверждённых инцидентов в рамках тенанта.
- `notification-dispatch`: доставка уведомлений дежурным инженерам по Email и Mattermost с дедупликацией и rate-limiting; каналы уведомлений настраиваются per-tenant.

## Влияние

- Новая кодовая база: существующие сервисы не затрагиваются.
- Внешние зависимости: Zabbix, Grafana, Prometheus Alertmanager (входящие вебхуки); SMTP-сервер; Mattermost (исходящие вебхуки).
- Инфраструктура: PostgreSQL, Redis — внутри кластера; RabbitMQ — внешний сервис в prod, поднимается в k3s через Helm только для локальной разработки.
- Kubernetes: каждый сервис развёртывается как отдельный Deployment со своим Service; Ingress для внешних вебхук-эндпоинтов.
