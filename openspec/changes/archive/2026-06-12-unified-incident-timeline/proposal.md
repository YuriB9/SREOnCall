## Why

Сейчас вкладка «История» в панели деталей инцидента показывает только журнал
самого инцидента (изменения статуса, лейблов, комментарии), а история эскалации
ведётся escalation-сервисом отдельно и в UI не отображается. Дежурному, чтобы
понять полную картину инцидента («когда сработал уровень, кого дёрнули, когда
подтвердили»), приходится держать два источника в голове. Объединённый таймлайн
даёт единую хронологию событий в одном месте.

## What Changes

- Вкладка «История» панели деталей инцидента ДОЛЖНА подмешивать события
  эскалации (`GET /api/escalations/v1/{tenant}/incidents/{id}/history`) в общий
  вертикальный таймлайн вместе с событиями инцидента
  (`GET /api/incidents/v1/{tenant}/incidents/{id}/history`), отсортированными по
  времени.
- Добавляются визуальные представления для типов событий эскалации:
  `triggered`, `tier_advanced`, `acknowledged`, `resolved`, `exhausted` — с
  номером уровня и дежурным (`oncall_username`) для срабатываний.
- Заголовок вкладки «История» ДОЛЖЕН показывать счётчик количества записей в
  объединённом таймлайне (по аналогии с «Алерты (N)» и «Комменты (N)»).
- Деградация: если escalation-сервис недоступен или вернул ошибку, вкладка
  ДОЛЖНА показывать журнал инцидента без событий эскалации (как сейчас), а не
  падать целиком — счётчик при этом считает доступные записи.
- Изменений в API бэкенда нет: оба эндпоинта истории уже существуют.

### Modified Capabilities

## Impact

- **Фронтенд** (`frontend/src/pages/IncidentDetailPanel.tsx`): рендер вкладки
  «История», объединение и сортировка двух источников, счётчик в `TabsTrigger`,
  иконки/подписи для типов событий эскалации.
- **Фронтенд API-клиент** (`frontend/src/api/escalations.ts`,
  `frontend/src/api/types.ts`): новый хук `useEscalationHistory` и тип
  `EscalationHistoryEntry` (поля `event_type`, `tier`, `oncall_username`,
  `created_at`); запрос к существующему эндпоинту истории эскалации.
- **Бэкенд**: изменений нет — используются существующие эндпоинты
  `incident .../history` и `escalation .../history`.
- **События RabbitMQ**: не затрагиваются.
- Не **BREAKING**: только добавление данных в существующую вкладку.
