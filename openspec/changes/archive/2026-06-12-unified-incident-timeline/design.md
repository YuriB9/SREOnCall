## Context

Вкладка «История» в [IncidentDetailPanel.tsx](../../../frontend/src/pages/IncidentDetailPanel.tsx)
сейчас рендерит только журнал инцидента из хука `useIncidentHistory`
(`GET /incidents/{id}/history`). История эскалации хранится в отдельном сервисе
`escalation` и доступна через `GET /api/escalations/v1/{tenant}/incidents/{id}/history`,
но во фронтенде клиента/хука для неё нет — есть только `useEscalationStates`
(bulk-состояния) в [escalations.ts](../../../frontend/src/api/escalations.ts).

Формат записи истории инцидента: `HistoryEntry` с `kind: 'status_change' |
'label_change' | 'comment_added'` и временной меткой. Формат записи истории
эскалации (бэкенд `EscalationHistory`): `event_type: triggered | tier_advanced |
acknowledged | resolved | exhausted`, опциональные `tier`, `oncall_user_id`,
`oncall_username`, `created_at`.

Изменение чисто фронтендовое: оба эндпоинта уже существуют, новых API/событий нет.

## Goals / Non-Goals

**Goals:**
- Подмешать события эскалации в единый таймлайн вкладки «История», отсортированный
  по времени.
- Показать счётчик количества записей в заголовке вкладки «История».
- Деградировать без падения, если escalation-сервис недоступен.

**Non-Goals:**
- Изменения бэкенда, API или событий RabbitMQ.
- Слияние/дедупликация совпадающих по смыслу событий (например, `acknowledged` в
  обоих журналах) — события показываются как есть из обоих источников.
- Realtime/стриминг истории; поведение автообновления остаётся как у текущих
  TanStack Query-хуков.

## Decisions

### Решение 1: Объединение на клиенте, два независимых запроса

Оставляем два отдельных запроса (`useIncidentHistory` + новый
`useEscalationHistory`) и объединяем результаты в компоненте, нормализуя обе
коллекции в единый тип элемента таймлайна `{ time, kind, ...payload }` и сортируя
по `time`.

- **Почему не бэкенд-агрегация:** потребовала бы новый межсервисный
  incident→escalation вызов и новый эндпоинт/событие — несоразмерно объёму
  UI-задачи. Граница сервисов сохраняется, escalation остаётся source of truth
  своей истории.
- **Альтернатива (отклонена):** один объединённый эндпоинт на incident-сервисе —
  больше связности и работы, без выгоды для UI.

### Решение 2: Хук `useEscalationHistory` с graceful-деградацией

Добавить в [escalations.ts](../../../frontend/src/api/escalations.ts) хук по
образцу `useEscalationStates`: при ошибке запроса возвращать пустой массив
(`[]`), а не пробрасывать ошибку. Это сохраняет работоспособность вкладки, когда
escalation-сервис недоступен — таймлайн показывает только журнал инцидента.

- **Почему:** в проекте уже есть прецедент мягкой деградации при недоступности
  escalation (`useEscalationStates` возвращает пустую Map). Консистентно.

### Решение 3: Счётчик = длина объединённого массива

Заголовок вкладки рендерит `История ({timeline.length})` по аналогии с
существующими `Алерты ({alerts.length})` и `Комменты ({comments.length})`.
Счётчик считает фактически отображаемые записи, поэтому при недоступности
эскалации он естественно отражает только доступные события.

### Решение 4: Иконки и подписи для типов событий эскалации

Расширить карты `HISTORY_ICON`/`HISTORY_LABEL` (или ввести параллельные для
эскалации) значениями для `triggered`, `tier_advanced`, `acknowledged`,
`resolved`, `exhausted`. Для срабатываний выводить «Уровень N · <oncall_username>».

## Risks / Trade-offs

- **Дублирование событий остановки** (`acknowledged`/`resolved` могут быть и в
  журнале инцидента, и в истории эскалации) → принято осознанно (см. Non-Goals);
  при необходимости дедуп можно добавить отдельным change. Подписи разных
  источников визуально различимы.
- **Расхождение временных меток между сервисами** (часы/таймзоны) → обе метки
  трактуются как UTC ISO-8601 от бэкенда; сортировка по `Date.parse`.
- **Дополнительный сетевой запрос на инцидент** → лёгкий GET, кэшируется
  TanStack Query по ключу инцидента; деградирует мягко.

## Migration Plan

Деплой обычной фронтенд-сборкой; обратная совместимость полная (бэкенд не
меняется). Откат — откат фронтенд-бандла. Миграций БД нет.
