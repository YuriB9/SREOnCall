## MODIFIED Requirements

### Requirement: Панель деталей инцидента с разделённым видом

Система ДОЛЖНА (SHALL) открывать панель деталей инцидента справа от списка при нажатии на строку инцидента. Панель ДОЛЖНА содержать три вкладки: «Алерты», «История», «Комментарии». ID выбранного инцидента ДОЛЖЕН синхронизироваться в search-параметр URL `?incident=<id>` для возможности поделиться ссылкой. При закрытии панели параметр ДОЛЖЕН удаляться. Заголовки вкладок «Алерты», «История» и «Комментарии» ДОЛЖНЫ показывать счётчик количества записей в соответствующей вкладке.

#### Scenario: Открытие деталей инцидента

- **WHEN** пользователь нажимает на строку инцидента в списке
- **THEN** система ДОЛЖНА открыть панель деталей без навигации со страницы списка и обновить URL, добавив `?incident=<id>`

#### Scenario: Вкладка «Алерты»

- **WHEN** активна вкладка «Алерты» в панели деталей
- **THEN** система ДОЛЖНА отобразить все алерты, сгруппированные в этот инцидент (`GET /incidents/{id}/alerts`), с бейджами источника, fingerprint и статусом (firing/resolved)
- **И** бейдж fingerprint ДОЛЖЕН выводить префикс `fingerprint:` и первые 8 символов значения (например, `fingerprint: c72697aa`), а полный хеш ДОЛЖЕН быть доступен в атрибуте `title` бейджа
- **И** при наличии у инцидента лейбла `instance` система ДОЛЖНА отобразить отдельный бейдж `instance: <значение>` (значение `incident.labels.instance`); при отсутствии лейбла бейдж не отображается

#### Scenario: Вкладка «История» — объединённый таймлайн

- **WHEN** активна вкладка «История» в панели деталей
- **THEN** система ДОЛЖНА отобразить единый вертикальный таймлайн, объединяющий журнал инцидента (`GET /api/incidents/v1/{tenant}/incidents/{id}/history`: изменения статуса, изменения лейблов, комментарии) и историю эскалации (`GET /api/escalations/v1/{tenant}/incidents/{id}/history`: срабатывания уровней, переходы уровней, остановки `acknowledged`/`resolved` и исчерпание), отсортированный по времени в хронологическом порядке
- **И** для событий эскалации типа `triggered`/`tier_advanced` система ДОЛЖНА отобразить номер уровня и имя дежурного на момент срабатывания (`oncall_username`), если оно присутствует

#### Scenario: Счётчик записей вкладки «История»

- **WHEN** панель деталей инцидента открыта
- **THEN** заголовок вкладки «История» ДОЛЖЕН показывать количество записей в объединённом таймлайне (сумма событий инцидента и событий эскалации), например «История (7)»

#### Scenario: Недоступность истории эскалации

- **WHEN** активна вкладка «История», но запрос истории эскалации завершился ошибкой или сервис escalation недоступен
- **THEN** система ДОЛЖНА отобразить таймлайн только из доступных событий журнала инцидента без падения вкладки, а счётчик в заголовке вкладки ДОЛЖЕН отражать число доступных записей

#### Scenario: Вкладка «Комментарии»

- **WHEN** активна вкладка «Комментарии» в панели деталей
- **THEN** система ДОЛЖНА отобразить список комментариев в стиле чата с именем автора, временной меткой и текстом, а также поле ввода для публикации нового комментария

#### Scenario: Публикация комментария

- **WHEN** пользователь отправляет непустой комментарий на вкладке «Комментарии»
- **THEN** система ДОЛЖНА выполнить POST на `POST /api/incidents/v1/{tenant}/incidents/{id}/comments` и добавить новый комментарий в список при успехе
