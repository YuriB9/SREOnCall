## Why

Событийный конвейер — то, ради чего система существует, — слеп для метрик
(находка аудита O2): нет ни одного счётчика принятых алертов, созданных
инцидентов, запущенных/исчерпанных эскалаций, отправленных уведомлений, ни
шинных «золотых сигналов» (ack/nack/requeue, длительность обработки,
publish-ошибки), ни метрик пула pgx. Невозможно ответить на базовые
операционные вопросы on-call системы: «доходят ли уведомления?», «не растёт ли
очередь?», «жив ли консьюмер?». Отдельно: HTTP-метрики метят сырой `r.URL.Path`,
что взрывает кардинальность на per-tenant путях (R1).

Это закрывает находки **O2 и R1** из [docs/audit/07-observability.md](../../../docs/audit/07-observability.md).
Находка **O5** (scrape-конфиг / alert-правила / дашборды) **вынесена из объёма
CH11** по решению владельца — будет сделана отдельно; до неё метрики из O2/R1
уже экспонируются на `/metrics` и готовы к потреблению.
Зависимости CH10 (общий `pkg/httpserver` с метрик-middleware) и CH07
(supervisor-петля консьюмера) уже в main — есть куда вешать метрики и что мерить.

## What Changes

- **R1 — кардинальность HTTP-метрик:** `pkg/metrics.Middleware` метит запрос по
  `chi` RoutePattern (`/api/incidents/v1/{tenant}/...`) вместо сырого
  `r.URL.Path`; неузнанный путь сводится к одному лейблу.
- **O2 — шинные «золотые сигналы» (централизованно в `pkg/amqp`):**
  `amqp_messages_processed_total{queue,result}`,
  `amqp_message_processing_seconds{queue}`,
  `amqp_publish_total{exchange,result}` — инструментируются один раз в
  `Consume`/`Publish`, метрики получают все консьюмеры и издатели.
- **O2 — метрики пула pgx:** коллектор `pgxpool.Stat()` в `pkg/db`
  (acquired/idle/total/max, время и число ожиданий), регистрируется из каждого
  `main` (опирается на конфиг пула из CH09/D4).
- **O2 — доменные метрики** по образцу `ingestion/dedup`:
  - ingestion: принято алертов `{source}`;
  - incident: инцидентов создано / авто-резолвнуто;
  - escalation: эскалаций запущено/продвинуто/исчерпано, провалов `GetOnCall`,
    gauge backlog просроченных state'ов;
  - notification: уведомлений отправлено/провалено `{channel,result}`, сработок
    rate-limit `{channel}`.
- Над каждым объявлением метрики — комментарий с PromQL/назначением (соглашение
  скила `golang-observability`).
- Архитектурное решение (имена метрик, централизация шинных метрик в `pkg/amqp`,
  стратегия алертинга) фиксируется в **ADR-0018**.

## Impact

**Затронутые сервисы:**
- **ingestion** — доменные метрики приёма алертов (`internal/handler`,
  `internal/publisher`), pool-метрики в `main`.
- **incident** — метрики создания/резолва инцидентов (`internal/consumer`),
  pool-метрики.
- **escalation** — метрики эскалаций и backlog (`internal/escalator`,
  `internal/monitor`), pool-метрики.
- **notification** — метрики доставки и rate-limit (`internal/notifier`),
  pool-метрики.
- **scheduling** — только R1 (через общий middleware) и pool-метрики.

**Общие модули:** `pkg/metrics` (R1), `pkg/amqp` (шинные метрики), `pkg/db`
(коллектор пула).

**Деплой:** не затрагивается. Находка O5 (scrape/alerts/дашборды,
`deploy/k8s/monitoring/`) вынесена из объёма CH11 — будет сделана отдельно.

**События RabbitMQ:** wire-формат `Envelope`/payload **не меняется** — метрики
снимаются по факту обработки, новых полей в сообщениях нет. **НЕ BREAKING.**

**API:** без изменений. Имена существующих метрик (`http_requests_total`,
`http_request_duration_seconds`, `ingestion_dedup_*`) сохраняются; меняется
только значение лейбла `path` у HTTP-метрик (R1).
