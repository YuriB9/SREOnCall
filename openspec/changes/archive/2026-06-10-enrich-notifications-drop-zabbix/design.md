# Дизайн: Обогащение уведомлений и вывод Zabbix из поддержки

## Context

Два P1-долга из `docs/openspec-code-verification.md`:

1. **Уведомления без содержимого.** Спека `notification-dispatch` требует в email заголовок, severity, статус и ссылку на инцидент; Mattermost — ID, severity, заголовок, статус. Фактически событие `escalation.triggered` несёт только `incident_id`, `tenant_id`, `tenant_slug`, `tier`, `oncall_user_id`, `oncall_username` (`escalation/internal/publisher`), и email сводится к «Escalation tier N» (`notification/internal/dispatcher/email.go:29-34`), а «Title» подменён строкой «Escalation tier %d» (`notifier.go:152`). При этом `incident.created` уже публикует `title`, `severity`, `status` (`incident/internal/publisher/publisher.go:11-18`) — данные есть, escalation их просто не сохраняет.
2. **Zabbix-фантом.** Спека `alert-ingestion` требует эндпоинт Zabbix — его нет (`ingestion/cmd/server/main.go:100-101`, только alertmanager и grafana). При этом `zabbix` принимается как валидный `source` webhook-токена (`scheduling/internal/handler/handler.go:596`), фигурирует в умолчаниях группировки (`incident/internal/store/store.go:495`) и в UI (`TenantSettingsPage.tsx:19`). Принято решение вывести Zabbix из поддержки, а не дореализовывать.

Плюс мёртвый код фронтенда: `useOnCallNow` и `useScheduleWindow` (`frontend/src/api/schedules.ts`) нигде не используются; второй обращается к `/oncall` с параметрами `from`/`to`, которые бэкенд игнорирует.

Зависимость: изменение `fix-p0-delivery-and-filters` (сервисный ключ `X-Admin-Key`, та же правка требования «Срабатывание эскалации по таймауту») применяется первым.

## Goals / Non-Goals

**Goals:**

- Дежурный получает в email/Mattermost достаточно контекста, чтобы принять решение, не открывая систему: заголовок, severity, статус, ссылка.
- Notification не делает синхронных вызовов ради содержимого уведомления — данные приходят в событии.
- Спецификации перестают обещать Zabbix; система не позволяет создать токен для несуществующего источника.
- Во фронтенде не остаётся API-хуков, обращающихся к несуществующим контрактам.

**Non-Goals:**

- Реализация приёма Zabbix (выведен из поддержки решением владельца).
- Очистка исторических данных (`zabbix`-токены, правила группировки) — только ручная рекомендация.
- Шаблонизация писем (HTML, локализация) — текст остаётся plain-text.
- Изменение payload `escalation.exhausted` (канальное оповещение без деталей достаточно).

## Decisions

### 1. Источник данных инцидента — событие, fallback — sync-запрос

**Решение**: escalation расширяет `escalation_states` колонками `incident_title`, `incident_severity`, `incident_status` и заполняет их из payload `incident.created` при авто-назначении политики. При ручной привязке (`POST /incidents/{id}/policy`), где события под рукой нет, escalation выполняет один `GET /api/incidents/v1/{tenant}/incidents/{id}` с сервисным ключом (механизм из `fix-p0-delivery-and-filters`); при сбое запроса поля остаются пустыми, привязка не блокируется.

**Обоснование**: события — основной транспорт данных в системе (тот же принцип, что и `oncall_user_id` в `escalation.triggered`); production-путь (авто-назначение) обходится без новых синхронных зависимостей. Fallback нужен только для редкого ручного пути.

**Альтернативы**:
- Notification сам запрашивает incident при отправке — отклонено: добавляет sync-зависимость на каждый dispatch и противоречит зафиксированному в спеке принципу «payload события самодостаточен».
- Всегда запрашивать у incident в момент триггера — отклонено: лишний вызов в горячем пути эскалации; статус и так известен (эскалация работает только по неподтверждённым инцидентам).

### 2. Ссылка — deep link фронтенда, а не API-эндпоинт

**Решение**: notification получает `FRONTEND_BASE_URL` из конфигурации и строит ссылку `{base}/{tenant_slug}/incidents?incident={incident_id}` — формат, который уже поддерживает дашборд (синхронизация выбранного инцидента в query-параметр). Требование спеки «ссылка на эндпоинт API инцидента» заменяется на ссылку в веб-интерфейс. Если `FRONTEND_BASE_URL` не задан, письмо отправляется без ссылки (с warn-логом), доставка не блокируется.

**Обоснование**: дежурному нужен кликабельный путь к инциденту, а не JSON; deep link уже реализован дашбордом.

**Альтернатива**: ссылка на API-эндпоинт (как в текущей спеке) — отклонено: бесполезна получателю, требование явно писалось как заглушка.

### 3. Обратная совместимость payload

**Решение**: новые поля `incident_title`, `incident_severity`, `incident_status` **добавляются** к `escalation.triggered`; существующие поля не переименовываются. Notification обрабатывает отсутствие новых полей (пустые строки) — деплой escalation и notification не требует синхронности; при пустом title используется текущий fallback «Escalation tier N».

### 4. Глубина удаления Zabbix

**Решение**: удаляются (а) требование из `alert-ingestion`, (б) `zabbix` из enum источников токенов (scheduling + UI), (в) zabbix-ветки умолчаний группировки (incident store/domain), (г) константа `SourceZabbix` из `pkg/domain`. Существующие строки БД (`source='zabbix'` в токенах и правилах) не мигрируются: токены перестают быть полезными (эндпоинта нет и не было), их отзыв — ручная операция администратора; правила группировки для zabbix просто не используются.

**Обоснование**: данные не мешают и не создают дыр (эндпоинта-потребителя не существует); миграция-чистка добавила бы риска без выгоды.

**Альтернатива**: миграция с удалением zabbix-строк — отклонено как необратимая операция ради косметики.

## Risks / Trade-offs

- **Рассинхронизация title/severity при изменении инцидента после создания** (лейблы/severity в событии фиксируются на момент `incident.created`). → Принято: severity инцидента в текущей реализации и так не пересчитывается; точность «на момент создания» достаточна для уведомления.
- **Ручная привязка политики при недоступном incident-сервисе даёт событ/я без title**. → Митигация: warn-лог + fallback-тема письма; привязка не блокируется.
- **`FRONTEND_BASE_URL` забыли задать** → письма без ссылки. → Митигация: warn-лог при старте сервиса, параметр в configmap из коробки.
- **Возврат Zabbix в будущем потребует «раз-удаления» спеки.** → Принято: REMOVED-дельта с указанием Migration документирует путь назад (новое изменение с ADDED-требованием).

## Migration Plan

1. Миграция БД escalation (новые nullable-колонки) — обратносовместима.
2. Деплой escalation (обогащение событий), затем notification (использование полей; до этого новые поля игнорируются) — порядок некритичен благодаря Decision 3.
3. Деплой scheduling/incident (enum без zabbix) и фронтенда (VALID_SOURCES, удаление хуков).
4. Откат — стандартный возврат образов; новые колонки и поля событий обратносовместимы в обе стороны.
5. Администраторам: отозвать существующие zabbix-токены через UI (рекомендация в release notes).

- Нужно ли событие `escalation.exhausted` обогащать теми же полями для канального сообщения — сейчас вне объёма, но колонки состояния делают это дешёвым follow-up'ом.
