## MODIFIED Requirements

### Requirement: Детали инцидента и его алерты

Сервис incident ДОЛЖЕН (SHALL) предоставлять GET `/api/incidents/v1/{tenant}/incidents/{incidentId}` (полные данные инцидента, включая лейблы) и GET `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts` (все привязанные алерты с их статусом firing/resolved, источником, fingerprint, лейблами и временем привязки).

Каждый привязанный алерт ДОЛЖЕН включать собственный набор лейблов (`labels`), сохранённый в момент привязки. Лейблы хранятся per-alert и не зависят от лейблов инцидента; для алертов, привязанных без лейблов (например, ручная привязка без передачи лейблов), `labels` ДОЛЖЕН быть пустым объектом. Это обеспечивает возможность различать алерты одного инцидента по их различающим лейблам (например, `instance`).

#### Scenario: Получение деталей инцидента

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}` участником тенанта
- **THEN** возвращаются данные инцидента: статус, severity, заголовок, лейблы, временные метки, `acknowledged_by`/`acknowledged_at` (если есть)

#### Scenario: Получение алертов инцидента

- **WHEN** выполняется GET на `/api/incidents/v1/{tenant}/incidents/{incidentId}/alerts`
- **THEN** возвращаются все привязанные алерты с fingerprint, источником, статусом, лейблами (`labels`) и временем привязки (`attached_at`)

#### Scenario: Лейблы различаются между алертами одного инцидента

- **WHEN** к инциденту привязаны несколько алертов с совпадающими группирующими лейблами, но разными значениями негруппирующего лейбла (например, `instance`)
- **THEN** ответ возвращает для каждого алерта его собственные `labels` с соответствующим различающим значением, а не лейблы инцидента

#### Scenario: Инцидент другого тенанта недоступен

- **WHEN** выполняется GET деталей инцидента, принадлежащего другому тенанту
- **THEN** сервис возвращает HTTP 404
