## Context

Алерты привязываются к инциденту двумя путями: автоматически в консьюмере (`handleFiring`, очередь `alerts.incident`) и вручную через `POST /api/incidents/v1/{tenant}/incidents/{incidentId}/alerts`. Доменный `IncidentAlert` хранит `fingerprint, source, group_key, status, attached_at`, но не лейблы. В консьюмере полный набор лейблов алерта (`alert.Labels`) доступен — сейчас он лишь мёржится в лейблы инцидента при создании (`MergeLabels`) и далее отбрасывается.

Лейблы инцидента — это снимок лейблов первого алерта на момент создания; они общие и не отражают различия между алертами (в частности, `instance`). Поэтому UI не может различать алерты, кроме как по нечитаемому `fingerprint`.

Спека `incident-management` уже декларирует доступность лейблов алерта на вкладке (требование «Лейблы инцидента»), но контракт `/alerts` их не перечисляет — данное изменение устраняет это расхождение.

## Goals / Non-Goals

**Goals:**
- Хранить лейблы каждого алерта в `incident_alerts` и возвращать их в `/alerts`.
- Дать UI возможность показывать различающие лейблы алертов (прежде всего `instance`).
- Сохранить обратную совместимость API (аддитивное поле) и существующих строк (дефолт `{}`).

**Non-Goals:**
- Изменение логики группировки или мёржа лейблов инцидента (требование «Лейблы инцидента» не меняется).
- Историзация изменений лейблов алерта.
- Обогащение ручной привязки (`POST .../alerts`) полным набором лейблов из внешнего источника — допускается пустой набор.
- Бэкофилл лейблов для уже привязанных (исторических) алертов — для них останется `{}`.

## Decisions

**1. Хранение как `jsonb` в `incident_alerts.labels`.**
Колонка `labels jsonb NOT NULL DEFAULT '{}'`. `jsonb` согласуется с тем, как уже хранятся лейблы инцидента (`incident_labels`/jsonb-конвенция), и позволяет произвольный набор ключей без новых таблиц. Альтернатива — отдельная таблица `incident_alert_labels` (ключ-значение) — отвергнута как избыточная: лейблы алерта читаются всегда целиком вместе с алертом, нормализация не нужна.

**2. Источник лейблов — потребляемое сообщение алерта в консьюмере.**
`alert.Labels` уже в руках в `handleFiring`; прокидываем в `IncidentAlert.Labels` перед `AttachAlert`. Никаких изменений в событийном контракте RabbitMQ не требуется.

**3. Ручная привязка — пустой набор по умолчанию.**
`POST .../alerts` принимает `fingerprint`+`source`; лейблы не передаются. Сохраняем `{}` (валидный пустой объект). При необходимости приём лейблов можно добавить позднее — вне scope.

**4. UI показывает различающие лейблы, а не все.**
На карточке алерта отображать лейблы, которых нет в общем (grouping) наборе инцидента, — в первую очередь `instance`. Это убирает шум от лейблов, одинаковых у всех алертов группы. Базовая реализация может просто показывать `instance` (и при наличии — пару значимых ключей); точный набор — деталь фронтенда.

**5. Дефолт колонки покрывает существующие строки.**
`DEFAULT '{}'` и `NOT NULL` обеспечивают, что старые алерты читаются как пустой объект без отдельного бэкофилла.

## Risks / Trade-offs

- **Исторические алерты без лейблов** → отображаются с `{}` (без различающего лейбла); приемлемо, новые алерты корректны. Бэкофилл не предусмотрен (исходные лейблы алерта в системе не сохранены отдельно).
- **Рост строки за счёт jsonb** → наборы лейблов малы; индексирование `labels` не требуется (читаются по `incident_id`).
- **Ручная привязка без лейблов** → `{}`; UI просто не покажет различающий лейбл для таких алертов.
- **Дополнительное поле в ответе** → аддитивно; существующие клиенты игнорируют незнакомое поле.

## Migration Plan

- Миграция `000004_incident_alert_labels.up.sql`: `ALTER TABLE incident.incident_alerts ADD COLUMN labels jsonb NOT NULL DEFAULT '{}';` Down: `DROP COLUMN labels;`
- Миграции применяются при старте сервиса incident (golang-migrate). Деплой обычный, без простоя: колонка с дефолтом.
- Откат — down-миграция и предыдущий бинарь; новые данные в колонке теряются, остальное не затрагивается.
- Проверка: отправить несколько алертов с одинаковыми `alertname`/`job` и разными `instance`, открыть инцидент — на вкладке «Алерты» у каждого свой `instance`.

## Open Questions

- Показывать на карточке только `instance` или вычислять полный «диф» лейблов алерта против grouping-набора инцидента? Базово — `instance`; расширяемо.
- Нужно ли расширить ручную привязку приёмом лейблов? Вне scope текущего изменения.
