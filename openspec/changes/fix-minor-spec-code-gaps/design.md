## Context

Семь пунктов раздела «Мелкие расхождения» аудита (`docs/spec-vs-code-audit.md`, пункты 8–14). Текущее состояние:

- **(8)** `GET /api/escalations/v1/{tenant}/incidents/{id}/state` возвращает `EscalationState` с полем `escalate_at` и без истории; спека требует `next_escalation_at` и «полную историю уровней». История доступна отдельным эндпоинтом `/history` (отдельное требование спеки). Потребителей `/state` во фронтенде нет.
- **(9)** `PutGroupingRule`/`DeleteGroupingRule` принимают любой `{source}` без валидации; спека ограничивает источники множеством `alertmanager | grafana` (вебхук-токены аналогичную валидацию имеют — 422).
- **(10)** Таблица инцидентов показывает колонку «Alertname» (`labels.alertname`); спека требует «Источник». У инцидента нет поля source (источник — атрибут привязанных алертов), реализация колонки «Источник» потребовала бы расширения модели инцидента.
- **(11)** Хоткей `/` в спеке «переводит фокус в поле поиска/фильтра»; поля поиска в UI нет, в коде — пустая привязка.
- **(12)** В `NotifyTriggered` ошибка `cache.Get` отбрасывается (`cfg, _ :=`); error-лог появляется только если у контакта включён Mattermost. Сценарий спеки «Сбой авторизации при получении конфигурации» требует error-лог с тенантом безусловно.
- **(13)** Мобильный список «Ближайшие 7 дней» фильтрует смены из окна текущего месяца Gantt; на границе месяца хвост из следующего месяца теряется.
- **(14)** Спеки пишут «до 3 повторных попыток»; `pkg/amqp.Publisher`, SMTP- и Mattermost-диспетчеры выполняют 3 попытки всего (первая + 2 повторные) с backoff 1s/2s.

## Goals / Non-Goals

**Goals:**
- Все семь пунктов закрыты; для каждого зафиксировано направление (8, 10, 11, 14 — спека к коду; 9, 12, 13 — код к спеке).
- Спецификации перестают обещать несуществующее поведение; код перестаёт молча отступать от спеки.

**Non-Goals:**
- Реализация поля поиска по инцидентам и хоткея `/` — при появлении поиска это будет отдельное изменение с возвратом хоткея в спеку.
- Добавление поля `source` в модель инцидента ради колонки «Источник» — источник остаётся атрибутом алертов (вкладка «Алерты»).
- Изменение количества попыток или задержек ретраев — только формулировки.
- Критичные и существенные пункты аудита (1–7) — покрыты изменениями `fix-incident-created-tenant-slug`, `remove-unmasked-notification-config-routes`, `fix-substantial-spec-code-gaps`.

## Decisions

**(8) Спека → код: `/state` без истории, поле `escalate_at`.**
Потребителей эндпоинта нет, история уже доступна через `/history` (отдельное требование) — дублировать её в `/state` значит держать два контракта на одни данные. Переименование `escalate_at` → `next_escalation_at` дало бы чуть более говорящее имя, но это изменение JSON-контракта ради косметики; оставляем фактическое имя и фиксируем его в спеке. *Альтернатива (отклонена):* доработать ответ до спеки — лишний код без потребителя.

**(9) Код → спека: 422-валидация source.**
Словарь источников — контракт всей платформы (токены, правила, алерты). Запись правила для произвольного source создаёт мусор в БД и ложное ощущение настройки. Валидация зеркалит уже существующую в `CreateWebhookToken` (`alertmanager | grafana` → иначе 422). DELETE тоже валидируется для симметрии (идемпотентному удалению несуществующего правила это не мешает — 422 до обращения к БД).

**(10) Спека → код: колонка «Alertname».**
У инцидента нет собственного source; колонка «Источник» потребовала бы миграции модели (source первого алерта), что несоразмерно «мелкому» пункту. `alertname` для SRE информативнее на обзорной таблице, а источник каждого алерта виден на вкладке «Алерты». *Альтернатива (отклонена):* добавить `incidents.source` — допустимо как отдельное продуктовое изменение, если колонка понадобится.

**(11) Удалить `/` из спеки и мёртвую привязку из кода.**
Поля поиска нет — требование невыполнимо по построению. Пустая привязка `'/': () => {}` при этом перехватывает `/` и вызывает `preventDefault`, блокируя поиск по странице браузера (quick find в Firefox) без какой-либо пользы — удаляем.

**(12) Логировать сбой `cache.Get` в `NotifyTriggered` безусловно.**
`cfg, _ :=` заменяется на обработку ошибки: `error`-лог с `tenant_slug` и причиной, далее доставка продолжается с `cfg == nil` (email с глобальным `smtp_from`, Mattermost пропускается с `failed`-записью — текущее поведение сохраняется). Спека уже требует ровно это; правка только в коде.

**(13) Отдельное 7-дневное окно для мобильного списка.**
Мобильный список получает собственные запросы `/shifts?from=<сегодня>&to=<сегодня+7д>` per schedule (или объединённое окно `min(monthStart, today)..max(monthEnd, today+7d)` одним запросом — решается при реализации, предпочтителен отдельный запрос: он не зависит от навигации по месяцам Gantt). TanStack Query кеширует по ключу окна — дублирование запросов минимально, выполняется только на мобильной ширине.

**(14) Спека → код: «до 3 попыток».**
3 суммарные попытки с backoff 1s/2s — разумный баланс (последняя попытка через ~3 секунды; больше — растёт латентность вебхука ingestion, который держит HTTP-соединение источника). Меняем формулировки в трёх требованиях; код и тесты не трогаем.

**Зависимость от `fix-substantial-spec-code-gaps`.**
Delta-спеки этого изменения для `incident-management` («Настройка группирующих лейблов per-source») и `incident-dashboard-ui` («Список инцидентов…») модифицируют те же требования, что и существенный чейндж, поэтому включают его текст (алиас `prometheus`, словарь severity) плюс свои правки. Синхронизация/архивация — строго после `fix-substantial-spec-code-gaps`, иначе его правки будут перезаписаны более ранней версией текста.

## Risks / Trade-offs

- [Перезапись delta-спек при неверном порядке архивации] → порядок зафиксирован в proposal/design; delta уже содержит объединённый текст, так что даже при одновременной архивации итог корректен, теряется только трассируемость.
- [(9) Существующие правила с невалидным source в БД] → маловероятно (UI их не создаёт); валидация не блокирует чтение, `ListGroupingRules` и так возвращает только два источника; при желании очистить — ручной DELETE по БД.
- [(11) Пользователи, привыкшие к `/`] → хоткей никогда не работал (пустой обработчик), терять нечего; наоборот, вернётся браузерный quick find.
- [(13) Дополнительные запросы на мобильной ширине] → по одному на расписание, с кешем TanStack Query; объём незначителен.

## Migration Plan

1. Код: incident (валидация), notification (лог), frontend (хоткей, мобильное окно) — обычный деплой в любом порядке, без миграций БД.
2. Спеки: sync/архивация после `fix-substantial-spec-code-gaps`.
3. Откат: свободный, изменения не затрагивают данные и контракты с потребителями (422 — ужесточение для заведомо невалидного ввода).
