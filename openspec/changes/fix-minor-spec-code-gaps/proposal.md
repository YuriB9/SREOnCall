## Why

Сверка спецификаций с кодом (`docs/spec-vs-code-audit.md`, раздел «Мелкие расхождения», пункты 8–14) выявила семь небольших несоответствий. По отдельности они некритичны, но в сумме размывают доверие к спецификациям как к контракту: где-то спека обещает несуществующее (хоткей `/`, история в `/state`, детали в формулировке ретраев), где-то код молча отступает от спеки (отсутствие 422-валидации source, проглоченная ошибка запроса конфигурации, потеря смен на границе месяца). Изменение закрывает весь раздел одним проходом, для каждого пункта фиксируя направление: спека → код для косметики, код → спека там, где спека описывает реальную ценность.

## What Changes

- **(8) Контракт `GET /state` эскалации** — спека выравнивается по коду: ответ содержит `current_tier`, `status` и `escalate_at` (время следующего перехода); полная история уровней доступна через отдельный эндпоинт `/history` (уже описан отдельным требованием). Код не меняется.
- **(9) Валидация source в grouping-rules** — код приводится к словарю спеки: `PUT/DELETE /grouping-rules/{source}` отклоняют source вне множества `alertmanager | grafana` с HTTP 422 (симметрично валидации вебхук-токенов); спека фиксирует сценарий.
- **(10) Колонка таблицы инцидентов** — спека выравнивается по коду: колонка «Alertname» (лейбл `alertname` инцидента) вместо «Источник»; источник каждого алерта виден на вкладке «Алерты» панели деталей. Код не меняется.
- **(11) Хоткей `/`** — удаляется из спеки (поля поиска в UI нет); из кода удаляется мёртвая пустая привязка `'/': () => {}`.
- **(12) Логирование сбоя запроса конфигурации** — код приводится к спеке: в `NotifyTriggered` ошибка получения per-tenant конфигурации логируется с уровнем `error` и тенантом, а не игнорируется молча. Спека не меняется (сценарий уже существует).
- **(13) Мобильный список смен** — код приводится к спеке: список «Ближайшие 7 дней» строится по окну `[сегодня, сегодня+7д]` независимо от отображаемого месяца Gantt, чтобы смены следующего месяца не терялись на границе. Спека не меняется.
- **(14) Формулировка ретраев** — спеки выравниваются по коду: «до 3 попыток (первая и до 2 повторных) с экспоненциальной задержкой» вместо «до 3 повторных попыток» — в alert-ingestion (публикация в RabbitMQ) и notification-dispatch (SMTP, Mattermost). Код не меняется.

## Capabilities

### New Capabilities

Нет — новые возможности не вводятся.

### Modified Capabilities

- `escalation-policies`: (8) требование «Видимость состояния эскалации» — состав ответа `/state` приводится к фактическому контракту, история выносится в ссылку на эндпоинт `/history`.
- `incident-management`: (9) требование «Настройка группирующих лейблов per-source» — source вне `alertmanager | grafana` отклоняется с HTTP 422.
- `incident-dashboard-ui`: (10) колонка «Alertname» вместо «Источник» в требовании списка; (11) хоткей `/` удаляется из требования горячих клавиш.
- `alert-ingestion`: (14) формулировка ретраев публикации в RabbitMQ.
- `notification-dispatch`: (14) формулировка ретраев SMTP и Mattermost.

## Impact

- `services/incident/internal/handler/handler.go` — 422-валидация source в `PutGroupingRule`/`DeleteGroupingRule`; тест.
- `services/notification/internal/notifier/notifier.go` — error-лог при сбое `cache.Get` в `NotifyTriggered`; тест.
- `frontend/src/pages/SchedulesPage.tsx` — отдельное 7-дневное окно запросов для мобильного списка.
- `frontend/src/pages/IncidentListPage.tsx` — удаление пустой привязки `'/'`.
- Delta-спеки пяти capabilities (см. выше); сервисы escalation, ingestion и scheduling кодом не затрагиваются.
- Зависимость: delta-спеки `incident-management` и `incident-dashboard-ui` включают текст изменений из `fix-substantial-spec-code-gaps` (модифицируются те же требования) — синхронизировать/архивировать это изменение следует **после** него.
- `docs/spec-vs-code-audit.md` — пометка пунктов 8–14 как исправленных.
