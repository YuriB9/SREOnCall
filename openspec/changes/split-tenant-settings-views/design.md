## Context

`TenantSettingsPage` ([frontend/src/pages/TenantSettingsPage.tsx](../../../frontend/src/pages/TenantSettingsPage.tsx)) сегодня — единственный экран `/[tenant]/settings`, рендерящий три секции стопкой: `WebhookTokensSection`, `NotificationConfigSection`, `MembersSection`. В боковой навигации ([frontend/src/layouts/GlobalLayout.tsx](../../../frontend/src/layouts/GlobalLayout.tsx)) «Настройки» — обычный `NavLink`. Маршрут объявлен в [frontend/src/routes.tsx](../../../frontend/src/routes.tsx) внутри `AdminGuard`.

Изменение чисто фронтендовое: те же три секции и те же API-хуки (`@/api/tenantSettings`) распределяются по трём отдельным маршрутам, а пункт «Настройки» в навигации становится разворачиваемой группой со ссылками-подпунктами.

## Goals / Non-Goals

**Goals:**

- Три раздела настроек — на отдельных маршрутах, каждый рендерит только свою секцию.
- Пункт «Настройки» в боковой навигации — разворачиваемая группа с тремя подпунктами.
- Базовый `/[tenant]/settings` редиректит на `/[tenant]/settings/webhook-tokens` (обратная совместимость).
- Защита `AdminGuard` распространяется на все подмаршруты.

**Non-Goals:**

- Никаких изменений в логике секций, API, валидации, модалках.
- Никаких изменений в бэкенде, событиях RabbitMQ, схемах БД.
- Не вводим новых UI-зависимостей (используем существующие `react-router-dom`, иконки `lucide-react`, утилиту `cn`).

## Decisions

### Структура маршрутов: вложенные дочерние под `settings` с общим guard

Заменяем единый маршрут `settings` на родительский маршрут с `AdminGuard` и дочерними:

```
settings (element: <AdminGuard><Outlet/></AdminGuard>)
  index            → <Navigate to="webhook-tokens" replace />
  webhook-tokens   → <WebhookTokensPage />
  notifications    → <NotificationConfigPage />
  members          → <MembersPage />
```

Guard ставится один раз на родителе и через `Outlet` покрывает все дочерние — не дублируем `AdminGuard` на каждом подмаршруте. Альтернатива — оставить плоские маршруты с guard на каждом — отклонена из-за дублирования и риска забыть guard на новом подразделе.

### Разбиение страницы: экспортировать секции как отдельные страницы

Текущие внутренние компоненты-секции (`WebhookTokensSection`, `NotificationConfigSection`, `MembersSection`) уже самодостаточны и принимают `tenant`. Выделяем три тонкие страницы-обёртки (`WebhookTokensPage`, `NotificationConfigPage`, `MembersPage`), каждая читает `tenant` из `useParams` и рендерит свою секцию. Секции переносим/переиспользуем без изменения их логики. Старый `TenantSettingsPage` удаляем (его роль берут вложенные маршруты + редирект).

Альтернатива — держать одну страницу и переключать секцию по under-route через таб-стейт — отклонена: реальные маршруты дают прямые ссылки, корректную работу кнопки «назад» и активного выделения.

### Навигация: разворачиваемая группа «Настройки»

В `GlobalLayout` пункт «Настройки» заменяем на кнопку-тогглер с локальным состоянием `useState(open)` плюс блок подпунктов (`NavLink` на каждый подмаршрут). Состояние раскрытия инициализируется открытым, если текущий путь начинается с `/[tenant]/settings`, чтобы при прямом заходе на подраздел подменю было видно. Активный подпункт подсвечивается штатным `isActive` от `NavLink` (стиль как у существующего `navLinkClass`, с отступом для вложенности).

## Risks / Trade-offs

- [Старые ссылки/закладки на `/[tenant]/settings`] → редирект `index` на `webhook-tokens` сохраняет их работоспособность.
- [E2E-тесты могли полагаться на единый экран настроек] → проверить и при необходимости обновить селекторы/переходы в `tests/e2e`.
- [Рассинхрон состояния раскрытия подменю при навигации] → инициализировать `open` из текущего pathname; обновлять при кликах.

## Migration Plan

Деплой одним фронтенд-билдом, бэкенд не затрагивается, миграции БД не требуются. Откат — возврат предыдущего билда фронтенда. Совместимость старых ссылок обеспечена редиректом.
