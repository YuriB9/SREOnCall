## ADDED Requirements

### Requirement: Валидация принадлежности пользователей ротации и переопределений тенанту

Сервис scheduling ДОЛЖЕН (SHALL) при записи расписаний и переопределений проверять, что каждый указанный `user_id` (Keycloak sub) принадлежит текущему участнику тенанта (членство определяется источником участников — Keycloak Admin API для группы `/{slug}` и её подгруппы `admins`). Неизвестные идентификаторы ДОЛЖНЫ отклоняться с HTTP 422 и телом, содержащим перечень непринятых идентификаторов в поле `unknown_members`.

Если источник участников недоступен или не сконфигурирован, валидация на запись ДОЛЖНА пропускаться (fail-open) с предупреждением в журнале, чтобы не блокировать работу среды без Keycloak; запись при этом выполняется в прежнем режиме.

#### Scenario: Создание расписания с неизвестным пользователем в ротации

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules`, и хотя бы один `user_id` из `rotation` не является участником тенанта
- **THEN** сервис возвращает HTTP 422 с полем `unknown_members`, содержащим эти идентификаторы; расписание не создаётся

#### Scenario: Создание расписания с валидной ротацией

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules`, и все `user_id` из `rotation` — участники тенанта
- **THEN** расписание создаётся и возвращается с HTTP 201

#### Scenario: Создание переопределения с неизвестным пользователем

- **WHEN** выполняется POST на `/api/schedules/v1/{tenant}/schedules/{id}/overrides` с `user_id`, не являющимся участником тенанта
- **THEN** сервис возвращает HTTP 422 с полем `unknown_members`; переопределение не создаётся

#### Scenario: Источник участников недоступен

- **WHEN** выполняется запись расписания или переопределения, а источник участников (Keycloak Admin API) недоступен или не сконфигурирован
- **THEN** валидация принадлежности пропускается с предупреждением в журнале, запись выполняется в прежнем режиме

## MODIFIED Requirements

### Requirement: Вычисление текущего дежурного

Сервис scheduling ДОЛЖЕН (SHALL) предоставлять GET `/api/schedules/v1/{tenant}/schedules/{id}/oncall`, возвращающий пользователя, который дежурит в данный момент.

Если вычисленный `user_id` не соответствует текущему участнику тенанта (например, ротация ссылается на устаревший Keycloak sub после переимпорта realm), ответ ДОЛЖЕН по-прежнему содержать вычисленный `user_id`, но помечать результат как устаревший признаком `stale: true` и оставлять `username` пустым; сервис ДОЛЖЕН писать предупреждение в журнал. При соответствии участнику признак `stale` ДОЛЖЕН быть `false`. Признак добавляется аддитивно и не меняет существующие поля ответа.

#### Scenario: Вычисление текущего дежурного

- **WHEN** выполняется GET-запрос на `/api/schedules/v1/{tenant}/schedules/{id}/oncall`
- **THEN** сервис возвращает ID и отображаемое имя текущего дежурного на основании ротации расписания и активных переопределений

#### Scenario: Вычисление дежурного на заданное время

- **WHEN** GET-запрос содержит параметр `?at=<временная метка RFC3339>`
- **THEN** сервис возвращает, кто был бы дежурным в это конкретное время

#### Scenario: Дежурный ссылается на устаревший идентификатор

- **WHEN** вычисленный `user_id` не соответствует ни одному текущему участнику тенанта
- **THEN** ответ содержит этот `user_id`, `stale: true`, пустой `username`, и в журнал пишется предупреждение
