## Why

Сейчас `CreateSchedule` проверяет только непустоту массива `rotation`, а `CreateOverride` — наличие `user_id`, но ни один из них не проверяет, что переданные `user_id` (Keycloak sub) действительно принадлежат участникам тенанта. Из-за этого в ротацию/переопределение можно записать произвольный или устаревший sub. На практике это уже проявилось: после переимпорта realm Keycloak идентификаторы пользователей сменились, а ротации сохранили старые subs — эндпоинт `oncall` теперь возвращает «призрачного» дежурного (`user_id`, которого нет в Keycloak), при этом `username` всё ещё резолвится из устаревшей строки `scheduling.users`. Это «битый» дежурный, на которого escalation авто-назначает инцидент, а notification шлёт уведомления.

## What Changes

- При создании расписания (`POST .../schedules`) каждый sub из `rotation` ДОЛЖЕН проверяться на принадлежность участникам тенанта; неизвестные subs отклоняются с HTTP 422 (`unknown_members`). Та же проверка применяется к обновлению расписания, если эндпоинт обновления присутствует.
- При создании переопределения (`POST .../overrides`) `user_id` ДОЛЖЕН проверяться так же.
- Эндпоинт вычисления дежурного (`GET .../oncall`) при возврате `user_id`, который не соответствует текущему участнику тенанта, ДОЛЖЕН помечать результат как устаревший (дополнительное поле в ответе) и писать предупреждение в журнал — чтобы выявлять уже испорченные ротации, которые валидация на запись не исправляет.
- Деградация: если источник участников (Keycloak Admin API) недоступен/не сконфигурирован, валидация на запись пропускается с предупреждением в журнале (fail-open), сохраняя работоспособность локальной среды без Keycloak.

Изменения аддитивны к контракту: новое поле в ответе `oncall` не ломает существующих потребителей; новое поведение — отклонение невалидных subs (HTTP 422) — ужесточает валидацию записи.

### Modified Capabilities
- `oncall-scheduling`: добавляется требование о валидации принадлежности пользователей ротации и переопределений тенанту при записи; уточняется поведение вычисления дежурного при устаревшем `user_id` (пометка устаревания).

## Impact

- **Сервисы:** `services/scheduling` — `internal/handler` (CreateSchedule, CreateOverride, обновление при наличии, GetOnCall), использование уже доступного `MembersClient` (Keycloak Admin API); возможен helper резолва множества членов тенанта.
- **API:** `POST .../schedules` и `POST .../overrides` могут возвращать HTTP 422 при неизвестных `user_id`; ответ `GET .../oncall` получает дополнительное поле-признак устаревания (аддитивно).
- **Косвенно:** `services/escalation` и `services/notification` — получают возможность отличать «битого» дежурного (поведение по этому признаку — вне scope данного изменения).
- **События RabbitMQ:** не затрагиваются.
- **БД:** схема не меняется.
