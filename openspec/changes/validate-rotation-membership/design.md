## Context

Расписания дежурств хранят ротацию как массив Keycloak sub (`domain.Schedule.Rotation []string`), переопределения — одиночный `user_id`. `GetOnCall` вычисляет дежурного через `rotation.OnCallAt` и дорезолвивает имя через `store.GetUserBySub`. Членство и роли — из Keycloak (группа `/{slug}` и подгруппа `admins`), доступны в хендлере через `MembersClient.GetMembers(ctx, slug)` (используется эндпоинтом `GetMembers`).

Точки записи без проверки членства: `CreateSchedule` (массив `rotation`), `CreateOverride` (`user_id`). Эндпоинт обновления расписания описан в спеке (PATCH), но в текущем коде хендлера отсутствует — валидация применяется к нему при наличии/добавлении.

Корневая проблема устаревания: при переимпорте realm Keycloak subs меняются, ранее сохранённые ротации становятся «битыми». Валидация на запись предотвращает появление новых битых ротаций, но не лечит уже сохранённые — поэтому нужен и сигнал на чтении.

## Goals / Non-Goals

**Goals:**
- Отклонять запись расписаний/переопределений с `user_id`, не принадлежащими тенанту (HTTP 422, `unknown_members`).
- На чтении `oncall` помечать «битого» дежурного (`stale: true`, пустой `username`, предупреждение в журнал).
- Не ломать локальную среду без Keycloak (fail-open валидации).

**Non-Goals:**
- Поведение escalation/notification по признаку `stale` (отдельное изменение).
- Автоматическая миграция/перепривязка устаревших ротаций к новым subs.
- Кэширование/синхронизация членов в БД scheduling как источника истины.
- Изменение схемы БД.

## Decisions

**1. Валидация против Keycloak `GetMembers`, а не против `scheduling.users`.**
`scheduling.users` наполняется upsert-ом при логине и сидом и может содержать устаревшие строки (как раз источник ложного резолва имени). Истина о членстве — Keycloak. Поэтому валидация строит множество допустимых subs из `GetMembers(slug)` и проверяет вхождение. Альтернатива (проверять по `scheduling.users`) отвергнута: не отличает устаревшие записи.

**2. Fail-open при недоступном источнике участников.**
Если `h.members == nil` или `GetMembers` вернул ошибку — валидация пропускается с warning. Обоснование: members-эндпоинт уже опционален, а блокировка создания расписаний без Keycloak сломала бы dev/локалку. Trade-off: при недоступном Keycloak можно записать невалидный sub — приемлемо, т.к. на чтении сработает пометка `stale`.

**3. Признак устаревания вычисляется на чтении, аддитивно.**
`GetOnCall` после вычисления `user_id` проверяет его принадлежность текущим членам (через `GetMembers`); при отсутствии — `stale: true`, `username: ""`, warning. Поле `stale` добавляется в `domain.OncallResult` (JSON `stale`, omitempty не используется, чтобы значение было явным). Существующие поля не меняются → обратная совместимость.

**4. Переиспользование одного helper.**
Ввести `h.tenantMemberSet(ctx, slug) (map[string]struct{}, ok bool)` — возвращает множество допустимых subs и флаг доступности источника. Использовать в CreateSchedule, CreateOverride и GetOnCall, чтобы логика членства не дублировалась.

## Risks / Trade-offs

- **Доп. вызов Keycloak на каждую запись/чтение oncall** → объём членов мал; при необходимости — короткий кэш членов на стороне scheduling (вне scope).
- **Fail-open оставляет лазейку при недоступном Keycloak** → компенсируется пометкой `stale` на чтении и предупреждениями в журнале.
- **Рассинхрон между моментом валидации и чтением** (член удалён из группы после создания расписания) → именно для этого добавлен сигнал `stale` на чтении.
- **PATCH-обновление расписания в коде отсутствует** → валидацию добавляем туда же при реализации обновления; в данном изменении основной фокус — Create-пути.

## Migration Plan

- Только код scheduling, без миграций БД.
- Деплой обычный; аддитивное поле ответа и более строгая валидация записи.
- Для текущих «битых» данных — пересев `scripts/seed-schedules.sh <tenant>` (тянет актуальных членов из Keycloak); после деплоя `oncall` для непересеянных расписаний будет отдавать `stale: true` вместо «призрака».
- Откат — предыдущий бинарь; данные не меняются деструктивно.

## Open Questions

- Нужно ли расширять `unknown_members` именами/`preferred_username` для UX, или достаточно subs? Базово — subs; UI может дорезолвить через members.
- Стоит ли в будущем хранить срез членов в scheduling для оффлайн-валидации при недоступном Keycloak? Вне scope.
